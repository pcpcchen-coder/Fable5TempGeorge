# EXEMPLAR — 技術寫作(technical-writing)黃金示範

## 本文件目的

本文件是「技術寫作」原型的完整黃金示範:對一個帶有模糊處的真實委託(為公開 REST API 的 v1→v2 破壞性變更寫遷移指南),把 PLANNING.md 八階段的思考全程外顯,產出一份可照做的完整遷移指南。執行模型(如 Opus)應以本文件的結構與品質為天花板來模仿——不是抄這裡的 API 細節,而是抄這裡「怎麼建模讀者、怎麼核對事實、怎麼自證」的過程。

## 如何使用本文件

1. 接到任何技術寫作任務時,先讀一遍本文件,再開始你自己的【澄清】。
2. 八個段落標籤(【澄清】…【自審】)與 PLANNING.md 八階段一一對應;你的產出必須含同樣八段,順序不可換。
3. 「◆ 為什麼這樣做」是給你看的後設註解,指出該步在防哪個反模式(AP ID 見 ANTIPATTERNS.md);實戰交付物中不需要輸出這些行。
4. 看完本文件後,對照同目錄的 CRITIQUE.md,看同一題的典型劣質解長什麼樣、差在哪。
5. 本題中的 API、網域、changelog 與所有數字均為**教學用虛構**;示範的是流程與品質標準,不是任何真實產品的文件。

## 使用者委託(原文)

> 我們的 Orders API 八月要上 v2,有幾個 breaking changes,v1 年底會關。麻煩幫我們寫一份遷移指南,放到開發者網站上,給外面串接我們 API 的整合開發者看,讓他們照著做就能從 v1 換過去。變更細節看下面這份內部 changelog。寫清楚一點,外面的工程師水準參差不齊,每次改版 support ticket 都爆量。
>
> ```text
> v2 changes(internal draft,勿外流)
> - auth:X-Api-Key header 移除。OAuth2 client credentials:POST /v2/auth/token(client_id/client_secret)
>   → access_token(expires_in 3600),所有端點改要求 Authorization: Bearer
> - GET /orders/archived 移除,併入 GET /orders,加 ?status= 過濾(open|archived,不帶 = 全部)
> - order.created 改名 order.created_at(格式不變,ISO 8601)
> - GET /orders 改為預設分頁:每頁上限 100,cursor 參數 + 回應 next_cursor;
>   回應由裸陣列改為 {data, next_cursor} 信封(v1 是一次全回)
> - 時程:v2 GA 2026-08-01;v1 sunset 2026-12-31(期間併行)
> - sandbox:api.sandbox.acme-commerce.example(測試帳號固定 237 筆訂單)
> ```

## 【重述】

問題:v2 上線帶四項破壞性變更,外部整合方不改程式會在認證處立刻報錯、在分頁處**默默漏資料**;使用者要的是讓水準參差的外部工程師照文件就能安全遷移、少開 support ticket(目的),手段是一份遷移指南。成功時:整合方在 sandbox 照文件走完檢核清單即完成遷移,不需開票。
原型:technical-writing(TW)。已載入 ANTIPATTERNS.md 的 TW 節,阻斷級條目:AP-TW-02/03/04/05。
可能誤解點:(1) 讀者是「外部」開發者——內部人才知道的前置(憑證怎麼拿、sandbox 在哪)必須補齊;(2)「寫清楚」不等於「寫長」——唯一判準是照做能成功;(3) changelog 是草稿,與實際行為可能有出入——一切以 sandbox 實測為準。

## 【範圍】

目標:
1. 一份單頁遷移指南:四項變更每項都有「對照表列 + 遷移步驟 + 實測範例 + 驗證點」。
2. 常見錯誤排除表與遷移完成檢核清單。
3. 指南內全部範例經 sandbox 實測並附預期輸出。

非目標(每條附不做的理由):
1. 不寫 v2 完整 API 參考文件——遷移指南只講「從 v1 改到 v2 的差異」;參考文件另有其主,重寫是範圍蔓延。
2. 不介紹 v2 新功能與賣點——讀者的任務是遷移不是採購;推銷內容會稀釋警告的能見度(AP-TW-06 的溫床)。
3. 不提供 curl / Python 以外的語言範例——已明示假設;每種語言都淺嘗即止不如兩種寫到可跑。
4. 不寫 OAuth2 原理教學——讀者需要的是能跑的指令,不是理論;有需要另開文件。

邊界裁定:changelog 未載明、但實測會踩到的細節(如 `limit` 上限)納入並標明來源;v2 的 rate limit 資訊不在手上——**不寫**,列入已知限制(沒查證就寫,正中 AP-TW-03)。

## 【方案取捨】

- 方案 A:線性任務式指南——按讀者的遷移順序排四個步驟,每步 before/after 範例 + 驗證點,前置警告區、對照表、排錯表齊備。優點:讀者可從頭走到尾,步驟間依賴(先有權杖才能測端點)自然表達;警告可放在對應操作之前。缺點:寫作成本中等;只想查單點的讀者要先掃目錄。成本:中。
- 方案 B:參考式變更目錄——逐端點、逐欄位列 diff,類似 changelog 的擴寫版。優點:寫作成本最低、單點查閱快。缺點:「預設分頁」這種跨所有呼叫的行為變更沒有自然的掛載點——讀者只查自己用的端點時**根本不會看到它**;步驟間依賴無法表達。成本:小。
- 方案 C:自動遷移工具(掃描並改寫客戶程式碼)+ 簡短說明。優點:理想上最省讀者力氣。缺點:客戶技術棧未知,對任意語言程式碼的自動改寫存在已知會改錯的情境;改壞客戶程式碼難以回復;成本遠超一份文件且超出委託。成本:大。

裁決(依 DECISION_PROTOCOL 第 1 節,逐準則比較):
- 準則 1 正確性:C 對未知程式碼庫的自動改寫「存在已知會產生錯誤結果的情境」,且錯誤發生在客戶的程式碼裡、難以回復 → **C 在正確性淘汰**。B 的每一條內容都可以寫對,但以「讀者照文件行動後的結果」衡量,其結構讓影響所有人的分頁變更對只查單點的讀者不可見——這是可預見會導致漏改、漏資料的情境;A 把該變更做成必經步驟 → **A 在正確性優於 B**,裁決結束,不再比較後續準則(B 的寫作成本較低屬準則 3,下位準則不得推翻上位)。

決策紀錄:[取捨] A vs B vs C → 選 A;裁決準則:準則 1 正確性;理由:C 有已知改錯情境且不可回復,B 的結構讓最危險的行為變更躲過只查單點的讀者。B 的單點查閱優點以「變更總覽對照表」一節併入 A。

◆ 為什麼這樣做:寫作成本最低的 B 是第一直覺的誘惑;逐準則比較把「讀者照做的結果」放回第一位,防 DECISION_PROTOCOL 1.4 的「偏好倒灌」,也是 **AP-TW-01** 在方案層的版本——選 B 等於按作者方便而非讀者需要組織內容。

## 【分解】

1. 事實核對:在 sandbox 逐條實測 changelog 四項變更的實際行為(高風險:changelog 是內部草稿,若與實際不符,指南每一步都是錯的)。完成定義:每項變更都有實測請求與回應紀錄。依賴:無。
2. 骨架:讀者與版本宣告、前置警告區、前置需求、變更總覽對照表。完成定義:骨架完成,四項變更在對照表各佔一列。依賴:1。
3. 四個遷移步驟:每步含前置狀態、v1/v2 對照範例、驗證點。完成定義:每步範例都有 sandbox 實測輸出。依賴:2。
4. 排錯表、檢核清單、回退方案、術語表。完成定義:排錯表涵蓋實測中觸發過的全部錯誤回應。依賴:3。
5. 完稿檢查:冷讀者走查、術語一致性搜尋、交叉引用核對。完成定義:三項檢查各有紀錄,發現的問題已修。依賴:4。

◆ 為什麼這樣做:項 1 排最前,把「changelog 說的」全部換成「我在 sandbox 看到的」,防 **AP-TW-03 文實不符** 與 **AP-TW-04 幻覺功能**——指南裡的每個參數、每個錯誤碼,先看到才准寫。

## 1. 開始前必須知道的三件事

> **警告 1:v1 於 2026-12-31 停止服務。** 該日之後所有 v1 請求一律回 `410 Gone`。
>
> **警告 2:v2 的列表端點預設分頁(每頁最多 100 筆)。** 這是本次唯一「不改程式也不會報錯」的變更:沿用 v1「一次拿全部」寫法的程式,會拿到不完整的資料而**沒有任何錯誤訊息**。訂單可能超過 100 筆的整合方必須完成步驟 4。
>
> **警告 3:存取權杖(access token)3600 秒過期。** v1 的 API key 永久有效,v2 的權杖不是;長時間執行的程式必須實作權杖更新(見步驟 1)。

遷移期間 v1 與 v2 併行至 2026-12-31,兩版共用同一份訂單資料——可以逐步遷移、隨時切回(見第 10 節)。

## 2. 前置需求

- [ ] v2 用戶端憑證:到開發者主控台(`https://developers.acme-commerce.example/console`)→「API 憑證」→「建立 v2 憑證」,取得 `client_id` 與 `client_secret`。**`client_secret` 只顯示一次**,請立即存入你的密鑰管理機制。
- [ ] 可對 sandbox 發出 HTTPS 請求的環境。本指南範例使用 curl 與 Python 3(requests);其他語言請對照轉換。

本指南所有範例都已在 sandbox 實測。請先在 sandbox 完成全部步驟與第 8 節檢核,再切換正式環境。

## 3. 變更總覽(v1 → v2 對照表)

| # | 項目 | v1 | v2 | 誰會受影響 | 對應步驟 |
|---|------|----|----|-----------|---------|
| 1 | 認證 | `X-Api-Key: <key>` 標頭 | `Authorization: Bearer <存取權杖>`;權杖由 `POST /v2/auth/token` 取得,3600 秒過期 | 所有整合方 | 步驟 1 |
| 2 | 端點 | `GET /v1/orders`(僅進行中)與 `GET /v1/orders/archived`(已封存)兩支 | 合併為 `GET /v2/orders`,以 `status` 參數過濾(`open` / `archived`;不帶 = 全部) | 有呼叫 archived 者;以及所有直接換端點的人(見步驟 2 注意事項) | 步驟 2 |
| 3 | 欄位 | 訂單物件 `created` | 改名 `created_at`(值與格式不變,ISO 8601) | 有讀取該欄位者 | 步驟 3 |
| 4 | 行為 | 列表一次回傳全部(裸 JSON 陣列) | 預設分頁:回應改為 `{"data": [...], "next_cursor": ...}` 信封,每頁最多 100 筆 | **所有列表呼叫;不改會默默漏資料** | 步驟 4 |

## 4. 步驟 1:改用 OAuth2 存取權杖

**前置狀態**:已取得 `client_id` / `client_secret`(第 2 節)。

v1 寫法(遷移前):

```bash
curl -s https://api.acme-commerce.example/v1/orders \
  -H "X-Api-Key: your_api_key"
```

v2 寫法(遷移後)——先換權杖,再帶權杖呼叫:

```bash
# 4-1. 以用戶端憑證換取存取權杖
curl -s -X POST https://api.sandbox.acme-commerce.example/v2/auth/token \
  -H "Content-Type: application/json" \
  -d '{"grant_type": "client_credentials",
       "client_id": "你的 client_id",
       "client_secret": "你的 client_secret"}'
```

預期輸出:

```json
{"access_token": "eyJhbGciOiJSUzI1NiJ9...", "token_type": "Bearer", "expires_in": 3600}
```

```bash
# 4-2. 之後所有請求都帶 Authorization 標頭(以下各步驟以環境變數 ACME_TOKEN 存放權杖為例)
curl -s https://api.sandbox.acme-commerce.example/v2/orders \
  -H "Authorization: Bearer $ACME_TOKEN"
```

**權杖更新**:長時間執行的程式,請在「取得權杖時間 + `expires_in` − 60 秒」前重新呼叫 `/v2/auth/token`;收到 `401 {"error": "token_expired"}` 時自動重取一次並重試原請求(只重試一次,避免無窮迴圈)。

**驗證點**:用新權杖呼叫 `GET /v2/orders` 應得 `200` 與 JSON 信封;改用舊的 `X-Api-Key` 呼叫 v2 應得 `401 {"error": "missing_bearer_token"}`。兩者都符合,認證遷移完成。

## 5. 步驟 2:改寫已合併的訂單端點

**前置狀態**:步驟 1 完成(以下範例需要有效權杖)。

| 你的 v1 呼叫 | 改為 v2 呼叫 |
|---|---|
| `GET /v1/orders` | `GET /v2/orders?status=open` |
| `GET /v1/orders/archived` | `GET /v2/orders?status=archived` |
| (v1 無此能力) | `GET /v2/orders`(不帶 `status` = 進行中 + 已封存全部) |

> **注意**:v1 的 `GET /orders` 只回進行中訂單。若直接改成不帶參數的 `GET /v2/orders`,會**多拿到已封存訂單**。要與 v1 行為一致,必須帶 `status=open`。

```bash
curl -s "https://api.sandbox.acme-commerce.example/v2/orders?status=archived&limit=2" \
  -H "Authorization: Bearer $ACME_TOKEN"
```

預期輸出(sandbox 固定測試資料):

```json
{
  "data": [
    {"id": "ord_7d01", "status": "archived", "total": 990, "created_at": "2026-03-14T02:11:09Z"},
    {"id": "ord_6b3a", "status": "archived", "total": 45500, "created_at": "2026-02-28T16:40:00Z"}
  ],
  "next_cursor": "Y3JlYXRlZF9hdDoyMDI2LTAyLTI4"
}
```

`status` 只接受 `open` 與 `archived`;其他值會得到 `400 {"error": "invalid_status"}`。

**驗證點**:呼叫舊路徑 `GET /v2/orders/archived` 應得 `404`;呼叫 `GET /v2/orders?status=archived` 應得 `200`。

## 6. 步驟 3:欄位改名 created → created_at

訂單物件的 `created` 在 v2 改名為 `created_at`,值與格式完全不變(ISO 8601,UTC)。

遷移方法:在你的程式碼庫全域搜尋訂單物件的取值處(如 `order["created"]`、`order.created`),逐一改為 `created_at`。只需改「讀取 API 回應」的程式;你自己資料庫的欄位名不受影響。

**驗證點**:全域搜尋零筆殘留;對照步驟 2 的預期輸出,回應中只有 `created_at`、沒有 `created`。

## 7. 步驟 4:處理預設分頁(最容易默默出錯)

> **先讀這段再動手**:`GET /v2/orders` 單次最多回 100 筆(`limit` 上限 100,超過會得到 `400 {"error": "invalid_limit"}`)。回應的 `next_cursor` 非 `null` 表示還有下一頁。**不跟隨 `next_cursor` 的程式,從訂單數超過 100 筆那天起開始默默漏資料,不會有任何錯誤。**

v1 寫法(遷移前)——一次拿全部:

```python
orders = requests.get(V1_BASE + "/orders",
                      headers={"X-Api-Key": API_KEY}, timeout=10).json()
```

v2 寫法(遷移後)——迴圈跟隨游標讀到底:

```python
import requests

V2_BASE = "https://api.sandbox.acme-commerce.example/v2"

def fetch_all_orders(token, status=None):
    """取回全部訂單:跟隨 next_cursor 直到 null。"""
    orders, cursor = [], None
    while True:
        params = {"limit": 100}
        if status is not None:
            params["status"] = status
        if cursor is not None:
            params["cursor"] = cursor
        resp = requests.get(f"{V2_BASE}/orders", params=params,
                            headers={"Authorization": f"Bearer {token}"},
                            timeout=10)
        resp.raise_for_status()
        body = resp.json()
        orders.extend(body["data"])
        cursor = body.get("next_cursor")
        if cursor is None:
            return orders
```

sandbox 實測:測試帳號固定 237 筆訂單,`len(fetch_all_orders(token))` 回傳 `237`,共發出 3 次請求(100 + 100 + 37)。

**驗證點**:在 sandbox 執行你遷移後的程式,取回筆數應等於 237(sandbox 固定測試資料);若拿到 100,表示你還在只讀第一頁。

## 9. 常見錯誤排除

| 症狀 | 原因 | 解法 |
|---|---|---|
| `401 {"error": "missing_bearer_token"}` | 仍在送 `X-Api-Key`,或 `Authorization` 標頭格式錯誤 | 改用 `Authorization: Bearer <存取權杖>`(步驟 1) |
| `401 {"error": "token_expired"}` | 存取權杖已超過 3600 秒 | 重新呼叫 `/v2/auth/token`;實作到期前自動更新(步驟 1) |
| `404`(呼叫 `/v2/orders/archived`) | 該路徑已在 v2 移除 | 改用 `GET /v2/orders?status=archived`(步驟 2) |
| `400 {"error": "invalid_status"}` | `status` 帶了 `open` / `archived` 以外的值 | 對照步驟 2 的參數表 |
| 程式拋 `KeyError: 'created'` | 欄位已改名 | 改讀 `created_at`(步驟 3) |
| 取回的訂單比 v1 少 | 只讀了第一頁 | 跟隨 `next_cursor` 讀到底(步驟 4) |
| 取回的訂單比 v1 多 | `GET /v2/orders` 未帶 `status`,多了已封存訂單 | 加上 `status=open`(步驟 2) |
| `400 {"error": "invalid_limit"}` | `limit` 超過 100 | `limit` 上限為 100(步驟 4) |
