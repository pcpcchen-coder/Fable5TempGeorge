# CRITIQUE — 技術寫作(technical-writing)劣質解批改

## 本文件目的

本文件呈現與 EXEMPLAR.md 同一道委託(Orders API v1→v2 遷移指南)的「典型劣質解」——模擬較弱模型的真實輸出:有章節、有程式碼區塊、有結語,表面煞有介事;實際上憑印象寫 API 行為、幻覺出不存在的參數、把最危險的變更埋在文末、對讀者的第一個問題(權杖怎麼拿)隻字未提——並逐點批改。目的是讓執行模型(如 Opus)能在自己的產出中辨認出同樣的病徵。

## 如何使用本文件

1. 先讀 EXEMPLAR.md 的正解(含【執行】項 1 的 sandbox 實測表——本文多處批改以該表為事實基準),再讀本文件的劣質解,對照落差。
2. 逐點批改表可當成「反向檢查表」:交付任何文件前,拿自己的產出對照,任何一列的病徵出現就回去修。
3. 注意劣質解「看起來很完整」——結構齊全、語氣專業、範例都有。批改的重點是教你看穿這種表面完整:它的每一個具體技術陳述,幾乎都經不起與實際 API 行為的比對。

## 劣質解原文(模擬較弱模型的輸出,全文照錄)

````markdown
# 邁向未來:Acme Commerce API v2 遷移完全指南

## 前言

在當今快速演進的數位生態系中,API 的持續迭代是保持技術領先的關鍵。Acme Commerce 團隊很高興地宣布,經過長時間的努力,全新設計的 v2 API 正式與大家見面!v2 在安全性、一致性與擴充性上都有長足的進步。本指南將協助您順利完成從 v1 到 v2 的升級之旅。

## v2 亮點搶先看

- 全新 OAuth2 認證機制,安全性大幅提升
- 更簡潔直觀的端點設計
- 更一致的欄位命名規範
- 即將推出:Webhooks 事件推播(beta)與 GraphQL 查詢介面,敬請期待!

## 認證方式升級

v2 採用了業界標準的 OAuth2 認證架構,取代 v1 較為簡單的 API key 機制,為您的整合提供更大的彈性與安全保障。

升級非常簡單,您只需要把原本的 API key 改放到 Authorization 標頭即可:

```bash
curl https://api.acme-commerce.example/v2/orders \
  -H "Authorization: Bearer YOUR_API_KEY"
```

如果您暫時不方便修改程式,v2 也貼心地提供了相容模式:在請求 URL 加上 `?legacy_auth=true`,即可繼續使用原本的 X-Api-Key 標頭,無痛過渡。

另外,考量到部分用戶的令牌管理需求,Token 支援自動續期,無需擔心過期問題。

## 端點整併

為了讓 API 更加簡潔,我們將部分端點進行了整併。原本的 `GET /v1/orders/archived` 已合併至 `GET /v2/orders`,現在一個端點就能滿足所有需求:

```bash
curl https://api.acme-commerce.example/v2/orders \
  -H "Authorization: Bearer YOUR_API_KEY"
```

回應範例:

```json
[
  {"id": "ord_8f2c", "status": "open", "total": 1280, "created": "2026-06-30T08:15:00Z"},
  {"id": "ord_7d01", "status": "archived", "total": 990, "created": "2026-03-14T02:11:09Z"}
]
```

若您只需要封存的訂單,加上 `archived=true` 參數即可輕鬆過濾。

## 欄位命名優化

為了提供更一致的開發體驗,v2 對部分欄位的命名進行了優化調整,整體風格更加統一。完整的欄位對照請參見附錄 A。

## 其他小改動

- 列表端點現在支援分頁功能,您可以使用 `page` 與 `limit` 參數自由控制回傳筆數,提升載入效能。
- 錯誤訊息格式更加友善易讀。
- 回應速度整體提升約 30%。

## Python 快速上手

```python
import requests

resp = requests.get("https://api.acme-commerce.example/v2/orders",
                    headers={"Authorization": "Bearer " + token})
print(resp.json()[0]["created"])
```

## 結語

恭喜您!完成以上步驟後,您的整合即可無縫升級至 v2,享受更快、更安全、更彈性的全新 API 體驗。v2 是我們邁向下一個十年的重要里程碑,期待與您一同成長。如果在升級過程中遇到任何問題,歡迎隨時聯繫我們的支援團隊,我們將竭誠為您服務。升級愉快!
````

## 這份劣質解最致命的三個問題

1. **幻覺功能與文實不符遍布全文(#2、#3、#4、#6、#8)**:`legacy_auth=true`、`archived=true`、`page`、「自動續期」全是憑相似 API 的印象補全出來的。錯誤的文件比沒有文件更糟——讀者會信任它、照它行動、然後在生產環境收穫 401 與資料缺漏,再回頭開 support ticket(委託想解決的問題反而被放大)。這是 AP-TW-04 / AP-TW-03 兩條阻斷級反模式,任何一條都足以否決交付。
2. **最危險的變更被埋成「其他小改動」(#8)**:預設分頁是唯一「不改程式也不報錯」的變更,漏資料無聲無息,卻被寫成文末的選配功能介紹。位置就是優先度——把炸彈放在附錄,等於拆掉讀者唯一的引信警報(AP-TW-06)。
3. **讀者照文件從第一步就走不下去(#1、#9)**:全文沒有教讀者取得存取權杖的任何方法,Python 範例的 `token` 憑空出現。教學文件的唯一品質標準是「照做能成功」;第一步就斷,後面寫得再多都是零(AP-TW-05)。
