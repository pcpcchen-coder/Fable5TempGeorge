# 黃金示範:研究分析(research-analysis)

## 本文件目的

本文件是研究分析(RA)原型的完整黃金示範:以一個真實委託(多 repo 是否遷移 monorepo)示範八階段規劃協議的完整走法,思考全程外顯。執行模型(如 Opus)接到研究分析類任務時,應以本文件為 few-shot 範本,模仿其階段順序、標籤格式、證據標註方式與自審深度。

## 如何使用本文件

1. 接到 RA 類任務時,先完整讀過本示範一次,再開始自己的【澄清】。
2. 八個思考段落標籤(【澄清】【重述】【範圍】【方案取捨】【分解】【驗收定義】【執行】【自審】)的順序與格式照抄,內容換成當前任務。
3. 「◆ 為什麼這樣做」是寫給你看的後設註解,說明該步在防哪個反模式(AP ID 見 `ANTIPATTERNS.md`);你自己的產出中不需要輸出這些註解,但必須做到它們指出的事。
4. 證據強度標註法(【事實】【慣例】【推論】【假設】+ 信心等級)是 RA 原型的必要格式,對應 `DECISION_PROTOCOL.md` 第 4 節,不得省略。

## 使用者委託(原文)

> 我們是一個 15 人的工程團隊,程式碼散在十幾個 repo 裡,大家常抱怨改個東西要動好幾個 repo 很麻煩。最近有人提議乾脆全部搬進一個 monorepo。網路上眾說紛紜:Google 都這樣做,但也有人說小團隊不要學。你幫我們研究一下:我們該遷移嗎?給我們一份分析報告。

## 【澄清】

任務描述缺少下結論所需的全部情境資料。逐句檢查後,命中 PLANNING.md 表 A 的兩種「必須先問」情況:#6(需要使用者才有的資訊:repo 現況、痛點、資源限制)與 #4(多種合理解讀:「分析報告」是否包含遷移計畫)。依規則一次問完:

```markdown
開工前需要確認 4 件事(可只回選項字母或簡答):
1. 現況盤點:repo 數量、各 repo 的語言/用途、程式碼總規模大約多少?
   (為何而問:monorepo 的成本與規模強相關,沒有這項資料,任何結論都無適用性。)
2. 想靠遷移解決什麼?請列出目前最痛的 2-3 個具體問題。
   (為何而問:「該不該」的判準是「能否解決你們的痛點」,不是 monorepo 的抽象優劣。)
3. 有沒有專職平台/DevEx 人力?CI 平台是什麼?能接受引入需專人維運的自建建置系統嗎?
   A. 有專職人力 B. 沒有,需用現成工具 —— 建議:照實回答
   (為何而問:大公司 monorepo 的效益建立在自建工具鏈上,這是適用性的分水嶺。)
4. 產出範圍:只要「該不該遷移」的決策分析,還是要連遷移計畫一起?
   A. 只要決策分析 B. 含遷移計畫 —— 建議:A(若結論是遷移,計畫另開任務)
```

**(示範用)假設使用者回覆如下——實戰中此處必須真的停下等待回覆,不得用假設的回答繼續:**

1. 12 個 repo:6 個後端服務(Node/TypeScript)、2 個前端(React)、3 個內部共用 npm 套件、1 個 Terraform infra。總計約 28 萬行。
2. 三大痛點:(a) 跨 repo 改動要開 3-4 個 PR 且難以同步上線;(b) 共用套件在各服務鎖定不同版本,造成行為不一致;(c) 12 份 CI 設定重複維護。
3. 沒有專職平台人力;GitHub + GitHub Actions;不接受需專人維運的自建系統(如 Bazel 級投資)。
4. A,只要決策分析報告。

已確認:
- 委託未附任何 repo 清單或現況資料,上述 1-2 為委託方口頭提供,標註來源為「委託方提供,2026-06」。

假設:
- 假設:「monorepo」指單一 git repository 容納多個專案,不含 Google 式自建 VCS 與自建建置農場;若不符請告知,影響範圍是外部證據的選取,約需重做半天。
- 假設:報告讀者為工程團隊與技術主管,可用工程術語,但關鍵術語首次出現時仍給一句定義;若不符請告知,影響範圍是行文風格,約 30 分鐘可調。
- 假設:證據查證基準日為 2026-07;若需以其他時點為準請告知,影響範圍是時效敏感段落。

◆ 為什麼這樣做:委託的原始問法(「該遷移嗎?」)是開放問題,直接回答會落入「用通論回答個案」。先取得情境資料,才有資格說「不知道你們的情況我無法回答」以外的話——這是 AP-RA-08(迴避「不知道」)的正面版:要嘛拿到資料好好答,要嘛明說答不了。

## 【重述】

問題:團隊要的不是「monorepo 好不好」的通論,而是「以 15 人、TypeScript 為主、無平台人力的現況,遷移 monorepo 的效益是否超過成本;若是,在什麼條件下成立」。monorepo 是手段,目的是消除跨 repo 原子變更的摩擦與共用套件版本飄移。
原型:research-analysis(RA)。
可能誤解點:
1. 把委託當成「寫一篇 monorepo 介紹文」——委託要的是針對此團隊的裁決,不是科普。
2. 痛點 (c)(CI 重複)可能根本不需要 monorepo 就能解——需在子問題中檢驗,不能預設三個痛點都指向遷移。
3. 「十幾個 repo」不等於「全部都該合併」——infra repo 的變更模式與權限需求可能不同。

## 【範圍】

目標:
1. 對「該不該遷移」給出明確的條件式結論(遷移/不遷移/在何條件下遷移),不騎牆。
2. 報告中每個關鍵陳述附證據強度標註(事實/慣例/推論/假設 + 信心等級 + 來源與時間)。
3. 給出建議的適用條件與不適用條件,以及不遷移時的重新評估觸發訊號。

非目標:
1. 不寫遷移執行計畫(時程、工作分解)——澄清問題 4 已確認另行委託。
2. 不做 PoC 或實測——委託為分析報告;實測建議可作為報告的「下一步」出現,但本次不執行。
3. 不評估 Google 式自建 VCS 與自建建置系統——澄清問題 3 已排除。
4. 不比較 CI 平台遷移(GitHub Actions 為既定前提)。

邊界裁定:「不遷移也能解痛點的替代方案」納入範圍——「該不該遷移」的合理答案必須與最佳替代方案比較,這是問題本身的構成部分,不是範圍蔓延。

## 【方案取捨】

三個候選的「研究方法」方案:

| 方案 | 一句話描述 | 優點 | 缺點 | 成本 |
|---|---|---|---|---|
| A 現成彙整轉述 | 找 2-3 篇「monorepo vs polyrepo」比較文,改寫成報告 | 最快 | 證據不可溯源;來源立場未知;與團隊情境脫節 | 小 |
| B 子問題分解+交叉驗證 | 收斂成可回答子問題,逐題多來源查證,以團隊現況為裁決軸 | 每條結論可獨立溯源、獨立修正;結論綁定情境 | 工時較高 | 中 |
| C PoC 實測 | 先合併 2-3 個 repo 建試點,用實測數據下結論 | 第一手數據 | 只覆蓋技術面子問題,組織面仍需文獻;工時最大;屬執行而非分析 | 大 |

比較(依 DECISION_PROTOCOL 第 1 節優先序):
- 準則 1 正確性:方案 A 存在已知會產生錯誤結果的情境——彙整文若有錯,錯誤整批進入報告且無法辨識;B、C 無此已知情境。**A 在正確性落敗,淘汰。**
- 準則 1(續)B vs C:C 對技術子題有實測優勢,但對組織面子題(權限、流程)無覆蓋,兩者整體打平。
- 準則 2 可回復性:B 的每條證據可獨立溯源、錯一條改一條;C 的實測結論若 PoC 設計有誤(選錯試點 repo)需整批重做。**B 勝。**
- 另 C 同時違反範圍紀律(委託為分析,PoC 是執行;見 GENERAL_RUBRIC R3)。

選定:B。
否決 A:依準則 1 正確性——轉述路線使錯誤無法定位,正確性不可被「快」交換。
否決 C:依準則 2 可回復性與 R3 範圍紀律——實測的錯誤成本整批化,且超出委託範圍。

[取捨] 方案A vs B vs C → 選B;裁決準則:準則1 淘汰A、準則2 B勝C;理由:逐源查證使錯誤可局部修正。

◆ 為什麼這樣做:在方案層就淘汰「轉述現成彙整」,是 AP-RA-04(單一來源定論)的結構性預防——不是等寫報告時才想起要交叉驗證,而是讓研究方法本身不允許單一來源。

### 0. 結論先行

**建議:遷移,但只遷移 TypeScript 生態系,且附前提條件。**(信心:中)

把 11 個 TS/JS repo(6 服務、2 前端、3 共用套件)合併為單一 monorepo,採 pnpm workspaces 加現成增量建置工具(Nx 或 Turborepo);Terraform infra repo 維持獨立。此建議在下列前提全部成立時才動手(詳見第 3 節);任一不成立,先採替代方案(changesets 自動化版本管理 + reusable workflows),六個月後依第 3.4 節的訊號重新評估。

|---|---|---|
| a | 跨 repo 改動需 3-4 個 PR,難同步上線 | 是【推論:原子變更被 repo 邊界切開,這是結構性後果】 |
| b | 共用套件在各服務鎖定不同版本,行為不一致 | 是【推論:版本飄移源自「發布—升級」的非同步性,repo 邊界是其載體】 |
| c | 12 份 CI 設定重複維護 | **否**。GitHub Actions 的 reusable workflows 允許跨 repo 共用 CI 定義【事實|GitHub 官方文件 "Reusing workflows",查證 2026-07】 |

**答案:三個痛點中 a、b 是 repo 邊界的結構性後果;c 不遷移也能解,不得計入遷移效益。**

#### SQ2:monorepo 在 15 人規模、此技術棧下,能解決哪些痛點?

- 痛點 a(原子變更):單一 repo 內一個 PR 即可完成跨服務、跨套件的修改與同步上線。這是 monorepo 的定義性效益【慣例;各主要來源無爭議】(信心:高)。
- 痛點 b(版本飄移):pnpm workspaces 的 workspace 協定讓內部套件以原始碼直接連結,內部不再有「發布版本→各自升級」的環節,飄移在結構上消失【事實|pnpm 官方文件 "Workspace",查證 2026-07】(信心:高)。
- 大公司證據的適用性限制:Google 的 monorepo 效益(原子變更、大規模重構、單一版本)有文獻記載【事實|Potvin & Levenberg, "Why Google Stores Billions of Lines of Code in a Single Repository", Communications of the ACM, 2016】,但同文獻明載這些效益依賴自建 VCS 與專職工具團隊的巨額投資。**15 人團隊不能直接外推 Google 的結論;可外推的只有「原子變更與單一版本策略的機制本身」**【推論】(信心:高)。

**答案:能結構性解決 a 與 b;不能引用大公司案例作為「對我們也划算」的證據。**

#### SQ3:遷移會引入哪些新成本與風險?

| 成本/風險 | 評估 | 證據強度 |
|---|---|---|
| CI 成本上升:不做增量建置,每次 push 觸發全部 11 個專案的建置與測試 | 需要導入 affected/增量建置。Nx 與 Turborepo 均內建「只建置受影響專案」與快取【事實|Nx 官方文件 "Run Only Tasks Affected by a PR"、Turborepo 官方文件 "Caching",查證 2026-07】 | 機制存在:信心高;實際 CI 時間變化:未實測,信心低 |
| git 效能:repo 變大變慢 | 28 萬行距 git 效能瓶頸甚遠。Microsoft 為 Windows 級規模(300GB 工作樹)才需要 VFS for Git【事實|Brian Harry, "The largest Git repo on the planet", Microsoft 部落格, 2017】;git 本身已內建 sparse-checkout 與 partial clone 供大型 repo 使用【事實|git 官方文件,查證 2026-07】 | 信心:高 |
| 權限粒度變粗:GitHub 權限以 repo 為單位,合併後無法對目錄做讀取隔離;審查邊界只能靠 CODEOWNERS 維持【事實|GitHub 官方文件 "About code owners",查證 2026-07】 | 委託方目前無外包或原始碼隔離需求(委託方提供),故非阻斷;列入不適用條件 | 信心:高 |
| 主幹破壞影響面擴大:main 一壞,全團隊受阻 | 反面證據 CE-1(見第 4 節);以 required status checks + 快速 revert 紀律緩解,殘餘風險存在【推論】 | 信心:中 |
| 工具維護負擔:monorepo 工具(Nx/Turborepo)本身需要有人跟版升級 | 估每月數小時級【假設:依同類工具的一般維護經驗,無本案實測】 | 信心:低 |

**答案:主要新增成本是 CI 增量建置的一次性導入與工具維護;git 規模與效能在此量級不構成風險;權限粒度是條件性風險。**

### 3. 建議與條件

#### 3.1 建議內容

合併 11 個 TS/JS repo 為單一 monorepo:pnpm workspaces 管理依賴,Nx 或 Turborepo 提供增量建置(兩者擇一,選型不在本報告範圍);內部共用套件停止發布版本,改為 workspace 原始碼依賴;Terraform infra repo 不併(變更頻率、審查人群與權限模型皆不同【推論】)。

#### 3.2 適用條件(全部成立才動手)

1. 遷移前先在 2-3 個套件上驗證增量建置可運作(見 3.3),驗證失敗則暫停。
2. 建立 CODEOWNERS 與 required status checks,維持既有審查邊界。
3. 團隊全體接受單一版本策略:內部套件不再獨立發版,跨模組改動由同一 PR 負責改到好。
4. 指定一名 monorepo 工具維護 owner(估每月數小時【假設,信心:低】)。

#### 3.3 建議的驗證性下一步(非遷移計畫)

以 3 個共用套件 + 1 個服務建一個試點 workspace,實測:(a) affected 建置是否只跑受影響專案;(b) 單次 CI 時間對比現況。兩週內可完成;結果不如預期即回到替代方案路線。(本段為決策驗證建議,遷移計畫依委託範圍另行處理。)

#### 3.4 不適用條件(任一成立則不建議遷移)

1. 近期將有需要原始碼讀取隔離的外部協作者或客戶交付(E6:GitHub 無目錄級讀取權限)。
2. 無人願任工具維護 owner。
3. 計畫引入非 JS 生態的大型子系統(Go/Python 服務群)——屆時工具選型與本報告前提不同,需重評。

若不遷移:立即導入 reusable workflows(解痛點 c)與 changesets(緩解 b);當「跨 repo PR 佔比連兩季上升」或「版本飄移造成的線上行為不一致事故再次發生」時,重新評估本決策。
