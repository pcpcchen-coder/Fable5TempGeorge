# 劣質解剖析:研究分析(research-analysis)

## 本文件目的

本文件呈現與 `EXEMPLAR.md` 同題(多 repo 是否遷移 monorepo)的一份典型劣質解,並逐點批改。劣質解模擬執行模型(如 Opus)在沒有協議約束時的真實輸出:表面煞有介事、實則踩滿研究分析原型的反模式。目的是讓執行模型學會「錯的長什麼樣」,在自審時能對號入座。

## 如何使用本文件

1. 先讀 `EXEMPLAR.md` 再讀本文件;批改點的「正確做法」欄多處指回黃金示範的具體段落。
2. 注意劣質解的危險之處:它**讀起來很順**。每個批改點都示範如何穿過流暢的行文,檢查證據與結論的對應關係。
3. 自審 RA 類產出時,把第二部分的批改表當作額外檢查清單:自己的產出若有任何一句與「引述」欄的句式相似,即為警訊。
4. AP ID 對照 `ANTIPATTERNS.md` 研究分析(RA)節;Rubric 維度對照 `GENERAL_RUBRIC.md`。

## 第一部分:典型劣質解(全文照錄)

以下是劣質解全文。閱讀時先自己找問題,再對照第二部分。

```markdown
# Monorepo 遷移評估報告

## 一、背景

貴團隊目前有 15 位工程師,程式碼分散在多個 repository 中,團隊反映跨
repo 修改較為麻煩,因此考慮遷移到 monorepo 架構。本報告將分析 monorepo
的優缺點,並給出遷移建議。

## 二、什麼是 Monorepo

Monorepo(單一儲存庫)是一種將多個專案的程式碼存放在同一個版本控制儲存
庫中的軟體開發策略。與之相對的是 polyrepo(多儲存庫)策略,即每個專案
擁有獨立的儲存庫。Monorepo 的概念體現了「單一事實來源」(Single Source
of Truth)的設計哲學,透過統一的程式碼管理實現更高的工程效率。這種模式
近年來隨著 DevOps 文化的興起而受到越來越多的關注。

## 三、Monorepo 的優點

1. 原子提交:跨專案的修改可以在一個 commit 中完成。
2. 統一依賴管理:所有專案共用一套依賴版本,避免版本衝突。
3. 程式碼共用更容易:不需要發布套件即可引用其他專案的程式碼。
4. 根據 2023 年 State of DevOps 報告,採用 monorepo 的團隊部署頻率提升
   了 40%,故障率下降了 25%。
5. Google、Meta、Microsoft 都採用 monorepo,證明這是業界最佳實踐,任何
   規模的團隊都適用。
6. 研究顯示,monorepo 可以顯著提升開發效率與團隊協作品質。

## 四、Monorepo 的缺點

1. Repo 體積會變大,但在現代硬體條件下這不是問題。
2. CI 可能變慢,但只要導入 Bazel 就能解決,Bazel 的增量建置技術已經在
   Google 內部驗證了十幾年,非常成熟。
3. 權限管理較複雜,不過大部分團隊用不到細粒度權限。

## 五、工具選擇

目前主流的 monorepo 工具有:

- Bazel:Google 開源的建置系統,功能最強大。
- Nx:功能豐富的 monorepo 框架。
- Turborepo:Vercel 推出的高效能建置工具。
- Lerna:目前最流行的 JavaScript monorepo 管理工具。
- Rush:微軟開發的可擴充 monorepo 管理器。
- Pants:適合 Python 專案的建置系統。

團隊可以根據自身需求選擇合適的工具。

## 六、遷移計畫

建議按以下 12 週計畫執行遷移:

- 第 1-2 週:盤點現有 repo,制定目錄結構規範。
- 第 3-4 週:搭建 Bazel 建置環境,配置遠端快取伺服器。
- 第 5-6 週:遷移共用函式庫,統一依賴版本。
- 第 7-9 週:逐一遷移各服務,同步改造 CI/CD 流水線,建議同時將 CI 遷移
  至自建 runner 以節省成本。
- 第 10-11 週:遷移前端專案,清理舊 repo。
- 第 12 週:團隊培訓與文件整理,舊 repo 歸檔。

## 七、結論

綜上所述,monorepo 是業界的主流趨勢,擁有原子提交、統一依賴管理等顯著
優勢,Google 等頂尖公司的實踐也證明了其價值。建議貴團隊可以考慮遷移到
monorepo;但具體還是要視團隊實際情況而定,如果團隊覺得目前多 repo 運作
良好,也可以維持現狀。無論如何,瞭解 monorepo 的理念對團隊都是有益的。
```

### 這份劣質解最致命的三個問題

1. **幻覺引用毀掉全部信任(批改 3、5;AP-RA-03)**:一條格式工整、數字精確的假引用,比沒有引用更危險——讀者一旦查證「State of DevOps 報告」發現對不上,整份報告的其餘陳述連帶作廢。RA 原型中這是阻斷級反模式,一票否決。
2. **騎牆結論等於沒有交付(批改 11;AP-RA-08、AP-RA-02)**:委託的核心是「幫我們裁決」,而結論同時說了遷移與不遷移、把「視實際情況」丟回給讀者。全文寫了七節,委託要的那一個判斷仍然不存在。
3. **與委託方情境零連結(批改 1、7、10)**:因為開頭沒有澄清,全文只能是「monorepo 通論」——推薦了不合身的 Bazel、寫了無根據的 12 週計畫,卻自始至終不知道這個團隊有幾個 repo、痛在哪裡。研究分析的價值在「證據 × 情境」的交集;少了情境,再流暢的通論都回答不了「**我們**該遷移嗎」。
