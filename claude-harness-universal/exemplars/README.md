# exemplars/ — 任務原型黃金示範庫

## 本文件目的

本目錄為六種任務原型各提供一組「思考外顯」的完整示範(EXEMPLAR.md)與同題的「典型劣質解+逐點批改」(CRITIQUE.md)。示範把 PLANNING.md 八階段的實際執行過程寫成看得見的文字,是整套 harness 中把協議「翻譯成行為」的部分。

## 如何使用本文件

### 基本用法:few-shot 貼入

1. 判定當前任務的原型(見總 README「任務原型判定」)。
2. 把對應原型的 `EXEMPLAR.md` 全文貼入執行模型(如 Opus)的對話,**貼在任務描述之前**,並加一句指示:「以下是同原型任務的黃金示範,請以相同的階段結構(【澄清】→…→【自審】)與相同的品質標準處理接下來的任務。」
3. 再貼上實際任務描述。

### CRITIQUE.md 的兩種用途

- **開工前預防**:讓執行模型先讀劣質解與批改表,再開工——明確指出「這些行為就是不合格」比抽象規則更有約束力。適合該原型曾經失手的模型。
- **事後對照**:產出完成後,把產出與 CRITIQUE.md 的批改表逐點對照,檢查是否犯了同型錯誤;也可作為評分者的參考答案。

### context 預算不足時的取捨

依序裁剪,砍到放得下為止:

1. 先不貼 CRITIQUE.md(保留 EXEMPLAR.md 全文)。
2. 再裁 EXEMPLAR.md:優先保留【澄清】~【驗收定義】段落(規劃紀律是弱模型最缺的部分),【執行】的完整產出可截短為開頭一段+結構目錄。
3. 最低配置:只貼【澄清】與【方案取捨】兩段+【自審】的自評表——這三段涵蓋「先想後做、留下取捨、誠實自查」三個最關鍵行為。

## 六組示範一覽

| 原型 | 目錄 | 題目 |
|---|---|---|
| 軟體實作(SW) | `software-implementation/` | 實作執行緒安全、支援 TTL 與容量上限(LRU 淘汰)的 in-memory 快取模組(Python,含測試) |
| 系統設計(SD) | `system-design/` | 為中型 SaaS 設計通知服務:站內+Email 雙通道、重試、重複抑制、偏好設定(設計文件) |
| 研究分析(RA) | `research-analysis/` | 15 人工程團隊應否從多 repo 遷移到 monorepo?(含證據強度標註的分析報告) |
| 技術寫作(TW) | `technical-writing/` | 為公開 Orders API 的 v1→v2 破壞性變更撰寫外部開發者遷移指南 |
| 除錯排錯(DB) | `debugging/` | 每天 03:00-03:20 備份時段偶發 502 的系統化除錯(假設→鑑別實驗→根因→防護) |
| 資料分析(DA) | `data-analysis/` | A/B 測試數據應否支持全量上線?(含新奇效應與分段悖論陷阱的判讀) |

每組兩檔:`EXEMPLAR.md`(黃金示範,思考外顯:【澄清】【重述】【範圍】【方案取捨】【分解】【驗收定義】【執行】【自審】,穿插「◆ 為什麼這樣做」註解並引用 AP ID)、`CRITIQUE.md`(逼真的典型劣質解+逐點批改表+R1-R8 總評分)。

## 注意事項

- 示範中的題目、數據、日誌均為虛構教學材料;示範內文已標明。
- 示範的價值在「階段結構與品質標準」,不在題目本身——不要因為當前任務與示範題目領域不同而不貼;原型相同即適用。
- 混合原型的任務,貼「主要交付物」對應的那組;context 允許時可加貼第二組的【澄清】~【驗收定義】段落。
