# 系統設計黃金示範:通知服務(EXEMPLAR)

## 本文件目的

本文件是 system-design(系統設計,SD)原型的黃金示範:以一個真實感的模糊委託為例,完整展示執行模型(如 Opus)依 `PLANNING.md` 八階段產出設計文件的全過程。思考過程全部外顯,並以「◆ 為什麼這樣做」註解標出每一步在防守哪個反模式(AP ID 見 `ANTIPATTERNS.md` 系統設計節)。

## 如何使用本文件

1. 接到系統設計類任務時,把本文件與同目錄 `CRITIQUE.md` 一併作為 few-shot 範例載入,依相同段落標籤與品質水準產出。
2. 段落標籤與順序固定:【澄清】【重述】【範圍】【方案取捨】【分解】【驗收定義】【執行】【自審】,不得增刪調序。
3. 「◆ 為什麼這樣做」是教學註解,實戰產出不需包含;其餘內容即實戰產出應有的樣子。
4. 本示範在【澄清】採「提問 → 模擬使用者回覆 → 繼續」的寫法以便示範完整流程;**實戰中提問後必須真的停下等待回覆,不得替使用者作答**。

## 使用者委託(題目原文)

> 我們的 SaaS 產品(團隊協作工具,中型規模)想加一個通知服務。需求大概是:使用者在站內要看得到通知,重要的也要寄 Email;寄失敗要能重試;不要一直重複轟炸使用者;另外使用者要能自己設定想收哪些通知。麻煩幫我出一份設計文件,要有元件、資料模型、關鍵流程跟失效模式,先不用寫程式。

## 【澄清】

任務描述逐句掃描,標出的模糊詞與缺口:「中型規模」(無數字)、「要看得到」(無時效要求)、「重試」(無次數與上限)、「重複轟炸」(至少兩種語意)、「重要的」(哪些型別要寄 Email 未定義)、完全未提既有系統現況。

**提問(實戰中必須一次問完並停下等待回覆;以下為示範,故接續模擬回覆):**

```markdown
開工前需要確認 5 件事(可只回選項字母):
1. 量級數字:註冊數 / 日活?目前每天寄多少 Email?尖峰集中在何時?
   (為何而問:量級決定要不要引入獨立佇列基礎設施,選型正確性完全取決於數字。)
   A. 提供實際數字  B. 由我以現有 Email 量推估並標為假設 —— 建議:A
2. 時效要求:站內通知多快要看得到?Email 多快要寄出?

3. 「不要重複轟炸」的語意?(直接決定資料模型的核心約束)
   A. 僅防技術性重複(呼叫端重試不得造成重複)  B. A + 同一事件短時間窗內抑制  C. 再加摘要合併(digest) —— 建議:B;C 建議另開任務
4. 通知子系統故障時,可否接受「與產品共命運」?(決定部署形態)
   A. 可,但通知故障不得拖垮主流程  B. 通知需獨立於產品存活 —— 建議:A
5. 現況:目前是否已有寄 Email 的程式?什麼形式?
   (為何而問:設計必須含從現況走到目標的遷移路徑,否則無法實施。)
```

**模擬使用者回覆:** 1. 註冊 6 萬、DAU 約 8 千;目前每天約 5 萬封 Email,尖峰集中在工作日上午;事件總量你推估。2. B。3. B,digest 以後再說。4. A。5. 有,單體應用內約 23 處直接呼叫 SMTP 寄信,失敗就丟,沒有站內通知。

**已確認:**
- 使用者 6 萬 / DAU 8 千;既有 Email 約 5 萬封/日;尖峰在工作日上午。
- 站內 p95 ≤ 60 秒可見;Email p95 ≤ 5 分鐘送出(ESP 受理)。
- 重複抑制語意 = 冪等(防呼叫端重試)+ 同事件 10 分鐘窗抑制;digest 不做。
- 通知與產品共命運,但故障不得拖垮主請求路徑;已受理的通知不得遺失。
- 現況:單體 + 23 處行內 SMTP 直寄,失敗即丟。

**假設(標準句式):**
- 假設:通知事件量以既有 Email 量 × 4 推估為 20 萬件/日、尖峰 70 件/s;若不符請告知,影響範圍是容量驗算一節與 Dispatcher 規模,重算約 30 分鐘。
- 假設:Email 改經外部 ESP(SendGrid / SES 同級)的 HTTPS API 寄送,不自建 SMTP;若不符請告知,影響範圍是 Email Adapter 單一元件的介面。
- 假設:主資料庫為 PostgreSQL 12+(具 `FOR UPDATE SKIP LOCKED` 與部分索引);若不符請告知,影響範圍是佇列領取機制,需改用等價鎖定語法或重評方案 B。
- 假設:抑制時間窗為 10 分鐘;若不符請告知,影響範圍是一個常數與 `suppression_bucket` 計算式。

**待確認問題:** 無(全數已於上輪回覆解決;事件量推估待需求方日後以實測校正,已列入已知限制)。

◆ 為什麼這樣做:委託一個數字都沒給就開始畫架構,是 **AP-SD-03(非功能需求無數字,阻斷級)**。所有選型的對錯都取決於量級——70 件/s 與 7 萬件/s 是兩個完全不同的設計。問不到就推估並「明標假設值」,絕不寫「高吞吐、高可用」這種裸形容詞。

## 【重述】

問題:一個既有的單體 SaaS(6 萬使用者)目前在請求路徑內直寄 Email 且失敗即丟;要新增站內通知,並把通知行為升級為「不遺失、可重試、不重複、尊重偏好」。使用者要的目的是**可靠且不擾人的通知能力**,手段(通知服務)已指定,但服務形態(獨立服務或單體內模組)未指定,由設計裁決。產出物是設計文件:元件、資料模型、關鍵流程、失效模式,不含程式碼。

原型:system-design(SD)。已載入 AP-SD-01 ~ AP-SD-11 反模式清單。

可能誤解點:
1. 「通知」可能被誤解為含行銷群發(campaign)——流量形態與退訂合規完全不同,經澄清問題 3 與範圍界定排除。
2. 「偏好」可能隱含「勿擾時段 / 時區排程」——委託未提,列非目標並明示。

## 【範圍】

目標(每條對應【驗收定義】至少一條):
1. 元件切分與元件間契約(協議、同步性、錯誤傳遞、資料擁有者)。
2. 資料模型(含重複抑制的唯一約束與偏好模型)。
3. 關鍵流程:入列、Email 投遞與重試、站內讀取、偏好變更。
4. 失效模式分析(每個元件與外部依賴的掛掉 / 變慢 / 錯誤行為)。
5. 容量驗算與資料保留策略。
6. 現況描述與從 23 處直寄程式碼走到目標架構的遷移路徑。

非目標(各附理由):
1. SMS / 行動推播通道——委託僅站內 + Email;型別註冊表保留通道欄位作為擴充點,僅此一句。
2. Digest / 摘要合併——澄清問題 3 使用者選 B。
3. 行銷群發——流量形態與合規(退訂)不同,另案處理。
4. Email 退信(bounce)/ 投訴處理管線——影響寄件信譽,建議為第一個後續任務;本設計以 `deliveries.status` 預留狀態擴充。
5. 勿擾時段與時區排程——委託未提;若需要,影響發送時機邏輯,另案。
6. 管理後台 UI 與通知歷史全文搜尋——基本列表查詢已足。

邊界裁定:偏好讀寫 API 屬「使用者要能自己設定」的必要配套 → 納入目標 3。

## 【方案取捨】

候選方案(至少一個「最小改動」候選):

- **方案 A:單體內模組 + PostgreSQL transactional outbox + 輪詢 Dispatcher。** 通知與業務同交易寫入既有 PG,獨立 Dispatcher 行程以 `SKIP LOCKED` 領取並寄送。優點:零新增基礎設施、入列與業務原子一致、可逆。缺點:PG 承擔佇列負載、輪詢延遲下限約 1 秒。成本:小。
- **方案 B:引入訊息佇列(RabbitMQ / Kafka)+ 獨立通知微服務。** 優點:通道擴充與多服務生產者容易、佇列吞吐上限高。缺點:新增有狀態基礎設施與部署單元;業務交易與發佇列存在雙寫不一致問題,實務上仍需 outbox 表,複雜度再疊一層。成本:大。
- **方案 C:維持請求路徑內同步直寄,加程式庫級重試 + Redis 去重。** 優點:改動最小。缺點:重試存活期綁定請求行程;ESP 故障時直接拖慢或失敗主請求。成本:極小。

依 `DECISION_PROTOCOL.md` 第 1 節逐準則裁決:

- **準則 1(正確性):方案 C 淘汰。** ESP 故障時通知遺失(行程內重試隨行程死亡)且拖垮主請求路徑,同時違反已澄清的耐久性(不得遺失)與隔離(不得拖垮主流程)兩條需求;正確性無例外,不再往下比較 C。A 與 B 皆可達成「原子入列 + 至少一次投遞」,打平。
- **準則 2(可回復性):方案 A 勝。** A 僅新增兩張表與一個行程,回復 = 停行程 + 功能旗標切回舊路徑,一步可回;B 引入有狀態 broker,建起來容易收掉難。依協議,裁決於準則 2,停止比較。(附註:準則 3 亦同向——A 新增元件 1 個,B 新增 ≥3 個。)

**選定:方案 A。** 否決紀錄:
- 否決 C:準則 1 正確性不通過,無例外可豁免。
- 否決 B:於本量級(尖峰 70 件/s,設計上限 350 件/s)引入 broker 是為不存在的規模付永久維運成本;且可回復性劣於 A。
- 選定方案的代價(明寫):PG 增加約 30 萬列/日寫入與每秒一次輪詢;站內以輪詢達成、延遲下限 30 秒;未來多服務生產者時 in-process API 不敷使用。
- **失效條件(命中即重評方案 B):** 持續事件率 > 350 件/s;或出現 ≥3 個獨立服務生產者;或 Email 佇列積壓常態 > 10 萬筆。

◆ 為什麼這樣做:只寫一個方案是 **AP-SD-02(單方案直衝)**;教科書式地上 Kafka 微服務是 **AP-SD-01(過度工程)**——70 件/s 用既有 PG 綽綽有餘;只列優點不寫代價與失效條件是 **AP-SD-10(取捨不寫代價)**。三者在此段一次防守。

### 通知服務設計文件

#### §1 現況與量級

現況:產品為單一單體應用 + PostgreSQL;通知行為僅有 Email,由約 23 處業務程式碼在請求路徑內直寄 SMTP,無重試(失敗即丟)、無站內通知、無偏好設定。

| 項目 | 數值 | 來源 |
|---|---|---|
| 註冊使用者 / DAU | 60,000 / 8,000 | 需求方提供 |
| 既有 Email 量 | 約 50,000 封/日 | 需求方提供 |
| 通知事件量(上線後) | 200,000 件/日(平均 2.3 件/s) | 假設值:既有 Email 量 × 4 |
| 尖峰事件率 | 70 件/s | 假設值:平均 × 30(工作日上午集中) |
| 設計上限 | 350 件/s | 尖峰 × 5(落在 2-10 倍餘裕規則內) |

| Email 送出延遲 | p95 ≤ 5 min(至 ESP 受理) | 需求方提供 |
| 耐久性 | 已受理(交易 commit)之通知不得遺失;投遞語意為至少一次 | 需求方提供 |
| 可用性 | 與產品共命運;通知故障不得拖垮主請求路徑 | 需求方提供 |

#### §2 元件總覽與命名

```text
┌────────────────────── 既有單體(Monolith)──────────────────────┐
│  業務程式碼 ──呼叫──> Notify API(程序內模組)                    │
│                        │ 同一 DB 交易:冪等寫入+抑制+讀偏好       │
│                        v                                        │
│  In-App API ──讀──> Notification Store(PostgreSQL:              │
│  Preference API ──寫─> notifications / deliveries / preferences)│
└──────────┬─────────────────────────┬────────────────────────────┘
     HTTPS,30s 輪詢            SQL,SKIP LOCKED 領取(1s 輪詢)
       前端(Web)              Dispatcher(獨立行程 × 2)
                                     │ HTTPS,timeout 10s,至少一次
                                     v
                              Email Adapter ──> ESP(外部郵件服務)
```

元件命名表(全文僅使用此名,禁止別名):

| 名稱 | 職責 | 部署形態 |
|---|---|---|
| Notify API | 供業務程式碼呼叫的入列介面(冪等、抑制、偏好展開) | 單體內模組 |
| Notification Store | 通知事實與投遞狀態的唯一真相(PostgreSQL) | 既有資料庫 |
| Dispatcher | 領取 pending Email 投遞、執行重試狀態機 | 獨立行程 × 2 |
| Email Adapter | 封裝 ESP API 呼叫與錯誤分類 | Dispatcher 內模組 |
| In-App API | 站內通知列表 / 未讀數 / 已讀回寫 | 單體內 HTTP 端點 |
| Preference API | 偏好讀寫(mandatory 型別拒絕停用) | 單體內 HTTP 端點 |
| ESP | 外部郵件遞送服務 | 外部依賴 |

◆ 為什麼這樣做:命名表是 **AP-SD-11(圖文不一致)** 的防守——同一元件在圖、表、內文只有一個名字,讀者不必猜 Gateway 和 Proxy 是不是同一個東西。

#### §3 資料模型

```sql
CREATE TABLE notifications (
  id                 UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id            BIGINT      NOT NULL REFERENCES users(id),
  type               TEXT        NOT NULL,               -- 型別註冊表中的 slug
  entity_ref         TEXT        NOT NULL DEFAULT '',    -- 業務實體參照,如 'task:123'
  payload            JSONB       NOT NULL DEFAULT '{}',
  idempotency_key    TEXT        NOT NULL,               -- 呼叫端提供,通常為事件 ID
  suppression_bucket BIGINT      NOT NULL,               -- floor(epoch / 600):10 分鐘桶
  created_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT uq_notifications_idem     UNIQUE (idempotency_key),
  CONSTRAINT uq_notifications_suppress UNIQUE (user_id, type, entity_ref, suppression_bucket)
);

CREATE TABLE deliveries (
  id              BIGSERIAL PRIMARY KEY,
  notification_id UUID   NOT NULL REFERENCES notifications(id),
  user_id         BIGINT NOT NULL,   -- 反正規化:值不可變,供未讀查詢免 join
  channel         TEXT   NOT NULL CHECK (channel IN ('in_app','email')),
  status          TEXT   NOT NULL DEFAULT 'pending'
                  CHECK (status IN ('pending','sent','dead','skipped')),
  attempt_count   INT    NOT NULL DEFAULT 0,
  next_attempt_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  last_error      TEXT,
  read_at         TIMESTAMPTZ,       -- 僅 in_app 使用
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT uq_deliveries_channel UNIQUE (notification_id, channel)
);
CREATE INDEX idx_deliveries_pending ON deliveries (next_attempt_at)
  WHERE status = 'pending' AND channel = 'email';
CREATE INDEX idx_deliveries_unread  ON deliveries (user_id)
  WHERE channel = 'in_app' AND read_at IS NULL;

CREATE TABLE notification_preferences (
  user_id    BIGINT  NOT NULL REFERENCES users(id),
  type       TEXT    NOT NULL,
  channel    TEXT    NOT NULL,
  enabled    BOOLEAN NOT NULL,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, type, channel)   -- 無列 = 預設啟用
);
```

型別註冊表(存於程式碼、版本控制,非資料庫):每個型別定義 slug、預設通道、`mandatory` 旗標(安全 / 帳務類不可停用)、樣板參照。新增通知型別 = 加一筆註冊 + 樣板,不改 schema。

重複抑制的兩層語意與其地基:
- **冪等**:`uq_notifications_idem`——呼叫端重試(同一 `idempotency_key`)絕不產生第二列。
- **時間窗抑制**:`uq_notifications_suppress`——同(使用者, 型別, 實體)在同一個 10 分鐘桶內只產生一列。已知邊界:桶是固定切割而非滑動窗,跨桶邊界 1 秒的兩則重複不會被抑制;代價是省去一次鎖定查詢或外部狀態。後果輕微(偶發多收一則),明示接受。

◆ 為什麼這樣做:抑制若靠「發送前查一下 Redis」,真相來源就漂移到易失性儲存,Redis 重啟即重複風暴——這是 **AP-SD-09(忽略資料一致性,阻斷級)** 的典型踩法。正確做法是把去重落在唯一真相(Notification Store)的資料庫唯一約束上,由 DDL 保證而非流程紀律保證。

#### §4 元件契約與資料擁有者

| # | 連線 | 協議 | 同步性 | 關鍵資料 | 錯誤傳遞 |
|---|---|---|---|---|---|
| 1 | 業務程式碼 → Notify API | 程序內函式 | 同步(同一 DB 交易) | user_id, type, entity_ref, payload, idempotency_key | 例外拋回呼叫端;業務交易回滾時通知一併回滾 |
| 2 | Notify API → Notification Store | SQL(同交易) | 同步 | notifications、deliveries 插入 | 唯一約束衝突轉為回傳值 `duplicate` / `suppressed` |
| 3 | Dispatcher → Notification Store | SQL 輪詢(間隔 1 s) | 非同步(拉式) | pending 投遞批次(≤100 列) | 連線失敗:本輪放棄下輪重領;行程死亡:列鎖自動釋放 |
| 4 | Dispatcher → Email Adapter → ESP | HTTPS API | 同步呼叫,timeout 10 s | 收件地址、渲染內容、idempotency key(若 ESP 支援) | 暫時性錯誤 → 重試排程;永久性錯誤 → `dead` |
| 5 | 前端 → In-App API | HTTPS REST,30 s 輪詢 | 同步請求 | 列表、未讀數、已讀回寫 | HTTP 4xx/5xx;前端退避重試 |
| 6 | 產品設定頁 → Preference API | HTTPS REST | 同步 | preferences upsert | 驗證錯 4xx;mandatory 型別停用回 409 |

| 資料 | 唯一擁有者 | 複本與一致性機制 |
|---|---|---|
| notifications / deliveries | Notification Store | `deliveries.user_id` 為反正規化欄位,值不可變,無失效問題 |
| notification_preferences | Notification Store(Preference API 為唯一寫入口) | 無快取複本;每次主鍵查詢直讀 DB |
| Email 受理後狀態(送達 / 退信) | ESP | 本系統僅記錄「ESP 已受理」;退信不回流(非目標 4) |
| 型別註冊表 | 程式碼(版本控制) | 隨部署生效,無執行期複本 |

外部依賴特性查證:PostgreSQL `SELECT ... FOR UPDATE SKIP LOCKED` 為 9.5+ 官方功能(出處:PostgreSQL 官方文件 SELECT 章節)。ESP 是否支援寄送 idempotency key:**未確認**——以選定 ESP 的文件為準;若不支援,退路是接受「至少一次」語意下的罕見重複 Email(頻率上界見 §6)。

◆ 為什麼這樣做:方塊圖上每條線都標了協議、同步性與錯誤走向,防 **AP-SD-07(方塊圖沒有契約)**;ESP 特性查不到就標「未確認 + 退路」而不是寫成事實,防 **AP-SD-04(幻覺元件能力,阻斷級)**。

#### §5 關鍵流程

**流程 A:入列(業務程式碼 → Notification Store)**
1. 業務程式碼在自身資料庫交易內呼叫 `notify(user_id, type, entity_ref, payload, idempotency_key)`。
2. Notify API 驗證 `type` 存在於型別註冊表,否則立即拋例外(fail fast,防打錯字的型別默默丟失)。
3. `INSERT INTO notifications ... ON CONFLICT DO NOTHING`;插入 0 列時以 `idempotency_key` 回查:查得到 → 回 `duplicate`(冪等重試,視為成功);查不到 → 回 `suppressed`(時間窗抑制)。
4. 讀 `notification_preferences` 決定通道集合;`mandatory` 型別忽略偏好。
5. 站內:插入 `deliveries(channel='in_app', status='sent')`——站內投遞即寫入本身,不經 Dispatcher。Email:若啟用,插入 `deliveries(channel='email', status='pending', next_attempt_at=now())`;停用則插入 `status='skipped'` 留審計軌跡。
6. 隨業務交易一起 commit:業務成功與通知入列原子一致(transactional outbox 性質);業務回滾則通知不存在。

**流程 B:Email 投遞與重試(Dispatcher)**
1. 每 1 秒執行:`SELECT ... FROM deliveries WHERE channel='email' AND status='pending' AND next_attempt_at <= now() ORDER BY next_attempt_at LIMIT 100 FOR UPDATE SKIP LOCKED`。
2. 逐筆載入通知、收件人與樣板並渲染。
3. 永久性錯誤(樣板缺失、地址格式無效)→ `status='dead'` + `last_error` + 告警計數;不重試(防毒藥訊息霸佔佇列)。
4. **發送前複查偏好**(主鍵一次查詢):若使用者已於入列後停用該型別 Email,改標 `skipped`。
5. 呼叫 Email Adapter → ESP(timeout 10 s)。成功 → `sent`。暫時性錯誤(5xx / 逾時)→ `attempt_count + 1`;超過 5 次 → `dead` + 告警;否則 `next_attempt_at = now() + backoff(attempt_count)`,backoff 序列 = 1 分、5 分、30 分、2 小時、8 小時。
6. 行程當機語意:未 commit 的狀態回復為 `pending`、列鎖釋放,他機重領 → 至少一次;已送 ESP 未記錄者會重送(上界:單批 ≤100 筆,見 §6)。

**流程 C:站內讀取(前端)**
1. `GET /api/notifications?cursor&limit=20`:以 `idx_deliveries_unread` 與 `created_at` 分頁。
2. `GET /api/notifications/unread_count`:30 秒輪詢,滿足 p95 ≤ 60 s 可見。
3. `POST /api/notifications/{id}/read`:寫 `read_at`,冪等(重複標記無害)。

**流程 D:偏好變更**
1. Preference API upsert 一列;`mandatory` 型別回 409 拒絕。
2. 生效語意:入列時展開通道 + 發送時複查(流程 B 步驟 4);已在佇列中的 Email 於下次投遞嘗試時即反映新偏好。代價:每封多一次主鍵查詢;換得 ESP 長時間故障期間不會寄出「使用者已停用」的通知。

#### §6 失效模式分析

| 元件 / 依賴 | 情境 | 系統行為 | 可接受? | 緩解 |
|---|---|---|---|---|
| PostgreSQL | 停機 | 產品整體停擺,通知同停;恢復後與業務交易一致,無半套狀態 | 可(共命運,澄清 4) | 沿用產品既有 PG HA 與備份 |
| PostgreSQL | 變慢 | 入列在業務交易內,延遲直接反映到主請求 | 邊界 | 入列僅 2-3 筆 INSERT + 1 次主鍵讀,常數成本;監控業務交易新增耗時 |
| Dispatcher | 全數當機 | Email 積壓、不遺失;站內完全不受影響 | 可(短時) | oldest-pending 年齡告警(> 10 分);supervisor 自動重啟 |
| Dispatcher | 送 ESP 後、寫回前當機 | 該批重送 → 罕見重複 Email(至少一次語意) | 可 | ESP idempotency key(未確認);重複上界 = 在途單批 ≤100 筆 |
| ESP | 全面故障(5xx / 逾時) | backoff 重試至第 5 次;之後 `dead` + 告警,可人工重放(§8 附程序) | 可(以延遲換不遺失) | 重試序列總跨度約 10.5 小時,涵蓋絕大多數故障窗 |
| ESP | 變慢 | 10 s timeout 視為暫時性失敗;吞吐下降 → 積壓告警 | 可 | worker 數與併發數可線上調整 |
| ESP | 受理後退信(地址失效) | 不處理(非目標 4),寄件信譽隨時間下降 | 明示不處理 | 建議為第一個後續任務;`status` 已預留擴充 |
| 樣板 / 資料 | 渲染失敗、地址格式無效 | 永久性錯誤:不重試,直接 `dead` + 告警 | 可 | 與暫時性錯誤分類處理,防重試佇列被毒藥訊息佔滿 |
| 前端輪詢 | 尖峰負載 | 約 67 req/s(§7),皆為部分索引單點查詢 | 可 | 超出時改 SSE(擴充點,本次不設計) |
| 事件風暴 | 持續 > 350 件/s | 入列仍成功(短時超載 PG 可吸收),Email 積壓拉長 | 邊界 | 命中選型失效條件 → 依【方案取捨】重評方案 B |

◆ 為什麼這樣做:成功路徑人人會畫;這張表逐一回答「它掛了會怎樣」,是 **AP-SD-06(無失敗模式分析,阻斷級)** 的直接防守。注意「不處理」也是一種明寫的決定(退信列),而非沉默缺口。

#### §7 容量驗算與資料保留

- 寫入:200,000 件/日 × 平均 1.5 列投遞 = 約 300,000 列/日;尖峰 70 件/s × 2.5 列 ≈ 175 列寫入/s,遠低於既有 PG 單機能力(數千小寫入/s 量級)。
- Dispatcher 吞吐:每 worker 10 併發 × ESP 平均 300 ms ≈ 33 封/s;2 workers ≈ 66 封/s ≥ Email 尖峰約 35 封/s(尖峰 70 件/s × Email 佔比約 0.5),餘裕約 1.9 倍;worker 無狀態,線性加行程即可擴充(SKIP LOCKED 在 ≤10 行程下競爭可忽略——慣例經驗值,未實測,列入已知限制)。
- 輪詢負載:DAU 8,000、同時在線峰值以 25% 估 ≈ 2,000 sessions ÷ 30 s ≈ 67 req/s,皆走部分索引。
- 保留:90 天 × 300,000 列/日 ≈ 2,700 萬列 deliveries——單表 + 部分索引即可,**不做分割表**;每夜批次刪除(單批 10,000 列)。重評條件:保留量 > 1.5 億列,或刪除批次影響 p99 延遲。

◆ 為什麼這樣做:「要不要分割表」用數字回答而不是用「大資料就該分割」的名詞反射回答,防 **AP-SD-08(名詞代替決策)** 與 AP-SD-01;每個「不做」都附了重評條件。

| 階段 | 內容 | 驗證方式 | 回滾 |
