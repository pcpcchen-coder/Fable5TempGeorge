# 系統設計批改示範:通知服務(CRITIQUE)

## 本文件目的

本文件呈現與 `EXEMPLAR.md` 同一委託的「典型劣質解」,並逐點批改。劣質解模擬較弱執行模型的真實輸出:表面煞有介事(有架構圖、有名詞、有章節),實際上跳過澄清、只畫成功路徑、堆砌元件。目的是讓執行模型(如 Opus)學會辨認並避免這些輸出形態。

## 如何使用本文件

1. 與 `EXEMPLAR.md` 成對使用:先讀黃金示範,再讀本文件,對照同一委託的兩種產出差距。
2. 自審時把「逐點批改表」當成負面檢核清單:自己的產出若出現任一相同形態,對應維度依表列扣分邏輯處理。
3. 批改表的 AP ID 對應 `ANTIPATTERNS.md` 系統設計節;Rubric 維度對應 `GENERAL_RUBRIC.md`。
4. 注意:劣質解的問題**不在文筆**——它讀起來很流暢;問題在每個關鍵決定都沒有根據。批改時盯決策鏈,不盯修辭。

## 委託原文(與 EXEMPLAR.md 相同)

> 我們的 SaaS 產品(團隊協作工具,中型規模)想加一個通知服務。需求大概是:使用者在站內要看得到通知,重要的也要寄 Email;寄失敗要能重試;不要一直重複轟炸使用者;另外使用者要能自己設定想收哪些通知。麻煩幫我出一份設計文件,要有元件、資料模型、關鍵流程跟失效模式,先不用寫程式。

## 一、典型劣質解(全文照錄)

````markdown
# 通知系統設計文件

## 1. 概述

本文件描述一個高可用、高吞吐、可水平擴展的分散式通知系統。系統採用微服務架構與
事件驅動設計,實現了各元件間的完全解耦,可支撐千萬級用戶規模,並為未來的業務增長
預留了充分的擴展空間。

## 2. 整體架構

    業務系統 → API Gateway → Notification Service → Kafka
                                                      ├→ Email Service → SMTP
                                                      ├→ In-App Service → WebSocket
                                                      ├→ SMS Service
                                                      └→ Push Service
    User Preference Service(MongoDB + Redis 緩存)
    Template Service(樣板管理)
    Analytics Service(Elasticsearch)

架構說明:
- API Gateway:統一入口,負責鑑權與限流。
- Notification Service:接收通知請求,去重後寫入 Kafka。
- Kafka:訊息佇列,exactly-once 語意確保通知不重複投遞。
- Email / In-App / SMS / Push Service:四個通道的獨立消費者服務,各自水平擴展。
- User Preference Service:管理用戶偏好,資料存 MongoDB,並緩存於 Redis,每日同步一次。
- Template Service:集中管理通知樣板,支援多語言。
- Analytics Service:通知數據寫入 Elasticsearch,支援即時分析與報表。

採用事件驅動架構,實現了系統的完全解耦與高擴展性,各服務可獨立部署、獨立擴容。

## 3. 資料模型

notifications 表(MySQL):
| 欄位 | 型別 |
|---|---|
| id | bigint |
| user_id | bigint |
| content | text |
| channel | varchar |
| status | varchar |
| created_at | datetime |

user_preferences(MongoDB):
{ user_id, email_enabled, in_app_enabled, sms_enabled, push_enabled }

## 4. 關鍵流程

1. 業務系統調用 API Gateway 發送通知請求。
2. 訊息中心(Notification Service)以 user_id + 通知內容的 MD5 作為去重 key 查詢
   Redis(TTL 24 小時),若已存在則直接丟棄,實現重複抑制。
3. 通過去重後寫入 Kafka 對應通道的 topic。
4. 各通道服務消費訊息,調用 User Preference Service 查詢用戶偏好,若用戶已關閉
   該通道則跳過。
5. 調用 Template Service 渲染樣板,執行發送。
6. 發送結果寫入 Analytics Service,供後續分析。

## 5. 可靠性設計

- Kafka 的 exactly-once 語意可確保每則通知只會被投遞一次,天然解決重複發送問題。
- 若通道發送失敗,立即重試 3 次,仍失敗則寫入錯誤日誌,運維人員可透過日誌排查。
- 通知中心與各通道服務均為無狀態服務,可水平擴展,系統無單點故障。

## 6. 擴展性設計

- 本設計已預先實作簡訊(SMS)與行動推播(Push)通道,未來業務需要時可直接啟用。
- 內建 AI 智能發送時間優化模組,根據用戶行為分析最佳觸達時間,提升開信率。
- 支援 A/B 測試框架,可對通知文案進行分組實驗,持續優化點擊率。

## 7. 總結

本架構具備高可用、高吞吐、鬆耦合、易擴展等優點,無明顯缺點,已在多個大型系統中
得到驗證,可支撐千萬級用戶,建議直接進入開發階段。
````

## 四、這份劣質解最致命的三個問題

1. **核心需求押在不存在的保證上(批改點 2、3、12)**:重複抑制靠「Kafka exactly-once + Redis 內容雜湊」,前者的保證邊界被誤解、後者在 Redis 重啟時歸零且會誤殺合法通知;偏好則有 24 小時的靜默不一致窗。委託的四項需求中,兩項(重複抑制、偏好)在正常營運事件下即失效——而文件把它們當已解決。
2. **零數字,所以一切選型皆不可審查(批改點 1、6)**:沒有任何量級、延遲、可用性數字,8 服務 + 5 儲存的架構無從判斷必要性;對照實際量級(DAU 8 千、尖峰約 70 件/s),幾乎整棟架構都是為想像中的規模建造的。沒有數字的設計文件不是保守,是不可證偽。
3. **只有成功路徑,沒有可實施性(批改點 4、5、9)**:重試三次即靜默遺失、八個元件無一被問過「掛了會怎樣」、對既有 23 處直寄程式碼與遷移隻字未提。這份文件即使架構合理也無法落地——它描述的是一張海報,不是一個可以從今天的系統走過去的目標。
