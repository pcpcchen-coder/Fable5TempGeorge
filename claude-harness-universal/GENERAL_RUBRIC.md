# GENERAL_RUBRIC.md — 通用品質評分標準

## 本文件目的

本文件定義跨領域通用的八維度品質 Rubric(R1-R8),適用於六種任務原型(軟體實作、系統設計、研究分析、技術寫作、除錯排錯、資料分析)的任何產出物。每個維度都以「可從產出物直接檢查的行為」定義 1 / 3 / 5 分錨點,消除主觀形容詞,讓執行模型(如 Opus)與外部評分者能得到一致的評分結果。

## 如何使用本文件

| 使用時機 | 使用方式 |
|---|---|
| 執行前(規劃階段) | 在 PLANNING.md 的「定義驗收」階段,把 R1-R8 各維度的 5 分行為納入驗收條件 |
| 交付前(自審階段) | 複製本文件〈交付前自評指令〉一節給執行模型,逐維度自評並修正低於 3 分的維度 |
| 交付後(外部評分) | 複製本文件〈Grader Prompt〉一節(該節自包含,可單獨使用)貼給任何模型評分 |
| 維度互相牽制時 | 依〈維度衝突時的優先序〉一節的規則裁決,不得自行發明優先序 |

評分規則:每維度 1-5 分整數。1 / 3 / 5 分有明確錨點;2 分 = 介於 1 與 3 之間(已脫離 1 分的失敗狀態,但未達 3 分的完整行為);4 分 = 介於 3 與 5 之間(具備全部 3 分行為,且部分達到 5 分行為)。

## 分數解讀與交付門檻

| 規則 | 內容 |
|---|---|
| 交付門檻 | 全部八維度 >= 3 分才可交付。以「最低分維度」為準,禁止用平均分掩蓋單一維度的失敗(例如七個 5 分加一個 1 分不等於可交付) |
| 分數意義 | 1-2 分 = 該維度失敗,必須修正;3 分 = 可交付的最低標準;4-5 分 = 超出最低標準,適用於高風險或會被長期維護的產出物 |
| 目標設定 | 在 PLANNING.md 的「定義驗收」階段,依任務風險為各維度設定目標分(預設全維度 3 分;高風險任務將重點維度提高到 4-5 分),寫入驗收條件 |
| 不可比較 | 分數只用於「同一產出物是否達標」的判斷,不用於跨任務、跨模型的橫向排名——不同任務的錨點落點難度不同 |

## 各任務原型的重點維度

八維度適用於全部六種原型,但各原型有「最常失敗、需優先投入」的重點維度。自評與評分時全部八維度都要評;規劃時把重點維度的目標分設為 4 以上。

| 原型 | 重點維度 | 原因 |
|---|---|---|
| software-implementation(軟體實作) | R1、R4、R5 | 程式碼的錯誤與邊界缺口成本最高,且最容易用測試建立可驗證性 |
| system-design(系統設計) | R2、R4、R7 | 設計文件的價值在於涵蓋面、失敗情境分析與他人能否據以實作 |
| research-analysis(研究分析) | R1、R5、R8 | 研究產出的核心風險是錯誤結論、無法溯源的主張與誇大信心 |
| technical-writing(技術寫作) | R1、R6、R7 | 文件的錯誤指令直接害人;冗長與結構混亂使文件不被使用 |
| debugging(除錯排錯) | R1、R3、R5 | 根因判斷必須正確且可重現驗證;修錯時最容易夾帶無關變更 |
| data-analysis(資料分析) | R1、R5、R8 | 數字錯誤與不可重現的分析流程會讓結論整體作廢;信心程度必須如實 |

## 八維度定義與評分錨點

各維度的通用範例統一使用兩個跨領域情境,方便對照:
- 情境 A:「寫一份資料庫遷移指南」(技術寫作 / 系統設計類產出)
- 情境 B:「為 API 服務實作快取層」(軟體實作類產出)

### R1 正確性

一句話定義:產出物中的事實陳述、程式行為、計算結果與引用內容,經檢查後與可驗證的事實一致。

| 分數 | 可觀察行為 |
|---|---|
| 1 分 | 產出物含有至少一項核心錯誤:程式主要路徑執行結果錯誤、關鍵事實陳述與來源矛盾、計算結果驗算不符,且該錯誤直接影響產出物的主要用途 |
| 3 分 | 主要路徑與核心陳述經檢查無誤;存在次要瑕疵(如非關鍵參數寫錯、邊緣描述不精確),但不影響主要用途 |
| 5 分 | 全部陳述與行為經檢查無誤,且產出物本身附帶了正確性的佐證(測試輸出、驗算過程、來源引用),檢查者不需自行重建驗證環境即可確認 |

範例(情境 A,遷移指南):1 分 = 指南中的遷移指令有語法錯誤,照做會失敗;3 分 = 主要遷移步驟正確可執行,但某個選用參數的預設值寫錯;5 分 = 所有指令正確,且每個步驟附上執行後的預期輸出供讀者比對。

範例(情境 B,快取層):1 分 = 快取回傳過期資料且永不失效,主路徑行為錯誤;3 分 = 讀寫與失效邏輯正確,但某個 log 訊息描述與實際行為不符;5 分 = 邏輯正確且附上通過的單元測試輸出,涵蓋命中、未命中、失效三種路徑。

### R2 完整性

一句話定義:任務要求的每一項交付物與每一個明說的需求,在產出物中都有對應的實作或內容,沒有靜默遺漏。

| 分數 | 可觀察行為 |
|---|---|
| 1 分 | 任務明說的需求中,至少一項在產出物中完全找不到對應內容,且產出物未說明為何缺少 |
| 3 分 | 每項明說需求都能在產出物中指出對應位置;隱含的必要配套(如文件更新、錯誤路徑)有部分缺漏,但已在產出物中列出缺漏清單 |
| 5 分 | 明說需求全數有對應內容,隱含的必要配套也已完成或明確標註「不做及理由」;可逐條建立「需求 → 產出位置」的對照且無空格 |

範例(情境 A):1 分 = 任務要求涵蓋「回滾程序」,指南中完全沒有回滾章節也未說明;3 分 = 正向遷移步驟齊全,回滾章節簡略但文末明列「回滾章節的已知缺漏清單」;5 分 = 遷移、回滾、失敗中斷後的續傳程序全數涵蓋,並附需求對照表。

範例(情境 B):1 分 = 任務要求「快取失效機制」,程式碼中沒有任何失效邏輯;3 分 = 讀寫與 TTL 失效齊全,主動清除 API 未做但已在交付說明中列出;5 分 = 所有要求的機制齊全,並逐條對應任務需求列出實作位置(檔案與函式名)。

### R3 範圍紀律

一句話定義:產出物只包含任務要求的變更與內容,不夾帶未被要求的重構、功能、風格改動或離題內容。

| 分數 | 可觀察行為 |
|---|---|
| 1 分 | 產出物中存在大量與任務無關的變更或內容(如順手重構了無關模組、加入未被要求的功能),且未經任何說明或核准 |
| 3 分 | 產出物以任務範圍為主體;存在少量超出範圍的小改動,但每一處都在產出物中標註「超出範圍,理由如下」 |
| 5 分 | 逐一檢查每個變更/章節都能對應到任務需求;發現的範圍外問題以「另列清單建議後續處理」方式回報,而非直接動手改 |

範例(情境 A):1 分 = 遷移指南中夾了三頁重新設計整個資料庫 schema 的提案;3 分 = 指南聚焦遷移,附帶修正了兩處舊文件錯字並標註「順手修正,超出範圍」;5 分 = 指南只含遷移內容,文末另列「遷移過程中發現的既有文件問題」建議清單。

範例(情境 B):1 分 = 實作快取時順便改寫了認證模組與程式碼風格,diff 中無關變更多於快取本身;3 分 = 以快取為主,附帶改了一個擋路的函式簽名並在說明中標註理由;5 分 = diff 只含快取相關檔案,發現的無關壞味道寫成 issue 建議而非直接改。

### R4 邊界處理

一句話定義:產出物明確處理或明確聲明了異常輸入、失敗路徑、極端條件與例外情境,而非只覆蓋理想路徑。

| 分數 | 可觀察行為 |
|---|---|
| 1 分 | 產出物只覆蓋理想路徑;可立即舉出一個常見異常情境(空輸入、依賴失敗、資料缺漏),產出物對它既無處理也無任何提及 |
| 3 分 | 常見的異常情境(至少:無效輸入、外部依賴失敗)有明確處理或明確聲明「不處理及後果」;罕見極端情境未涵蓋 |
| 5 分 | 產出物內含一份邊界情境清單(輸入邊界、狀態邊界、依賴失敗、並發/時序等適用類別),每項標明「已處理(位置)」或「不處理(理由與後果)」 |

範例(情境 A):1 分 = 指南假設遷移一次成功,對「中途失敗」隻字未提;3 分 = 指南含「遷移失敗時的回滾步驟」,但未討論部分成功的中間狀態;5 分 = 指南列出失敗情境表(連線中斷、鎖等待逾時、資料不相容),每種都有對應處置步驟。

範例(情境 B):1 分 = 快取後端連不上時整個服務直接崩潰,程式碼無任何錯誤處理;3 分 = 快取失敗時降級為直接查詢來源並記 log,但未處理快取雪崩;5 分 = 程式碼與說明涵蓋後端斷線、序列化失敗、key 過長、併發寫入,每項有處理或明確聲明不處理的理由。

### R5 可驗證性

一句話定義:第三方可依產出物提供的步驟、測試或證據,在合理成本內獨立確認產出物是否達成任務目標。

| 分數 | 可觀察行為 |
|---|---|
| 1 分 | 產出物只有結論沒有驗證途徑:無測試、無可重跑的步驟、無來源引用,檢查者只能選擇相信或全部重做 |
| 3 分 | 產出物附有可執行的驗證途徑(測試指令、重現步驟、來源連結)且照做可得結果,但覆蓋不全或需要檢查者自行補環境細節 |
| 5 分 | 產出物附有一鍵或逐步可重跑的完整驗證程序,含預期結果供比對;每個關鍵主張都能對應到一個可獨立執行的檢查點 |

範例(情境 A):1 分 = 指南宣稱「遷移後資料一致」但沒給任何檢查方法;3 分 = 指南提供遷移後的抽查 SQL,但未涵蓋所有資料表;5 分 = 指南附完整的前後筆數與校驗和比對腳本,以及每步驟的預期輸出。

範例(情境 B):1 分 = 交付說明寫「已測試通過」但 repo 中找不到任何測試;3 分 = 附單元測試與執行指令,但整合情境需自行架設快取後端且步驟未寫;5 分 = 附單元與整合測試、一行指令可全部執行、並附本次執行的通過輸出。

### R6 簡潔性

一句話定義:產出物在滿足需求的前提下,不含冗餘內容、重複邏輯或非必要的抽象層,規模與任務複雜度相稱。

| 分數 | 可觀察行為 |
|---|---|
| 1 分 | 可指出大段可直接刪除而不影響任何需求的內容:重複貼上的段落、為單一用途建的多層抽象、與結論無關的長篇鋪陳 |
| 3 分 | 主體精煉;存在少量可合併的重複或略深的抽象,刪改它們可縮減篇幅但需要小幅重組 |
| 5 分 | 逐段/逐函式檢查,找不到可整段刪除而不損失需求覆蓋的內容;相同邏輯只出現一次;抽象層數與實際變化點一一對應 |

範例(情境 A):1 分 = 指南前六頁是資料庫發展史,與遷移操作無關;3 分 = 內容聚焦,但三個章節各自重複解釋了一次備份指令;5 分 = 每個章節直接進入操作,共用內容抽成一節被引用,無重複。

範例(情境 B):1 分 = 為單一 Redis 後端建了四層介面與工廠類別,實際只有一種實作;3 分 = 結構合理,但讀與寫路徑各自複製了一份序列化邏輯;5 分 = 單一快取介面對應現有的一種後端,序列化邏輯只有一份,總行數與功能相稱。

### R7 可維護性

一句話定義:未接觸過此產出物的人,能依其結構、命名與說明,在不詢問原作者的情況下理解、修改與擴充它。

| 分數 | 可觀察行為 |
|---|---|
| 1 分 | 產出物缺乏可辨識的結構:無意義命名(如 tmp、data2)、無任何說明關鍵決定的註解或章節、修改任一處都需要先逆向理解全貌 |
| 3 分 | 結構與命名可自我解釋,關鍵的非顯然決定附有理由說明;但部分區塊間的耦合或前後依賴需要讀者自行推敲 |
| 5 分 | 新讀者可依產出物內的導覽(目錄、模組說明、決定記錄)直接定位到任一部分進行修改;每個非顯然的取捨都留有一句理由;模仿既有慣例即可正確擴充 |

範例(情境 A):1 分 = 指南是一整篇無標題的流水帳,步驟間依賴關係未說明;3 分 = 章節分明、步驟編號,但「為何選此遷移策略」未說明;5 分 = 有目錄、每章開頭說明前置條件、附「策略選擇理由」一節,他人可直接改寫其中一章。

範例(情境 B):1 分 = 單一 800 行函式、變數名為 a/b/c、無註解;3 分 = 模組切分與命名清楚,但 TTL 常數的選值理由沒寫;5 分 = 模組職責單一、命名遵循 codebase 慣例、非顯然常數與取捨附註解、新增一種後端只需模仿現有實作。

### R8 誠實度

一句話定義:產出物如實揭露自身的限制、未完成項、不確定之處與失敗嘗試,聲稱的完成度與實際狀態一致。

| 分數 | 可觀察行為 |
|---|---|
| 1 分 | 產出物聲稱「全部完成/測試通過/已驗證」,但檢查後發現有未完成項、失敗的測試或未執行的驗證,聲稱與實況不符 |
| 3 分 | 聲稱與實況一致;主要限制有揭露,但部分低信心的推測未標註信心程度,讀者無法區分「已驗證」與「合理推測」 |
| 5 分 | 產出物含明確的「限制與未驗證事項」段落;每個主張可辨識為「已驗證(附證據)」或「推測(附信心程度)」;做過但失敗的嘗試有記錄,無任何誇大 |

範例(情境 A):1 分 = 指南宣稱「已在生產環境驗證」,實際只在空資料庫試過;3 分 = 如實說明只在測試環境驗證過,但對大表遷移耗時的估計未標明是推算值;5 分 = 明確區分「實測過的步驟」與「依文件推斷的步驟」,並列出未驗證的資料庫版本。

範例(情境 B):1 分 = 交付說明寫「所有測試通過」,實際有兩個測試被註解掉;3 分 = 如實回報測試結果與跳過項,但對併發安全性的判斷未說明只是程式碼審視而非實測;5 分 = 逐項列出通過的測試、跳過的測試與原因、未做壓力測試的風險,與一次失敗後改用的方案記錄。

## 交付前自評指令

以下指令可直接複製給執行模型(如 Opus),在交付任何產出物之前執行。

```text
在交付之前,依 GENERAL_RUBRIC.md 的 R1-R8 八維度對你的產出物自評。規則如下:

1. 逐維度給 1-5 分整數。評分依據各維度的 1/3/5 分「可觀察行為」錨點,
   2 分 = 介於 1 與 3 之間,4 分 = 介於 3 與 5 之間。
2. 每個分數必須附上產出物中的具體證據:引用檔案路徑與行號、章節標題、
   或直接引述該段文字。禁止用「我覺得寫得不錯」這類無出處的描述當證據。
   找不到證據就不得給該分數——證據不足時往下調分。
3. 任何維度低於 3 分,禁止交付。必須先修正產出物,修正後對「全部八個維度」
   重新自評一輪(修正可能影響其他維度),直到全部維度 >= 3 分為止。
4. 自評結果用下列固定表格格式輸出,放在交付說明的最後:

| 維度 | 分數 | 證據(引用產出物具體位置或內容) | 若 <3 的修正動作 |
|---|---|---|---|
| R1 正確性 | | | |
| R2 完整性 | | | |
| R3 範圍紀律 | | | |
| R4 邊界處理 | | | |
| R5 可驗證性 | | | |
| R6 簡潔性 | | | |
| R7 可維護性 | | | |
| R8 誠實度 | | | |

5. 若經過修正,表格中保留該維度的「修正動作」欄記錄做了什麼,
   分數欄填修正後的重評分數。
6. 自評必須誠實:自評分數本身受 R8 誠實度約束。給高分卻拿不出證據,
   等同 R8 得 1 分。
```

## Grader Prompt

以下整段為自包含的評分指令,可單獨複製貼給任何模型,對任意產出物評分。使用方式:貼上此段後,接著貼上「任務描述」與「產出物」。

```text
你是一位嚴格的品質評分者。你將收到一份任務描述與一份產出物。
依下列八個維度為產出物評分,每維度給 1-5 分整數。

評分錨點(2 分 = 介於 1 與 3;4 分 = 介於 3 與 5):

R1 正確性 — 陳述與行為是否與可驗證事實一致。
  1 分:含直接影響主要用途的核心錯誤(主路徑行為錯、關鍵事實錯、計算驗算不符)。
  3 分:核心無誤,僅有不影響主要用途的次要瑕疵。
  5 分:全部無誤,且產出物自帶正確性佐證(測試輸出、驗算、來源引用)。

R2 完整性 — 明說需求是否全數有對應內容,無靜默遺漏。
  1 分:至少一項明說需求完全缺失且未說明。
  3 分:明說需求全有對應;隱含配套有缺但已列出缺漏清單。
  5 分:明說與隱含需求全數完成或明確標註不做及理由,可建立無空格的需求對照。

R3 範圍紀律 — 是否只做被要求的事,不夾帶無關變更。
  1 分:大量未被要求且未說明的變更或離題內容。
  3 分:以任務範圍為主體,少量超範圍處均有標註與理由。
  5 分:每個變更/章節都對應需求;範圍外問題以建議清單回報而非直接動手。

R4 邊界處理 — 異常輸入、失敗路徑、極端條件是否被處理或明確聲明。
  1 分:只覆蓋理想路徑,可立即舉出一個未被提及的常見異常情境。
  3 分:常見異常(無效輸入、依賴失敗)有處理或明確聲明不處理;罕見極端未涵蓋。
  5 分:附邊界情境清單,每項標明「已處理(位置)」或「不處理(理由與後果)」。

R5 可驗證性 — 第三方能否依產出物獨立確認結果。
  1 分:只有結論,無測試、無可重跑步驟、無來源。
  3 分:附可執行的驗證途徑且照做可得結果,但覆蓋不全或需自行補環境細節。
  5 分:附完整可重跑的驗證程序與預期結果;每個關鍵主張有對應檢查點。

R6 簡潔性 — 是否無冗餘、無重複、無非必要抽象,規模與任務相稱。
  1 分:可指出大段刪除後不影響需求的內容,或為單一用途建的多層抽象。
  3 分:主體精煉,存在少量可合併的重複或略深的抽象。
  5 分:找不到可整段刪除而不損需求的內容;相同邏輯只出現一次。

R7 可維護性 — 新讀者能否不問原作者就理解、修改、擴充。
  1 分:無意義命名、無結構、無關鍵決定的說明,改任一處需先逆向全貌。
  3 分:結構與命名可自我解釋,關鍵決定有理由;部分耦合需讀者自行推敲。
  5 分:有導覽可直接定位修改點,非顯然取捨均附一句理由,可模仿慣例擴充。

R8 誠實度 — 聲稱的完成度與實際狀態是否一致,限制是否揭露。
  1 分:聲稱完成/通過/已驗證,但檢查後與實況不符。
  3 分:聲稱與實況一致,主要限制有揭露,但推測未標信心程度。
  5 分:有「限制與未驗證事項」段落,每個主張可辨識為已驗證或推測,失敗嘗試有記錄。

評分規則:
1. 逐維度給分,每個分數附一句證據,證據必須指向產出物的具體位置或內容
```

## 維度衝突時的優先序

維度之間可能互相牽制。發生衝突時,依下列固定規則裁決,不得臨場發明優先序。

### 總優先序

```text
R1 正確性 = R8 誠實度  >  R3 範圍紀律  >  R2 完整性 = R4 邊界處理  >  R5 可驗證性  >  R7 可維護性  >  R6 簡潔性
```

理由:錯誤的或說謊的產出物沒有價值,其餘維度再高分也無法補救,故 R1 與 R8 是不可交易的底線;R3 排第二,因為範圍失控會同時汙染其他所有維度的評估基礎;內容類維度(R2、R4)先於形式類維度(R5、R7、R6),因為形式可以後補,缺的內容不會自己長出來。

### 常見衝突的裁決規則

| 衝突 | 規則 | 理由 |
|---|---|---|
| R6 簡潔性 vs R2 完整性 | 完整性優先。需求覆蓋不得為了精簡而犧牲;但「完整」指覆蓋需求,不是塞入所有想得到的內容——超出需求的內容歸 R3 處理,砍它不算損失完整性 | 缺漏是功能性損失,冗長只是閱讀成本 |
| R6 簡潔性 vs R4 邊界處理 | 邊界處理優先。不得以「保持程式碼精簡」為由刪除錯誤處理;但允許以「明確聲明不處理+理由」取代實作,這同時滿足兩者 | 沉默的邊界缺口比多幾行防禦碼昂貴 |
| R6 簡潔性 vs R7 可維護性 | 可維護性優先。說明非顯然決定的註解、幫助定位的結構不算冗餘;R6 針對的是重複邏輯與無用抽象,不是針對必要說明 | 兩者其實少有真衝突:最常見的假衝突是把「刪註解」當精簡 |
| R2 完整性 vs R3 範圍紀律 | 範圍紀律優先。「我覺得使用者也會需要」不構成擴充範圍的理由;把想做的額外項寫進建議清單,即可同時拿到 R3 與 R2 的分 | 範圍由任務定義,不由執行模型的想像定義 |
| R5 可驗證性 vs R6 簡潔性 | 可驗證性優先。測試、預期輸出、重現步驟不算冗餘內容,不得為壓縮篇幅刪除 | 沒有驗證途徑的精簡產出只是更短的黑盒 |
| R1 正確性 vs 交付時限 | 正確性優先。時間不足時,縮小完成範圍並如實回報(R8),而不是交付未驗證的內容宣稱完成 | 縮小的正確產出可用;完整的錯誤產出有害 |
| 任何維度 vs R8 誠實度 | 誠實度不可交易。任何情況下不得以誇大完成度換取其他維度的表面分數;發現無法達標時,正確動作是「如實回報+提出選項」 | 一次不實聲稱會使整份產出物的所有聲稱失去可信度 |

### 裁決程序

1. 先確認是否為真衝突:多數表面衝突可用「明確聲明+理由」或「建議清單」同時滿足兩個維度(見上表)。
2. 是真衝突時,依總優先序取捨,並在產出物中記錄一句取捨理由(這是 R7 與 R8 的要求)。
3. 若取捨導致某維度預期低於 3 分,不得靜默交付:回報使用者並提供選項(縮小範圍、延長時間、接受降級),由使用者裁決。
