# ANTIPATTERNS.md — 通用反模式庫

## 本文件目的

本文件收錄六種任務原型中「較弱模型會犯、高階模型不會犯」的具體錯誤行為,每條都寫成可對照、可修正的形式。
它是執行模型(如 Opus)的護欄:動手前用來預防,交付前用來自查。
所有條目描述的都是可觀察的行為,不是態度勸告;能對照 diff、報告或輸出直接判定「犯了 / 沒犯」。

## 如何使用本文件

執行模型(如 Opus)使用方式(強制):

1. **動手前**:依 PLANNING.md 判定任務原型後,**完整讀對應原型的整節**(不是掃過標題),把該節的阻斷級條目記入工作筆記。
2. **執行中**:發現自己正要做出某條「絕不做 Y」描述的行為時,立即停下,改走該條的「改做 Z」。
3. **交付前**:回到對應原型的節,**逐條對照**自己的產出;節末的「快速自查」是最低標準,30 秒內跑完。
4. **處置規則**:
   - 觸犯**阻斷**級 → 不得交付,必須修正。
   - 觸犯**降級**級 → 修正;確實無法修正時,在自審(【自審】)中明確揭露並說明原因。
   - 觸犯**風格**級 → 盡量修正;成本過高時可保留,但不得累積三條以上。
5. 跨原型任務(例如「除錯後補文件」)讀所有涉及原型的節。

## 嚴重度三級定義

| 嚴重度 | 定義 | 交付前處置 |
|--------|------|-----------|
| 阻斷 | 會直接導致產出錯誤或不可用、造成損害(錯誤結論、壞掉的程式碼、虛構的事實) | 必須修正,否則不得交付 |
| 降級 | 產出可用,但品質明顯下降(可維護性差、可信度受損、遺漏重要面向) | 修正,或在自審中揭露 |
| 風格 | 不影響正確性,但增加溝通或維護成本 | 盡量修正 |

## 條目格式說明

每條反模式使用固定五欄位格式,全文件一致:

```
### AP-{原型前綴}-{兩位數} 標題
- 嚴重度:阻斷 / 降級 / 風格
- 情況 X:什麼情境下會出現這個反模式
- 絕不做 Y:弱模型的典型錯誤行為(具體、可觀察)
- 改做 Z:正確做法(可執行的替代行為)
- 因為 W:背後原因(一到兩句)
```

原型前綴對照:SW 軟體實作、SD 系統設計、RA 研究分析、TW 技術寫作、DB 除錯排錯、DA 資料分析。
條目觸犯與否會反映在 GENERAL_RUBRIC.md 的對應維度分數上(尤其 R1 正確性、R3 範圍紀律、R8 誠實度)。

## 軟體實作(SW)

### AP-SW-01 過早動手
- **嚴重度**:阻斷
- **情況 X**:接到的任務描述只有一兩句話,或任務「看起來很像做過的東西」,產生立刻動手的衝動。
- **絕不做 Y**:讀完任務第一句就開始寫程式碼;跳過需求澄清與範圍界定,邊寫邊猜規格;寫到一半發現理解錯誤,卻硬改已寫的程式碼去遷就錯誤方向。
- **改做 Z**:先完成 PLANNING.md 的前三階段(需求澄清 → 問題重述 → 範圍界定),把「要做什麼、不做什麼、怎樣算完成」各寫成一行,才允許開第一個檔案。
- **因為 W**:已寫程式碼的沉沒成本會讓人捨不得丟棄錯誤方向;方向錯時,越早寫越難回頭。

### AP-SW-02 幻覺 API
- **嚴重度**:阻斷
- **情況 X**:使用不熟悉的函式庫、框架或專案內部模組,憑印象寫出呼叫。
- **絕不做 Y**:憑記憶寫出不存在的函式或參數(例如給 pandas 函式塞一個聽起來合理但不存在的參數);假設內部模組有 `utils.retry()` 這種「照理應該有」的工具;import 沒安裝的套件。
- **改做 Z**:呼叫任何非標準函式前,先用 Grep/Read 確認它在程式碼庫中存在,或查官方文件確認簽名;查不到就明說「未確認」,改用已驗證的寫法。
- **因為 W**:弱模型對 API 的記憶是機率性拼貼,長得最合理的呼叫最容易騙過自己與審閱者。

### AP-SW-03 只走快樂路徑
- **嚴重度**:阻斷
- **情況 X**:實作輸入處理、外部呼叫、檔案或網路操作。
- **絕不做 Y**:只處理「輸入合法、網路正常、檔案存在」的情境;對空清單、None、超長字串、逾時、權限不足一律當作不會發生;測試也只測正常輸入。
- **改做 Z**:為每個對外函式列出至少三個異常輸入(空、極端值、非法型別)與一個外部失敗情境,明確決定各自的行為(拋錯 / 預設值 / 重試),並寫進測試。
- **因為 W**:快樂路徑只覆蓋執行路徑的一小部分;生產環境的故障幾乎都發生在沒寫的那部分。

### AP-SW-04 破壞既有介面
- **嚴重度**:阻斷
- **情況 X**:修改公開函式簽名、回傳格式、設定檔欄位或 API 回應結構。
- **絕不做 Y**:為了新需求直接改既有函式的參數順序或回傳型別,不搜尋呼叫端;改 JSON 回應的欄位名稱,理由是「應該沒人在用」。
- **改做 Z**:改介面前先 Grep 所有呼叫端與序列化使用處;無法全部同步修正時採「加不減」策略(新增欄位或參數、保留舊行為),並在交付說明中列出受影響面。
- **因為 W**:呼叫端散布在你看不到的地方;介面破壞是編譯期抓不到、上線才爆炸的典型錯誤。

### AP-SW-05 忽略錯誤處理
- **嚴重度**:阻斷
- **情況 X**:呼叫可能失敗的操作(I/O、解析、外部服務),或收到帶錯誤語意的回傳值。
- **絕不做 Y**:對回傳錯誤碼的函式不檢查回傳值;讓例外一路往上冒卻沒有任何一層決定要處理;錯誤訊息只寫 "error occurred",不含任何上下文。
- **改做 Z**:對每個可失敗點明確三選一:就地處理、包裝後上拋、記錄後降級;錯誤訊息必含操作對象與失敗原因(檔名、URL、狀態碼)。
- **因為 W**:未處理的錯誤不會消失,只會在更遠、更難除錯的地方以更怪的形式重新出現。

### AP-SW-06 範圍蔓延
- **嚴重度**:降級
- **情況 X**:實作過程中看到「順便可以改」的東西:醜的命名、可優化的迴圈、想加的小功能。
- **絕不做 Y**:在修 bug 的變更裡順手重構三個無關檔案;任務要求 A,交付 A+B+C,而 B 和 C 沒人要求、沒有驗收標準、沒有測試。
- **改做 Z**:發現範圍外的改進機會時,記入交付說明的「後續建議」清單,不動手;檢查 diff 中每個變更都能對應到任務敘述的某一句話。
- **因為 W**:範圍外變更沒有驗收標準、沒人審查其必要性,是回歸錯誤的主要來源。

### AP-SW-07 沒跑就說完成
- **嚴重度**:阻斷
- **情況 X**:寫完程式碼或測試,準備回報結果。
- **絕不做 Y**:回覆「測試已通過」但實際沒有執行過測試;用「應該可以動」代替實際執行;貼出想像中的執行輸出當成證據。
- **改做 Z**:宣稱任何驗證結果前,實際執行指令並引用真實輸出;跑不了(缺環境、缺憑證)就明確寫「未執行,原因是 X」。
- **因為 W**:未經執行的「完成」只是機率性猜測;把猜測寫成驗證結果,等於同時犯錯與說謊。

### AP-SW-08 複製貼上式重複
- **嚴重度**:降級
- **情況 X**:需要寫「和現有某段很像」的邏輯,或同一份修改要套用到多處。
- **絕不做 Y**:複製一段 30 行的函式、改兩個變數名就存檔;同樣的驗證邏輯貼到四個地方,之後修 bug 只修到其中兩處。
- **改做 Z**:相同邏輯第二次出現時抽成函式或共用常數;若刻意保留重複(例如避免不當耦合),在註解寫明理由。
- **因為 W**:重複的程式碼會各自分岔演化,修正只落在部分副本上,造成行為不一致。

### AP-SW-09 吞掉例外
- **嚴重度**:降級
- **情況 X**:程式拋出例外導致測試失敗或流程中斷,想「先讓它跑起來」。
- **絕不做 Y**:用空的 `try/except`(或 `catch (e) {}`)包住報錯區塊;把例外轉成回傳 None,而呼叫端沒有任何檢查。
- **改做 Z**:至少記錄例外內容與脈絡,再決定是否降級處理;只捕捉能明確列舉的錯誤型別,其他一律上拋。
- **因為 W**:被吞掉的例外會把故障點位移到資料已損壞的下游,除錯成本放大數倍。

### AP-SW-10 無視既有慣例
- **嚴重度**:風格
- **情況 X**:進入既有程式碼庫工作;專案已有命名、目錄、測試、錯誤處理的固定模式。
- **絕不做 Y**:專案全用 snake_case 卻寫出 camelCase;別人都用專案的 logger 你用 print;自創目錄結構放新檔案;自帶一套與專案無關的測試風格。
- **改做 Z**:動手前先讀同目錄 2-3 個相鄰檔案,模仿其命名、import 風格與測試組織方式;不確定時以最近修改過的檔案為準。
- **因為 W**:違反慣例的程式碼即使正確,也提高審查與維護成本,並暗示作者沒讀過周圍的程式碼。

### AP-SW-11 硬編碼環境假設
- **嚴重度**:降級
- **情況 X**:需要路徑、URL、憑證、埠號等與環境相關的值。
- **絕不做 Y**:把 `/home/user/...`、`localhost:8080`、API key 直接寫死在程式碼裡;假設執行目錄一定是專案根目錄;假設目標機器裝了你機器上有的工具。
- **改做 Z**:環境相關值一律走設定檔、環境變數或參數傳入;預設值必須能在乾淨環境下運作,或失敗時給出明確錯誤訊息。
- **因為 W**:硬編碼值讓程式只在作者的環境正確;憑證寫進程式碼則直接構成安全事故。

### AP-SW-12 順手改壞格式
- **嚴重度**:風格
- **情況 X**:編輯器或工具自動重排檔案,或想「順便統一格式」。
- **絕不做 Y**:交付的 diff 中一半是空白、引號、換行的無關變動,淹沒真正的邏輯修改;任務只改三行,卻對整個檔案跑格式化工具。
- **改做 Z**:diff 只含任務必要的變更;格式化需求另開獨立變更,並標明「純格式,無邏輯變更」。
- **因為 W**:混入格式雜訊的 diff 無法有效審查,邏輯錯誤會藏在幾百行無關變動裡。

**快速自查(SW)**
- diff 裡每一行變更都能對應到任務敘述嗎?
- 每個可失敗點(I/O、解析、外部呼叫)都有明確的錯誤行為嗎?
- 所有非標準 API 呼叫都在程式碼庫或文件中確認過存在嗎?
- 你宣稱的每個「已通過」都有真實執行輸出佐證嗎?
- 有沒有改到別人正在用的介面,而你沒搜尋過呼叫端?

## 系統設計(SD)

### AP-SD-01 過度工程
- **嚴重度**:降級
- **情況 X**:設計新系統或新模組;需求目前規模小,但「未來可能長大」。
- **絕不做 Y**:為日活一千的內部工具設計微服務加訊息佇列加三層快取;引入設計模式只因為「比較專業」;抽象層包抽象層,而實際上只有一個實作。
- **改做 Z**:以「目前需求的 2-10 倍」為設計上限;每引入一個元件就寫一句「刪掉它會發生什麼具體損失」,答不出來就刪。
- **因為 W**:弱模型傾向堆砌教科書元件來展示能力;每個多餘元件都是永久的維運與認知成本。

### AP-SD-02 單方案直衝
- **嚴重度**:降級
- **情況 X**:問題存在多種合理解法(同步 vs 非同步、SQL vs NoSQL、自建 vs 採購)。
- **絕不做 Y**:想到第一個可行方案就直接展開細節;設計文件裡只有一個方案,沒有任何比較或替代方案的痕跡。
- **改做 Z**:列出 2-3 個真實可行的方案,各寫優缺點與適用條件,再說明選擇理由;被否決的方案保留一行「為何不選」。
- **因為 W**:沒有比較的選擇無法被審查;第一個想到的方案往往只是訓練資料中最常見的,不是最適合這個問題的。

### AP-SD-03 非功能需求無數字
- **嚴重度**:阻斷
- **情況 X**:設計文件談到效能、規模、可用性。
- **絕不做 Y**:通篇寫「高效能」「可擴展」「高可用」卻沒有任何數字;不詢問也不假設 QPS、資料量、延遲上限,就直接選定架構。
- **改做 Z**:每項非功能需求給出數字與來源(實測、需求方提供,或明確標示「假設值」);用數字反推元件選型(例如 100 QPS 用單機即可,不需要分散式)。
- **因為 W**:沒有數字的設計無法驗證對錯;架構選型的正確性完全取決於量級。

### AP-SD-04 幻覺元件能力
- **嚴重度**:阻斷
- **情況 X**:設計中引用資料庫、訊息佇列、雲端服務等外部元件的特性。
- **絕不做 Y**:假設 Redis 提供跨鍵交易、假設某訊息佇列保證「恰好一次」投遞、假設某雲端服務有實際不存在的功能等級,並讓整個設計依賴這個假設。
- **改做 Z**:設計所依賴的每個元件關鍵特性(一致性、順序保證、容量上限)都查文件確認並附出處;查不到就標「未確認,需驗證」並準備退路方案。
- **因為 W**:對元件能力的錯誤假設會讓整個架構建立在不存在的地基上;到實作階段才發現,代價就是重做。

### AP-SD-05 無視既有系統
- **嚴重度**:阻斷
- **情況 X**:在既有系統上加功能或改架構。
- **絕不做 Y**:設計假設可以砍掉重練;忽略現有資料格式、既有客戶端、進行中的流量,畫出一張「全新世界」的架構圖。
- **改做 Z**:設計必含「現況」一節(既有元件、資料流、限制)與「遷移路徑」一節(如何從現況走到目標、新舊並存期間怎麼運作)。
- **因為 W**:絕大多數設計工作是在活著的系統上動手術;沒有遷移路徑的設計無法實施。

### AP-SD-06 無失敗模式分析
- **嚴重度**:阻斷
- **情況 X**:設計含多個元件協作的系統。
- **絕不做 Y**:架構圖只畫成功流程;不回答「這個元件掛了會怎樣」;單點故障存在卻沒有被指出。
- **改做 Z**:對每個元件與每條依賴寫一行失敗行為:它掛掉 / 變慢 / 回傳錯誤資料時,系統整體表現為何、是否可接受;不可接受的加上緩解手段。
- **因為 W**:分散式系統的設計品質主要體現在失敗路徑;成功路徑人人會畫。

### AP-SD-07 方塊圖沒有契約
- **嚴重度**:降級
- **情況 X**:畫元件圖、做模組切分。
- **絕不做 Y**:圖上兩個方塊之間只有一條線寫著「呼叫」;不定義介面內容、資料格式、同步或非同步、誰擁有哪份資料。
- **改做 Z**:每條元件間連線標明:協議、關鍵資料欄位、同步/非同步、錯誤如何傳遞;每個資料實體標明唯一擁有者。
- **因為 W**:方塊圖的價值全在邊界定義;沒有契約的圖只是裝飾,實作時每條線都會變成爭議。

### AP-SD-08 名詞代替決策
- **嚴重度**:降級
- **情況 X**:撰寫選型與技術決策段落。
- **絕不做 Y**:把「採用事件驅動架構以實現解耦」這種句子當成決策依據;用 CQRS、Service Mesh 等名詞代替「為什麼這裡需要它」的論證。
- **改做 Z**:每個技術選擇都寫出它解決的、本專案實際存在的問題,以及不採用它的具體代價;寫不出具體問題就不引入。
- **因為 W**:名詞堆砌是弱模型模仿設計文件表面形式的典型輸出,掩蓋了沒有實質推理的事實。

### AP-SD-09 忽略資料一致性與遷移
- **嚴重度**:阻斷
- **情況 X**:設計涉及資料複製、快取、跨服務資料或 schema 變更。
- **絕不做 Y**:同一份資料存在三個地方卻不指定誰是真相來源;加快取不定義失效策略;改 schema 不寫既有資料如何遷移。
- **改做 Z**:每份被複製的資料指定唯一真相來源與同步/失效機制;schema 變更附遷移步驟與回滾方式。
- **因為 W**:資料不一致的錯誤最難除錯、最難修復;缺席的遷移計畫會在上線日變成事故。

### AP-SD-10 取捨不寫代價
- **嚴重度**:降級
- **情況 X**:說明選型理由、撰寫 ADR。
- **絕不做 Y**:選定方案的段落只列優點;缺點欄寫「無明顯缺點」;不寫這個選擇在什麼條件下會變成錯誤。
- **改做 Z**:每個決策明確寫出付出的代價(複雜度、延遲、成本、廠商鎖定)與失效條件(「若流量超過 X 或需求變成 Y,此決策應重新評估」)。
- **因為 W**:工程沒有免費的選擇;寫不出代價,代表沒有真正理解這個決策。

### AP-SD-11 圖文不一致
- **嚴重度**:風格
- **情況 X**:設計文件同時包含圖與文字,或包含多張圖。
- **絕不做 Y**:同一元件在圖上叫 Gateway、內文叫 Proxy、表格叫 Edge Service;圖已改版,但引用該圖的文字還在描述舊版流程。
- **改做 Z**:全文件為每個元件定一個唯一名稱並貫穿使用;每次改圖後,逐段檢查引用該圖的文字是否同步。
- **因為 W**:名稱漂移迫使讀者自行猜測對應關係;猜錯,就會依錯誤理解去實作。

**快速自查(SD)**
- 每項非功能需求都有數字(或明標為假設)嗎?
- 每個元件掛掉時系統的行為,你都答得出來嗎?
- 至少有第二個方案被認真比較過嗎?
- 依賴的每個外部元件關鍵特性都查過文件嗎?
- 從現況到目標架構的遷移路徑寫了嗎?

## 研究分析(RA)

### AP-RA-01 把猜測寫成事實
- **嚴重度**:阻斷
- **情況 X**:資料不足但問題需要答案,或對某領域「有印象」。
- **絕不做 Y**:把「我記得大概是」直接寫成肯定句;數字、年份、版本號憑印象填入;推論結果與查證事實混寫在同一段且無任何標記。
- **改做 Z**:每個關鍵陳述標明類型:查證事實(附出處)、推論(附推理鏈)、假設(附標記);查不到的直接寫「未能查證」。
- **因為 W**:讀者無法區分你的事實與猜測;一個錯誤猜測會污染整份報告的可信度。

### AP-RA-02 掩飾不確定性
- **嚴重度**:阻斷
- **情況 X**:證據薄弱、來源互相矛盾,或問題本身沒有定論。
- **絕不做 Y**:把六成把握的結論用十成的語氣寫出;來源矛盾時只挑一邊呈現、不提矛盾存在;用流暢的行文掩蓋證據缺口。
- **改做 Z**:對每個結論標註信心等級(高/中/低)與依據;矛盾證據並列呈現並說明取捨;明確區分「證據顯示」與「我判斷」。
- **因為 W**:下游決策依賴你的信心標度;掩飾不確定性等於替讀者做了他沒同意的風險決策。

### AP-RA-03 幻覺引用
- **嚴重度**:阻斷
- **情況 X**:需要佐證來源、文獻或數據出處。
- **絕不做 Y**:編造論文標題、作者、年份;寫「某研究顯示」卻說不出是哪個研究;貼出格式完美但不存在的 URL 或 DOI。
- **改做 Z**:只引用實際打開驗證過的來源;引用時附可追溯資訊(URL、標題、日期);沒有來源的內容誠實標為「常識性判斷,未查證」。
- **因為 W**:幻覺引用長得比真引用還工整,是弱模型最具破壞性的行為;一條假引用被抓到,整份報告作廢。

### AP-RA-04 單一來源定論
- **嚴重度**:降級
- **情況 X**:找到第一個看似回答問題的來源。
- **絕不做 Y**:憑單一部落格文章或單一論文摘要就下結論;不檢查來源的利益關係,把廠商白皮書當中立評測引用。
- **改做 Z**:關鍵結論至少用兩個獨立來源交叉驗證;標明每個來源的性質(官方文件/論文/媒體/廠商行銷)與潛在偏向。
- **因為 W**:單一來源可能過時、片面或帶有立場;交叉驗證是「研究」與「轉述」的分界線。

### AP-RA-05 確認偏誤搜尋
- **嚴重度**:降級
- **情況 X**:心中已有預期答案,才開始找資料。
- **絕不做 Y**:只用支持預期結論的關鍵字搜尋;跳過與預期不符的搜尋結果;報告只收錄支持面的證據。
- **改做 Z**:對每個主要結論至少執行一次反向搜尋(「X 的缺點」「X failure」「X 批評」);找到的反面證據必須出現在報告中。
- **因為 W**:搜尋策略決定證據樣本;只搜一邊,等於在資料收集階段就預定了結論。

### AP-RA-06 時效未標註
- **嚴重度**:降級
- **情況 X**:主題涉及快速變化的領域(價格、版本、市占、法規)。
- **絕不做 Y**:引用三年前的效能比較當成現況;不標註資料的時間點;把「截至訓練資料為止」的知識當成當下事實。
- **改做 Z**:每個時效敏感的數據標註資料時間;超過合理時效的標明「可能已過時」;領域變化快時明確建議讀者再次確認。
- **因為 W**:過時資訊比缺失資訊更危險——它以正確的形式承載錯誤的內容。

### AP-RA-07 摘要失真
- **嚴重度**:阻斷
- **情況 X**:轉述來源內容、壓縮長文。
- **絕不做 Y**:把來源的「在特定條件下可能」轉述成「會」;把樣本數 20 的初步研究轉述成定論;摘要時丟掉原文的限定條件與反例。
- **改做 Z**:轉述時保留原文的強度詞與限定條件;拿不準時直接引用原句;摘要完成後回頭比對原文,自問「原作者會同意這個轉述嗎」。
- **因為 W**:每次轉述都是一次失真機會,而弱模型的失真方向系統性偏向「更肯定、更簡單」。

### AP-RA-08 迴避「不知道」
- **嚴重度**:阻斷
- **情況 X**:問題超出可得資料,或問題本身預設了錯誤前提。
- **絕不做 Y**:資料查不到仍硬生成一個看似完整的答案;問題前提有誤仍照著錯誤前提回答;用大量正確的背景資訊掩蓋「核心問題沒答」的事實。
- **改做 Z**:直接回答「現有資料無法回答 X,原因是 Y」;前提有誤先指出;能部分回答時,明確劃出已答與未答的邊界。
- **因為 W**:弱模型被訓練成傾向產出完整回答;但「誠實的不知道」比「流暢的錯誤」對讀者有價值得多。

### AP-RA-09 羅列不綜合
- **嚴重度**:降級
- **情況 X**:收集了大量資料,要產出報告。
- **絕不做 Y**:把搜尋結果逐條貼出當成報告;每個來源一段摘要,卻沒有回答原始問題的結論段;讓讀者自己從十段材料裡拼答案。
- **改做 Z**:報告以「問題的答案」開頭,證據作為支撐組織在後;每段材料必須明確連回它所支撐或反駁的論點。
- **因為 W**:羅列是搜尋引擎的工作;分析的價值在綜合與判斷,缺了這一步任務就沒有完成。

### AP-RA-10 結論強度超過證據
- **嚴重度**:降級
- **情況 X**:撰寫結論與建議段。
- **絕不做 Y**:三個弱證據加起來寫成「顯然」「毫無疑問」;從單一案例推廣成全稱陳述(「A 公司如此,故整個產業皆然」)。
- **改做 Z**:結論的強度詞與證據等級對齊:多個獨立強證據才用「顯示」,弱證據用「初步跡象顯示」;全稱陳述改為量化範圍。
- **因為 W**:讀者依結論強度決定行動力度;超載的結論會引導超載的決策。

### AP-RA-11 術語堆砌
- **嚴重度**:風格
- **情況 X**:主題含專業術語,或想讓報告「看起來專業」。
- **絕不做 Y**:整段使用讀者不熟的縮寫而不解釋;用術語復述問題假裝回答(「這是典型的 XY 問題,體現了 Z 效應」然後就結束)。
- **改做 Z**:每個關鍵術語首次出現時用一句話定義;寫完後檢查:把術語刪掉,句子還剩下內容嗎?沒有就重寫。
- **因為 W**:術語應該壓縮資訊,不是替代資訊;堆砌術語常是內容空洞的遮羞布。

**快速自查(RA)**
- 每個關鍵數字與陳述都能指出來源嗎?
- 報告裡出現過任何反面證據嗎?一條都沒有就回去搜。
- 結論的信心用詞和證據強度對齊了嗎?
- 有沒有哪句話其實是猜測,卻寫成了事實?
- 讀者能在前三段找到問題的直接答案嗎?

## 技術寫作(TW)

### AP-TW-01 寫給自己而非讀者
- **嚴重度**:降級
- **情況 X**:撰寫任何文件之前;尤其是作者剛完成實作、滿腦子細節的時候。
- **絕不做 Y**:假設讀者知道你剛經歷的脈絡;按「我做事的順序」而非「讀者需要的順序」組織內容;跳過對你顯然、對讀者未必的前置知識。
- **改做 Z**:動筆前寫一行「讀者是誰、讀完要能做到什麼」;完稿後以該讀者身分重讀,在每個「他讀到這裡會卡住」的點補上橋接說明。
- **因為 W**:寫作者被知識詛咒綁架是預設狀態;不主動建模讀者,文件就只是作者的記憶轉存。

### AP-TW-02 範例沒跑過
- **嚴重度**:阻斷
- **情況 X**:文件包含程式碼範例、指令或設定片段。
- **絕不做 Y**:憑印象寫範例程式碼直接貼進文件;改了正文邏輯卻沒改對應範例;範例 import 了正文根本沒提的模組。
- **改做 Z**:每個範例實際執行一次並保留輸出;無法執行的(偽代碼、片段)明確標註「示意,不可直接執行」。
- **因為 W**:讀者做的第一件事就是複製範例來跑;跑不動的範例會讓讀者否定整份文件的可信度。

### AP-TW-03 文實不符
- **嚴重度**:阻斷
- **情況 X**:為既有系統寫文件,或改了程式碼要更新文件。
- **絕不做 Y**:憑對程式碼的印象描述行為,不打開程式碼確認;描述「設計上應該的行為」而非「實際的行為」;參數預設值、回傳格式照舊版文件抄。
- **改做 Z**:每個行為描述都對照當前程式碼或實測結果;寫「預設值為 X」之前實際查到定義處;文件與程式碼不一致時以程式碼為準並註明。
- **因為 W**:錯誤的文件比沒有文件更糟——讀者會信任它,並依它行動。

### AP-TW-04 幻覺功能
- **嚴重度**:阻斷
- **情況 X**:撰寫 API 文件、參數說明、功能列表。
- **絕不做 Y**:記載實際不存在的參數、旗標或設定選項;把「類似工具有的功能」寫進這個工具的文件;自動補全「照慣例應該存在」的選項。
- **改做 Z**:每個記載的參數與選項都在原始碼或 `--help` 輸出中確認存在;確認不了的就不寫。
- **因為 W**:弱模型會用訓練資料中相似工具的介面自動補全;文件中一個虛構參數,就足以讓使用者浪費數小時。

### AP-TW-05 步驟跳躍
- **嚴重度**:阻斷
- **情況 X**:撰寫安裝指南、教學、操作手冊。
- **絕不做 Y**:漏掉「顯然」的步驟(先啟動服務、先切換目錄、先設環境變數);步驟三的指令依賴步驟一沒提的前置條件;假設讀者的環境和你的一樣。
- **改做 Z**:在乾淨環境(或想像的全新環境)從第一步照文件執行到最後;每個指令標明執行位置與前置狀態;文件開頭列出前置需求清單。
- **因為 W**:教學文件的唯一品質標準是「照做能成功」;跳一步,就斷一次。

### AP-TW-06 埋沒重點
- **嚴重度**:降級
- **情況 X**:內容包含警告、限制、破壞性操作或常見錯誤。
- **絕不做 Y**:把「此操作會刪除所有資料」藏在第七段中間;關鍵限制寫在附錄;讀者必讀的資訊與可選的背景知識混排在一起。
- **改做 Z**:警告與破壞性後果放在對應操作「之前」,以醒目格式(警告區塊)標出;文件開頭列出「開始前必須知道的事」。
- **因為 W**:讀者是掃讀不是精讀;位置就是優先度,埋沒等於刪除。

### AP-TW-07 同義詞漂移
- **嚴重度**:風格
- **情況 X**:長文件,或經過多次編輯的文件。
- **絕不做 Y**:同一概念交替使用「設定檔 / 配置檔 / config」;同一操作一下叫「部署」一下叫「發布」;為了行文變化刻意換詞。
- **改做 Z**:建立文件級術語表,每個概念一個詞用到底;在技術寫作中重複同一個詞是正確行為,不是文筆差。
- **因為 W**:讀者會假設不同的詞指不同的東西;每次換詞都製造一次「這是同一個東西嗎」的認知稅。

### AP-TW-08 抽象不落地
- **嚴重度**:降級
- **情況 X**:解釋概念、架構或設計理念。
- **絕不做 Y**:連續多段抽象描述沒有任何範例(「本系統提供彈性的可組合抽象」);用比喻代替精確定義,而比喻經不起推敲。
- **改做 Z**:每個抽象概念後緊跟一個具體範例(輸入、輸出、實際場景);把「一個概念、一個例子」當成行文節奏原則。
- **因為 W**:抽象描述在不同讀者腦中的解讀是發散的;範例把發散收斂為一致理解。

### AP-TW-09 改正文不改周邊
- **嚴重度**:降級
- **情況 X**:更新既有文件的一部分。
- **絕不做 Y**:改了第三節內容,但目錄、摘要、交叉引用還是舊的;改了參數名稱,但後文範例沒同步;新增一節,但「本文結構」的介紹沒更新。
- **改做 Z**:每次修改後,全域搜尋被改動的名詞與章節編號;固定檢查四個位置:目錄、摘要、範例、交叉引用。
- **因為 W**:局部更新製造內部矛盾;讀者無從判斷哪一半才是對的。

### AP-TW-10 填充式行文
- **嚴重度**:風格
- **情況 X**:任何文件;尤其想讓篇幅「看起來夠份量」的時候。
- **絕不做 Y**:「在當今快速發展的技術環境中」式開場;每節開頭復述上一節結尾;結論段重抄一遍摘要;用三句話說一句話的內容。
- **改做 Z**:每句話自問「刪掉它,讀者損失什麼」,答不出來就刪;直接以資訊開頭;結論只寫新增的判斷與行動項。
- **因為 W**:填充稀釋資訊密度,而讀者的注意力是文件最稀缺的資源。

### AP-TW-11 版本與環境未標註
- **嚴重度**:降級
- **情況 X**:內容因版本、平台、環境而異(指令旗標、API、UI 路徑)。
- **絕不做 Y**:寫「執行以下指令即可」,但該指令只在特定版本之後才存在;混寫多個版本的行為而不區分;截圖與現行 UI 不符也照用。
- **改做 Z**:文件開頭標明適用版本與平台;版本間有差異的段落分別標註;引用其他版本的行為時明確標示。
- **因為 W**:版本錯配的文件對讀者而言就是錯的文件,而讀者無法自行分辨。

**快速自查(TW)**
- 所有範例都實際跑過了嗎?
- 每個描述的參數與功能都在程式碼或 `--help` 裡確認過存在嗎?
- 破壞性操作的警告放在操作步驟之前了嗎?
- 一個不具備你腦中脈絡的讀者,照文件從頭走到尾會卡在哪一步?
- 同一個概念全篇都用同一個詞嗎?

## 除錯排錯(DB)

### AP-DB-01 未重現先開刀
- **嚴重度**:阻斷
- **情況 X**:收到 bug 報告或錯誤描述,看起來「大概知道原因」。
- **絕不做 Y**:沒有重現錯誤就開始改程式碼;依 bug 描述的文字猜測位置直接動手;改完後因為「本來就重現不了」而無法確認是否修好。
- **改做 Z**:先建立最小重現步驟(指令、輸入、預期輸出 vs 實際輸出);重現不了就先回報並收集更多資訊,不動程式碼。
- **因為 W**:不能重現就不能驗證;沒有重現步驟的修復只是換一個地方賭博。

### AP-DB-02 只改症狀不找根因
- **嚴重度**:阻斷
- **情況 X**:找到了能讓錯誤消失的最近手段(加 null 檢查、包 try/except、加 sleep)。
- **絕不做 Y**:在報錯的那一行加防禦程式碼讓錯誤不再出現,卻不回答「這個值為什麼會是 null」;用 sleep 蓋掉競態條件;把復現機率壓低當成修好。
- **改做 Z**:沿著資料流回溯到「值第一次變得不對」的位置,在那裡修;症狀處的防禦可以保留,但必須另外寫明根因與在該處防禦的理由。
- **因為 W**:症狀修補讓同一根因以新形態在別處復發;累積的防禦程式碼還會掩蓋未來的錯誤訊號。

### AP-DB-03 一次改多個變因
- **嚴重度**:阻斷
- **情況 X**:有多個懷疑點,或想「順便把可疑的都修掉」。
- **絕不做 Y**:同時改設定、升級套件、改三處程式碼,然後只跑一次測試;錯誤消失了卻說不出是哪個改動生效;錯誤還在,也不知道哪個改動引入了新問題。
- **改做 Z**:一次只改一個變因 → 跑重現步驟 → 記錄結果 → 再改下一個;無效的改動立刻還原,不留在工作區。
- **因為 W**:多變因同時變動摧毀因果推理的基礎;你會永久失去「哪個改動導致哪個結果」的資訊。

### AP-DB-04 不讀錯誤訊息
- **嚴重度**:降級
- **情況 X**:出現 stack trace、錯誤碼、log 輸出。
- **絕不做 Y**:掃一眼錯誤類型就開始憑印象推理;跳過 stack trace 中間層,直接假設是「這類錯誤的常見原因」;錯誤訊息明明寫了檔名與行號,卻跑去別處找。
- **改做 Z**:逐行讀完錯誤原文,先回答三個問題:錯誤發生在哪一行、錯誤型別是什麼、訊息中的具體值是什麼;之後才開始建立假設。
- **因為 W**:錯誤訊息是成本最低的一手證據;弱模型傾向用「這類錯誤通常是」的先驗,蓋過眼前的具體事實。

### AP-DB-05 猜測式亂槍打鳥
- **嚴重度**:降級
- **情況 X**:第一個假設驗證失敗;時間壓力下想快點修好。
- **絕不做 Y**:隨機改動可疑處「試試看」;連續套用五種網路上查到的解法,每種都沒驗證是否適用於本案;試過什麼、結果如何完全不記錄。
- **改做 Z**:每次嘗試前寫下假設(「我認為原因是 X;若成立,改 Y 之後 Z 應消失」);驗證後記錄結果;假設用盡時回到收集證據,不繼續亂試。
- **因為 W**:無假設的嘗試不會縮小搜尋空間,只是把時間換成隨機數,而且亂試會污染現場。

### AP-DB-06 修好不驗證
- **嚴重度**:阻斷
- **情況 X**:改完程式碼,錯誤「看起來應該修好了」。
- **絕不做 Y**:改完不重跑重現步驟就回報修復;只跑本來就通過的測試,不跑原本失敗的那一個;修復宣稱依賴「理論上這樣就對了」。
- **改做 Z**:重跑最小重現步驟,確認原錯誤消失;再跑周邊測試,確認沒有引入回歸;兩份輸出都貼進交付說明。
- **因為 W**:未驗證的修復有相當機率沒修好或修壞別處;「理論上正確」在除錯領域的預測力很低。

### AP-DB-07 改測試遷就錯誤
- **嚴重度**:阻斷
- **情況 X**:測試持續失敗,而修程式碼看起來很難。
- **絕不做 Y**:刪掉失敗的測試;把斷言的期望值改成目前的錯誤輸出;把測試標記為 skip,然後回報「測試全過」。
- **改做 Z**:先判定測試本身是否正確;測試對,就修程式碼;測試確實錯了才改測試,並在交付說明中單獨列出「變更了測試期望,理由是 X」。
- **因為 W**:測試是規格的執行化;改測試遷就實作等於偷改規格,把錯誤合法化。

### AP-DB-08 flaky 當修好
- **嚴重度**:降級
- **情況 X**:錯誤是間歇性的(競態、時序、外部依賴)。
- **絕不做 Y**:重跑一次通過就宣稱修好;出現機率三成的錯誤,跑三次沒出現就當解決;不區分「修好了」與「這次剛好沒發生」。
- **改做 Z**:間歇性錯誤用多次重跑驗證,次數依原始出現機率決定(例如原本約 1/3 出現,至少連續 20 次通過);交付說明中標注驗證方式與剩餘不確定性。
- **因為 W**:間歇性錯誤的單次通過幾乎沒有證據力;誤判修復會讓問題帶著「已修復」的標籤重新上線。

### AP-DB-09 破壞現場
- **嚴重度**:降級
- **情況 X**:拿到唯一能重現錯誤的環境、log 或 core dump。
- **絕不做 Y**:在唯一能重現的環境上直接試各種修改,改到不再重現也不知道為什麼;清掉 log「讓輸出乾淨一點」;重建環境後才發現忘了保存關鍵狀態。
- **改做 Z**:動手前先保存現場:複製 log、記錄環境狀態、能快照就快照;修改實驗在複製環境上做,原始現場留作對照。
- **因為 W**:現場是不可再生資源;破壞現場之後,連「回到原始錯誤」都做不到。

### AP-DB-10 順手重構
- **嚴重度**:降級
- **情況 X**:除錯過程中看到醜程式碼,想「既然都打開了就整理一下」。
- **絕不做 Y**:在修 bug 的同一批變更裡重新命名變數、搬移函式、改寫邏輯結構;diff 膨脹到審查者找不到真正的修復在哪一行。
- **改做 Z**:修復的 diff 只含修復本身;重構需求記下來,等修復被驗證與合併後另開任務處理。
- **因為 W**:重構混入修復,會讓「錯誤消失」無法歸因;而重構本身可能引入新錯誤,兩者互相掩蓋。

### AP-DB-11 不留除錯紀錄
- **嚴重度**:風格
- **情況 X**:除錯超過幾個回合,或需要交接、中斷後繼續。
- **絕不做 Y**:試過的假設、排除的原因、關鍵觀察全放在腦中;交付時只說「修好了」,不說根因;下次同類錯誤出現時一切從零開始。
- **改做 Z**:維護簡短的除錯日誌:假設 → 驗證方式 → 結果;交付說明必含一句話根因與驗證證據。
- **因為 W**:除錯的中間發現是有價值的產出;不記錄,等於把已排除的假設留給未來重新排除一遍。

**快速自查(DB)**
- 你有最小重現步驟嗎?修復前後各跑過一次嗎?
- 你能用一句話說出根因嗎?說不出來就還沒修完。
- 最後一輪變更只動了一個變因嗎?
- 有沒有任何測試被刪除、跳過或改了期望值?
- 錯誤訊息的每一行你都真的讀過嗎?

## 資料分析(DA)

### AP-DA-01 不驗資料先分析
- **嚴重度**:阻斷
- **情況 X**:拿到資料集,任務是分析或建模。
- **絕不做 Y**:拿到檔案直接開始算統計量;不檢查缺值、重複列、型別錯亂、明顯異常值(年齡 999、負數金額);不確認資料的時間範圍與涵蓋對象就下結論。
- **改做 Z**:分析前固定執行資料體檢:列數、每欄缺值率、重複列、數值欄的 min/max 與分布、時間範圍;體檢結果寫進報告開頭。
- **因為 W**:垃圾進、垃圾出;分析結論的上限由資料品質決定,跳過體檢的分析從第一個數字起就不可信。

### AP-DA-02 用平均數掩蓋分布
- **嚴重度**:降級
- **情況 X**:報告集中趨勢;資料可能偏態或多峰。
- **絕不做 Y**:只報平均數不看分布(平均薪資被極端值拉高);對雙峰分布報一個「典型值」;比較兩組只比均值、不看變異與重疊。
- **改做 Z**:報平均數時同時報中位數與分布形狀(直方圖或分位數);偏態明顯時以中位數為主要指標;組間比較附上分布重疊程度。
- **因為 W**:平均數只在分布對稱且無極端值時代表典型;其他情況下,它描述的是一個不存在的個體。

### AP-DA-03 相關當因果
- **嚴重度**:阻斷
- **情況 X**:發現兩變數相關;報告需要回答「為什麼」或給出行動建議。
- **絕不做 Y**:把「X 與 Y 相關」直接寫成「X 導致 Y」;依相關性給干預建議(「提高 X 就能提高 Y」);不考慮混淆變數與反向因果的可能。
- **改做 Z**:相關性陳述保持相關性用語;要下因果結論,必須有實驗設計(隨機分派)或明確論證因果識別策略;至少列出一個可能的混淆變數。
- **因為 W**:同一組相關性有多種因果解讀,資料本身無法區分;基於錯誤因果的干預建議會造成實際損失。

### AP-DA-04 幻覺數據
- **嚴重度**:阻斷
- **情況 X**:分析中缺某個數字,或被要求「補齊」表格。
- **絕不做 Y**:編造沒有實際計算過的統計量;資料裡沒有的欄位假裝有並填上數值;引用「業界基準」卻說不出來源;示範用的假數字未標註就混進真實結果。
- **改做 Z**:報告中每個數字都能對應到一段可重跑的程式碼或明確出處;缺的數字就寫「無資料」;示範數據明確標註「示意用,非實際資料」。
- **因為 W**:分析報告的全部價值建立在數字為真;一個被發現是編造的數字,整份分析歸零。

### AP-DA-05 樣本偏差視而不見
- **嚴重度**:降級
- **情況 X**:資料的收集方式有選擇性(只有活躍使用者、只有回覆問卷者、只有存活下來的公司)。
- **絕不做 Y**:把留存使用者的滿意度調查寫成「使用者滿意度高」;用申訴案件推斷整體錯誤率;完全不問「誰不在這份資料裡」。
- **改做 Z**:報告開頭寫明資料的母體與收集機制;列出被排除的群體與可能的偏差方向;結論的適用範圍限定在資料實際代表的母體。
- **因為 W**:樣本偏差不會在資料內部留下痕跡,必須靠收集機制的外部知識識別;忽略它會把局部現象放大成全局結論。

### AP-DA-06 資料洩漏
- **嚴重度**:阻斷
- **情況 X**:建預測模型;切分訓練/測試集;做特徵工程。
- **絕不做 Y**:用全體資料算完標準化參數再切分訓練與測試集;特徵混入結果發生之後才知道的資訊(用「是否退款」預測「是否購買」);時間序列用未來資料預測過去。
- **改做 Z**:先切分、再做任何具擬合性質的前處理;逐一檢查每個特徵在預測時點是否可得;時間序列一律按時間切分。
- **因為 W**:洩漏讓離線指標虛高、模型上線即失效;這是「指標漂亮但產品失敗」的頭號原因。

### AP-DA-07 多重比較不校正
- **嚴重度**:降級
- **情況 X**:同時檢定多個假設、多個切片、多個指標,或反覆嘗試直到「顯著」為止。
- **絕不做 Y**:跑二十個切片,找到一個 p<0.05 就報告「發現顯著差異」;不斷換指標、換時間窗直到出現想要的結果;只報顯著的比較,不報總共做了幾次。
- **改做 Z**:事先宣告主要假設與主要指標;報告所有做過的比較次數;多重比較使用校正方法,或明確標註「探索性發現,需獨立驗證」。
- **因為 W**:比較做得夠多,純雜訊也會產生「顯著」;不揭露比較次數的顯著性沒有意義。

### AP-DA-08 口徑不一致
- **嚴重度**:阻斷
- **情況 X**:整合多個來源的指標,或指標本身有多種定義(活躍使用者、營收、轉換率)。
- **絕不做 Y**:同一份報告混用日活與月活來比較增長;把含稅營收與未稅營收放進同一張表;不同時區、幣別、單位的數據直接相加。
- **改做 Z**:每個指標首次出現時寫明定義(分子、分母、時間窗、過濾條件);跨來源整合前先對齊口徑並記錄轉換方式;無法對齊的不放進同一張表。
- **因為 W**:口徑錯配產出的數字看起來正常但完全錯誤,是報表類事故的最大來源。

### AP-DA-09 小樣本大結論
- **嚴重度**:降級
- **情況 X**:切片後樣本變小;新功能剛上線、資料還很少。
- **絕不做 Y**:對 n=15 的切片報告「轉換率提升 40%」而不附樣本數;把一週資料的走勢外推成長期趨勢;對小樣本差異不做任何顯著性或區間評估。
- **改做 Z**:每個統計量旁附樣本數;小樣本結論附信賴區間,或明標「樣本不足,僅供參考」;設定最低樣本門檻,低於門檻的切片不單獨下結論。
- **因為 W**:小樣本的波動天然巨大;不附樣本數的百分比是統計上的無擔保支票。

### AP-DA-10 圖表誤導
- **嚴重度**:降級
- **情況 X**:製作圖表呈現差異或趨勢。
- **絕不做 Y**:截斷 Y 軸把 2% 的差異畫成兩倍高;雙 Y 軸各自縮放暗示兩線相關;用 3D 圓餅圖扭曲比例;挑選起訖時間點讓趨勢看起來符合想講的故事。
- **改做 Z**:比較量值的長條圖 Y 軸從零開始;避免雙軸,必要時拆成兩張圖;時間範圍以資料完整範圍為預設,截選必須註明理由。
- **因為 W**:圖表是讀者最信任、最少驗算的元素;視覺誤導比數字錯誤更難被發現。

### AP-DA-11 結果不可重現
- **嚴重度**:降級
- **情況 X**:分析包含多步驟處理;使用了互動式工具或手工操作。
- **絕不做 Y**:在試算表裡手工改值、手工排序後截圖進報告;清洗步驟只存在於做完就丟的指令歷史;隨機過程不設種子,重跑一次數字全變。
- **改做 Z**:從原始資料到最終數字的每一步都寫成可重跑的程式碼;隨機過程固定種子;報告附程式碼位置與執行方式。
- **因為 W**:不可重現的分析無法被審查、更新或除錯;它的正確性只能靠信仰。

### AP-DA-12 圖表缺標註
- **嚴重度**:風格
- **情況 X**:任何圖表產出。
- **絕不做 Y**:圖沒有標題、軸沒有單位、看不出時間範圍與樣本數;圖例缺失,讓讀者猜哪條線是哪一組;圖表離開上下文後完全無法獨立解讀。
- **改做 Z**:每張圖固定檢查五件事:標題、軸標籤與單位、時間範圍、樣本數、圖例;以「這張圖被單獨轉傳時仍可正確解讀」為標準。
- **因為 W**:圖表會脫離報告單獨流傳;缺標註的圖在流傳中必然被誤讀。

**快速自查(DA)**
- 資料體檢(缺值、重複、異常值、時間範圍)做了嗎?結果寫進報告了嗎?
- 有沒有任何因果語句,背後其實只有相關性證據?
- 每個數字都能對應到可重跑的程式碼或明確出處嗎?
- 每個百分比旁邊有樣本數嗎?
- 所有指標的口徑(分子、分母、時間窗)都寫明了嗎?
