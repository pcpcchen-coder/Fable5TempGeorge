# PLANNING.md — 通用規劃協議

## 本文件目的

本文件把「專家接到陌生任務時的思考過程」外化成八階段協議。執行模型(如 Opus)接到任何任務時,必須依本協議逐階段推進,並在動手前完整填寫文末的「規劃輸出模板」。目標是讓產出品質取決於流程,而不是模型的即興發揮。

## 如何使用本文件

1. 接到任務後,先判斷是否符合「輕量模式」條件(見本文最末節);不符合就走完整八階段。
2. 依序執行八階段:**需求澄清 → 問題重述 → 範圍界定 → 方案枚舉與取捨 → 分解 → 定義驗收 → 執行 → 自審**。順序不可調換;前一階段的品質門檻全數通過前,不得進入下一階段。
3. 每階段結束時,把該階段的「產出格式」寫進工作紀錄(對應示範文件的思考段落標籤:【澄清】【重述】【範圍】【方案取捨】【分解】【驗收定義】【執行】【自審】)。
4. 進入「執行」階段前,必須先完整填寫文末「規劃輸出模板」的所有欄位。
5. 自審時對照 `GENERAL_RUBRIC.md` 的八維度(R1 正確性、R2 完整性、R3 範圍紀律、R4 邊界處理、R5 可驗證性、R6 簡潔性、R7 可維護性、R8 誠實度)與 `ANTIPATTERNS.md` 中對應任務原型的反模式清單。
6. 需要在多個合理方案間做選擇時,套用 `DECISION_PROTOCOL.md`。

## 八階段總覽

| # | 階段 | 一句話定義 | 段落標籤 |
|---|------|-----------|---------|
| 1 | 需求澄清 | 消除歧義:該問的先問,能假設的明示 | 【澄清】 |
| 2 | 問題重述 | 用自己的話重寫任務,證明真的理解 | 【重述】 |
| 3 | 範圍界定 | 明確寫下做什麼、不做什麼 | 【範圍】 |
| 4 | 方案枚舉與取捨 | 列出至少兩個可行方案並說明取捨 | 【方案取捨】 |
| 5 | 分解 | 把選定方案拆成可獨立驗證的工作項 | 【分解】 |
| 6 | 定義驗收 | 在動手前寫下可驗證的完成條件 | 【驗收定義】 |
| 7 | 執行 | 按分解順序實作,偏離計畫必須記錄 | 【執行】 |
| 8 | 自審 | 以審查者身分對照驗收標準與 Rubric 檢查 | 【自審】 |

## 階段 1:需求澄清

### 目的

任務描述幾乎永遠不完整。此階段的目的是在投入任何工作前,把「會導致產出方向錯誤」的歧義消除:必須問的先問,能安全假設的寫成明示假設。跳過此階段的典型後果是做完才發現做錯方向,全部重來。

### 具體動作

1. 通讀任務描述兩次,逐句標出:模糊詞(「優化」「改善」「支援」)、未定義名詞、隱含前提。
2. 對照下方「必須先問使用者」清單,逐條檢查任務是否命中;命中任一條,就先提問,不動手。
3. 對照「可自行合理假設但必須明示」清單,把可假設的項目寫成標準格式的假設句。
4. 檢查任務涉及的檔案、系統、資料是否真的存在且可存取;不確定就先驗證(讀檔、列目錄),不要憑想像規劃。
5. 把「已確認事實 / 明示假設 / 待確認問題」三類分開列出,寫進【澄清】段落。

### 產出格式

【澄清】段落包含三個小節:已確認、假設、提問。範例:

```markdown
【澄清】
已確認:
- 目標檔案 src/parser.py 存在,現有測試 12 個全數通過。
假設:
- 假設:輸出格式沿用現有 JSON schema;若不符請告知,影響範圍是輸出層一個函式。
提問:
- 無(未命中「必須先問」清單)。
```

### 品質門檻

- [ ] 「必須先問使用者」清單的每一條都已逐一檢查,命中的都已提問,且提問後未擅自動手。
- [ ] 每一個假設都用標準格式寫出(含「若不符請告知」與影響範圍),沒有藏在腦中的隱性假設。
- [ ] 任務中每個模糊詞(「優化」「支援」「盡快」等)都已被翻譯成可驗證的具體描述,或列入提問。
- [ ] 任務提到的每個檔案 / 系統 / 資料來源,都已實際驗證存在,或明確標記為待確認。

### 常見錯誤

1. **過早動手**:看到任務就開始寫程式碼,把澄清當成可跳過的儀式。歧義不會因為動手而消失,只會變成返工。
2. **假問題**:問使用者「要不要我做得好一點」這類無資訊量的問題,或問文件裡已寫明的事。
3. **隱性假設**:心裡假設了「使用者要的是 X」但沒寫出來,產出後才發現使用者要的是 Y。
4. **一次問一題**:分五輪各問一個問題,浪費使用者五個來回。問題必須一次問完。

### 提問框架(需求澄清專用)

#### A. 必須先問使用者的情況

命中下列任一情況時,**禁止**以假設代替提問:

| # | 情況 | 範例 |
|---|------|------|
| 1 | 目標互相矛盾 | 任務要求「回應時間降到 100ms 以下」同時「不得增加任何快取或機器」,兩者在現有架構下不可能同時成立。 |
| 2 | 涉及不可逆動作 | 任務含刪除正式環境資料表、強制推送覆蓋遠端分支、寄信給外部客戶——做錯無法復原。 |
| 3 | 缺少驗收標準且無法從上下文推得 | 「把報表弄得更好看」——沒有任何線索指出「好看」的判準是誰說了算、對照什麼。 |
| 4 | 多種合理解讀導向完全不同的產出 | 「支援多語言」可以指 UI 翻譯(i18n)、也可以指程式解析多種程式語言,兩者工作量與產出完全不同。 |
| 5 | 涉及安全、法務、金流、權限等高風險領域的判斷 | 「幫使用者自動同意條款」——是否合規不是執行模型能替使用者決定的。 |
| 6 | 需要使用者才有的資訊或權限 | 任務需要正式環境的 API 金鑰、內部規範文件、或只有使用者知道的業務規則。 |
| 7 | 任務規模與明示的時間 / 資源限制明顯衝突 | 「今天內重寫整個認證系統」——必須先確認是縮範圍還是延時程。 |
| 8 | 會覆蓋或大幅改動使用者現有成果 | 任務隱含要重寫使用者剛寫好的模組——先確認保留哪些、可動哪些。 |

#### B. 可自行合理假設但必須明示的情況

下列情況允許不問、直接假設,但**每一條假設都必須寫出來**:

| # | 情況 | 範例 |
|---|------|------|
| 1 | 慣例性技術選擇 | 專案已用 pytest,新測試沿用 pytest 而不另問。 |
| 2 | 可低成本修改的細節 | 變數命名、錯誤訊息措辭、報告的章節順序——改起來只要幾分鐘。 |
| 3 | 業界通行預設值 | 未指定編碼就用 UTF-8;未指定時區就用 UTC 並明示。 |
| 4 | 專案內已有一致前例 | 現有程式碼全部用 4 空格縮排,直接沿用。 |
| 5 | 不影響驗收結果的內部實作選擇 | 內部輔助函式要不要抽出來,只要外部行為與驗收標準不變即可自行決定。 |

**明示假設的標準寫法**(固定格式,不得省略任一部分):

```text
假設:{假設內容};若不符請告知,影響範圍是 {改動成本與波及範圍}。
```

範例:

```text
假設:日期格式輸出為 ISO 8601(YYYY-MM-DD);若不符請告知,影響範圍是格式化函式一處,約 5 分鐘可改。
```

#### C. 提問的品質要求

1. **一次問完**:把所有問題整理成一則訊息、編號列出。禁止分多輪逐題問。
2. **給選項與預設**:每個問題附上你認為合理的選項(A/B/C),並標出你建議的預設值,讓使用者可以只回「都用預設」。
3. **說明為何而問**:每題一句話說明這個答案會如何改變產出,證明不是為問而問。

提問格式範例:

```markdown
開工前需要確認 2 件事(都可只回選項字母):
1. 「多語言支援」指哪一種?(影響整體架構)
   A. UI 介面翻譯(i18n) B. 解析多種程式語言 —— 建議:A
2. 舊資料表可否刪除?(不可逆動作,必須明確授權)
   A. 可刪 B. 保留並改名封存 —— 建議:B
```

## 階段 2:問題重述

### 目的

用自己的話把任務重寫一遍,是最便宜的理解檢查。能正確重述,才證明理解的是任務本身而不是字面;重述若有偏差,使用者能在任何工作發生前就糾正,成本趨近於零。

### 具體動作

1. 用 2-5 句話重寫任務:要解決的問題是什麼、為誰解決、成功時世界有什麼不同。
2. 區分「使用者說的手段」與「使用者要的目的」:如果任務指定了手段(「加一個快取」),寫出背後目的(「讓查詢變快」),並檢查手段是否真能達成目的。
3. 標出任務所屬的任務原型(software-implementation / system-design / research-analysis / technical-writing / debugging / data-analysis 之一或組合),據此載入對應的反模式清單。
4. 列出 1-3 條「如果我理解錯,最可能錯在哪」的自我懷疑點。

### 產出格式

```markdown
【重述】
問題:查詢頁面在資料量超過 1 萬筆時載入超過 10 秒,使用者要的是把體感等待降到可接受範圍(目的),提議的手段是加快取。
原型:software-implementation(SW)+ debugging(DB)。
可能誤解點:使用者說的「載入慢」也可能是前端渲染慢而非查詢慢——執行前先量測確認瓶頸。
```

### 品質門檻

- [ ] 重述沒有逐句抄任務原文;任何一句都換成了自己的表達。
- [ ] 已寫出「目的」而不只是「手段」;若任務指定的手段可能達不成目的,已明確標出。
- [ ] 已標出任務原型(六種之一或組合)。
- [ ] 至少列出一條具體的可能誤解點(不是「可能有誤解」這種空話)。

### 常見錯誤

1. **複讀機式重述**:把任務原文換幾個字抄一遍,沒有增加任何理解。
2. **手段當目的**:使用者說「加快取」就只想快取,不檢查瓶頸是否真的在查詢層。
3. **原型誤判**:把 debugging 任務當成 software-implementation 做,跳過重現與定位直接改程式碼。

## 階段 3:範圍界定

### 目的

明確寫下「做什麼」與「不做什麼」。不寫非目標,執行時每遇到一個相鄰問題就多做一點,最後產出膨脹、核心反而沒做好——這是 R3 範圍紀律直接對應的階段。

### 具體動作

1. 把任務拆成「使用者明確要求的」與「我認為順便做比較好的」兩堆;後者預設全部列為非目標。
2. 寫出目標清單:每條目標都必須能對應到至少一條後續的驗收標準。
3. 寫出非目標清單:明確列出「相鄰但這次不做」的事,並各附一句不做的理由。
4. 標出邊界模糊處:哪些項目介於做與不做之間,寫下裁定與理由。
5. 執行中若發現範圍必須擴大,回到此階段更新清單並明示,不得默默擴張。

### 產出格式

```markdown
【範圍】
目標:
1. 修復 /search 端點在 1 萬筆以上資料的逾時問題。
非目標:
1. 不重構整個查詢模組(只動慢的那條路徑;重構另開任務)。
2. 不新增前端載入動畫(屬 UX 改善,非本次要求)。
邊界裁定:順手發現的 N+1 查詢在同一條路徑上,納入目標;不在路徑上的不動。
```

### 品質門檻

- [ ] 非目標清單至少有一條,且每條都附了不做的理由。
- [ ] 每條目標都能指到一條(後續將寫出的)驗收標準;沒有無法驗收的目標。
- [ ] 「順便做比較好」的項目全數列入非目標,或已取得使用者同意才納入。
- [ ] 範圍在執行中的任何變動都有書面紀錄,沒有默默擴張或默默縮水。

### 常見錯誤

1. **範圍蔓延**:修一個 bug 順便重構三個檔案,diff 大到無法審查。
2. **只寫目標不寫非目標**:非目標清單空白,等於沒有界定範圍。
3. **默默縮水**:做不完就悄悄少做,不告知使用者哪些沒做——這同時違反 R8 誠實度。

## 階段 4:方案枚舉與取捨

### 目的

第一個想到的方案往往不是最好的。強制枚舉至少兩個可行方案並寫下取捨,能避免「錨定在第一直覺」,也留下決策紀錄讓使用者與後人能檢驗選擇是否合理。

### 具體動作

1. 枚舉至少 2 個(建議 2-4 個)真正可行的候選方案;「什麼都不做 / 最小改動」永遠算一個合法候選。
2. 對每個方案寫出:一句話描述、主要優點、主要缺點、大致成本。
3. 依任務目標挑出 2-4 個比較維度(如:正確性風險、實作成本、可維護性、可逆性),逐方案比較。
4. 選定一個方案,寫出選擇理由;對每個被否決的方案,寫出一句否決理由。
5. 若兩個方案難分高下,套用 `DECISION_PROTOCOL.md`;若選擇會大幅影響使用者(成本、鎖定效應),回到階段 1 的「必須先問」判斷。

### 產出格式

```markdown
【方案取捨】
方案 A:對現有查詢加複合索引。優點:改動最小、可逆。缺點:寫入略慢。成本:小。
方案 B:引入 Redis 快取層。優點:讀取最快。缺點:新增基礎設施、快取失效複雜。成本:大。
比較維度:可逆性、實作成本、維運負擔。
選定:A。理由:先用最小改動驗證瓶頸假設;不足再升級到 B。
否決 B:在未證實索引不夠之前,引入新基礎設施是過度設計。
```

### 品質門檻

- [ ] 至少列出 2 個真正可行的方案;沒有為了湊數而寫的稻草人方案(明顯不可行、只為襯托首選)。
- [ ] 每個被否決的方案都有具體否決理由(能回答「為什麼不選它」)。
- [ ] 比較維度明確寫出,且與任務目標相關,而不是泛泛的「效能、可讀性」套話。
- [ ] 選定理由引用了比較結果,而不是「我覺得比較好」。

### 常見錯誤

1. **單一方案直衝**:只想到一個做法就開工,從未考慮更簡單的替代。
2. **稻草人比較**:列一個明顯爛的方案來襯托首選,假裝做過取捨。
3. **過度設計**:在小問題上選大架構(見反模式清單中各原型的過度設計條目),忽略「最小改動」候選。

## 階段 5:分解

### 目的

把選定方案拆成可獨立完成、可獨立驗證的工作項,並排出依賴順序。好的分解讓執行變成「按清單走」,讓進度可見、讓中途失敗時能知道停在哪、已完成的哪些仍然有效。

### 具體動作

1. 把方案拆成 3-10 個工作項;每項都要小到「單次專注可完成」且「完成與否可客觀判斷」。
2. 為每個工作項寫一句完成定義(這一項做完的可見證據是什麼)。
3. 標出工作項之間的依賴,排出執行順序;能提早暴露風險的項目(如驗證關鍵假設)排最前。
4. 標出可並行與必須序列的項目;標出哪些項目失敗時會動搖整個方案(高風險項)。
5. 檢查分解的聯集是否覆蓋全部目標:每條目標至少被一個工作項覆蓋。

### 產出格式

```markdown
【分解】
1. 量測確認瓶頸在 DB 查詢(高風險:若否,回到方案取捨)。完成定義:有量測數據。依賴:無。
2. 加複合索引 migration。完成定義:migration 檔存在且可執行。依賴:1。
3. 補回歸測試(1 萬筆資料下查詢 < 2 秒)。完成定義:測試存在且通過。依賴:2。
4. 更新文件。完成定義:README 索引說明更新。依賴:2。(可與 3 並行)
```

### 品質門檻

- [ ] 每個工作項都有完成定義,且完成與否可由第三者客觀判斷。
- [ ] 依賴關係已標出,順序無循環;高風險 / 驗證假設類項目排在前面。
- [ ] 所有目標都被至少一個工作項覆蓋;沒有任何工作項落在非目標範圍內。
- [ ] 沒有「其餘部分」「等等」這類含糊項;每項粒度小到單次專注可完成。

### 常見錯誤

1. **偽分解**:拆成「1. 開始做 2. 做完」這種沒有資訊量的清單。
2. **風險後置**:把最不確定的部分排最後,前面做完才發現方案根本不通,前功盡棄。
3. **粒度失控**:單項大到「重寫整個模組」,或細到「打開檔案」——前者不可驗證,後者是噪音。

---

## 階段 6:定義驗收

### 目的

在動手前寫下「做到什麼程度才算完成」的可驗證條件。驗收標準寫在執行之前,才能約束執行;寫在執行之後,只會變成替既有產出辯護的說詞。這是 R5 可驗證性的直接來源。

### 具體動作

1. 為每條目標寫至少一條驗收標準;每條標準必須寫成「可執行的檢查」:跑什麼命令、看什麼輸出、比對什麼條件。
2. 加入負面條件:哪些東西必須「沒有壞」(既有測試全過、不新增警告、非目標區域零改動)。
3. 為不可自動驗證的產出(文件、分析報告)定義人工檢核清單:逐條可勾選的判斷句。
4. 標出每條標準的驗證方式與成本;成本高的標準,寫出退而求其次的抽查方式。
5. 把全部標準填入規劃輸出模板的「驗收標準」欄;若專案使用 `bootstrap/templates/verify.sh`,把可自動化的條目接上。

### 產出格式

```markdown
【驗收定義】
1. `pytest tests/ -x` 全數通過(含新增回歸測試 test_search_10k)。
2. 在 1 萬筆種子資料下,`GET /search?q=abc` 回應時間 < 2 秒(以 scripts/bench.sh 量測)。
3. `git diff --stat` 只涉及 migrations/、tests/、docs/ 三個目錄——非目標區域零改動。
4. 人工檢核:migration 有對應的 down 腳本,可回復。
```

### 品質門檻

- [ ] 每條標準都寫明「怎麼驗」:命令、輸入、預期輸出或判斷句;不存在「品質良好」「運作正常」這類不可判斷的字句。
- [ ] 每條目標至少對應一條標準;每條標準都能回溯到某條目標。
- [ ] 至少包含一條負面條件(不破壞既有行為 / 不越出範圍)。
- [ ] 驗收標準在執行開始前就已定稿;執行中若修改,已記錄修改原因。

### 常見錯誤

1. **事後補寫**:先做完再回頭寫驗收標準,標準退化成產出的描述。
2. **不可驗證的標準**:「程式碼要乾淨」「文件要清楚」——沒有任何人能客觀判定通過與否。
3. **只驗新功能**:忘記寫「既有測試全過」,結果修好一處弄壞三處而不自知。

---

## 階段 7:執行

### 目的

按分解順序把工作項逐一做完。此階段的紀律是:跟著計畫走、偏離必須記錄、卡住必須誠實。執行不是重新規劃;發現計畫錯了,就明示地回到對應階段修訂,而不是邊做邊漂移。

### 具體動作

1. 依分解順序逐項執行;每完成一項,立刻對照該項完成定義自查,再標記完成。
2. 執行前先讀相關的現有程式碼 / 文件 / 資料,遵循現有慣例;不確定 API 或行為時,實際查證而不是憑記憶猜。
3. 任何偏離計畫的決定(改順序、換做法、加減工作項)都寫一行紀錄:偏離什麼、為什麼、影響哪些後續項。
4. 遇到與關鍵假設矛盾的事實,停下:回到受影響的階段(通常是方案取捨或範圍界定)修訂,再繼續。
5. 同一個錯誤嘗試兩次仍失敗,停止盲試:寫下已排除的可能與目前卡點,換診斷方法或向使用者回報。
6. 過程中產生的臨時檔、除錯輸出,在階段結束前清理。

### 產出格式

```markdown
【執行】
- [x] 項 1:量測完成,DB 查詢佔 9.2s/10.1s,瓶頸確認。
- [x] 項 2:migration 0042 已建立並在本地執行成功。
- 偏離紀錄:項 3 原計畫用既有 fixture,實測資料量不足,改為新增 seed 腳本(影響:項 3 多花約 20 分鐘,無其他波及)。
- [x] 項 3:回歸測試通過(1.4s < 2s)。
- [x] 項 4:文件更新。
```

### 品質門檻

- [ ] 每個工作項都對照其完成定義自查後才標記完成;沒有「大概好了」就打勾的項目。
- [ ] 所有偏離計畫之處都有一行紀錄(偏離內容、原因、影響);紀錄為零且產出與計畫不符,即為不通過。
- [ ] 不確定的 API / 行為 / 資料格式都經過實際查證(讀原始碼、跑最小實驗),沒有憑記憶硬寫。
- [ ] 沒有為了通過驗收而弱化驗收標準(如放寬閾值、刪測試);若標準確需修改,已記錄原因。

### 常見錯誤

1. **邊做邊漂移**:做著做著偏離計畫卻不記錄,最後產出與規劃對不上,無從審查。
2. **盲目重試**:同一招失敗三四次還在試,不改變診斷方法、不回報。
3. **憑記憶寫 API**:不查證就寫,參數名、回傳格式全靠印象,錯了再猜。
4. **矇混驗收**:發現達不到標準,就悄悄放寬標準或跳過該測試,而不是誠實回報。

---

## 階段 8:自審

### 目的

交付前切換成「審查者」身分,把自己的產出當成別人的來挑毛病。自審的產出是誠實的品質聲明:哪些驗收過了、哪些沒過、哪裡是弱點——這是 R8 誠實度的最後防線。

### 具體動作

1. 逐條執行階段 6 的驗收標準,記錄每條的實際結果(通過 / 未通過 / 未驗,附證據)。
2. 對照 `GENERAL_RUBRIC.md` 八維度(R1-R8)逐一給出簡評,重點寫最弱的兩個維度。
3. 對照 `ANTIPATTERNS.md` 中本任務原型的反模式清單(AP-{前綴}-NN),逐條檢查是否踩中;踩中「阻斷」級的必須修復後重審。
4. 檢查範圍紀律:diff / 產出範圍與階段 3 的目標與非目標比對,越界處要麼修掉、要麼明示理由。
5. 寫出交付聲明:完成了什麼、沒完成什麼、已知限制與弱點、建議的後續事項。

### 產出格式

```markdown
【自審】
驗收結果:1 通過(pytest 47 passed)/ 2 通過(1.4s)/ 3 通過(diff 僅三目錄)/ 4 通過(down 腳本已驗證)。
Rubric 弱點:R4 邊界處理——空查詢字串路徑未加測試,行為沿用舊版;R6 簡潔性——seed 腳本可再精簡。
反模式檢查:AP-SW 清單逐條過,未踩中阻斷級。
已知限制:僅在本地 PostgreSQL 15 驗證,未在正式環境量測。
```

### 品質門檻

- [ ] 每條驗收標準都有「通過 / 未通過 / 未驗」的明確結果與證據;沒有籠統的「都通過了」。
- [ ] 自審找出了至少一個具體弱點或限制;「完美無缺」的自審視為未認真執行。
- [ ] 阻斷級反模式為零;降級 / 風格級的殘留項目已列出並附理由。
- [ ] 未通過或未驗的項目在交付聲明中誠實列出,沒有隱瞞或美化。

### 常見錯誤

1. **走過場自審**:寫一句「已檢查,品質良好」——沒有逐條證據就是沒有自審。
2. **只審新增部分**:不檢查是否弄壞既有功能、是否越出範圍。
3. **報喜不報憂**:明知有未處理的邊界情況,交付聲明卻隻字不提。

## 輕量模式

### 適用條件

同時滿足以下**全部**條件的任務,可用輕量模式:

- [ ] **小**:預估單次專注內可完成,改動範圍不超過 1-2 個檔案或一份短文件。
- [ ] **可逆**:做錯可以低成本復原(版本控制內的改動、可重跑的腳本);不涉及階段 1 表 A 中任何「必須先問」情況。
- [ ] **範圍明確**:任務只有一種合理解讀,驗收標準顯而易見或已由使用者給定。

任一條不滿足,走完整八階段。拿不準時,一律走完整流程——把大任務誤判成小任務的代價,遠高於多花十分鐘規劃。

### 縮減版流程

輕量模式把八階段壓縮為三步,**不可跳過:任務重述、驗收標準、自審**:

| 步驟 | 內容 | 對應完整階段 |
|------|------|-------------|
| 1. 任務重述 | 1-2 句重述任務與範圍;有假設就用標準格式寫出。 | 需求澄清 + 問題重述 + 範圍界定 |
| 2. 驗收標準 | 動手前寫下 1-3 條可驗證的完成條件(含一條「不破壞既有行為」)。 | 定義驗收(方案取捨與分解可省略) |
| 3. 執行與自審 | 做完後逐條核對驗收標準,一句話聲明結果與已知限制。 | 執行 + 自審 |

輕量模式範例(完整示範):

```markdown
【重述】修正 README 安裝章節中過期的套件名(foo-cli → foo-tool);僅改文件,不動程式碼。
【驗收定義】1. README 中不再出現 foo-cli(grep 為零筆)。2. 依新指令可實際安裝成功。3. diff 僅涉及 README.md。
【自審】三條全過(grep 零筆;本地安裝成功;diff 單檔)。限制:未驗證 Windows 環境。
```

### 升級規則

輕量模式執行中,一旦出現下列任一情況,立即停止並升級為完整八階段:發現任務有第二種合理解讀;改動範圍超出原估的檔案;命中任何「必須先問」情況;第一次嘗試失敗且原因不明。升級時,已寫下的重述與驗收標準直接沿用,不算浪費。
