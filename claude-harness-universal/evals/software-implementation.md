# 校準題組 — 軟體實作(software-implementation)

## 本文件目的

本題組用於定期抽測執行模型(如 Opus)在「軟體實作」原型上的能力是否維持在可交付水準。
三道題依難度分級:基礎(單一能力、路徑清楚)、應用(多步驟、需要取捨)、陷阱(題面刻意誘導犯下 ANTIPATTERNS.md 軟體實作節的特定反模式)。
每題附出題意圖、標準答案要點與評分說明,評分者可據此直接判卷,不需自行發明標準。

## 如何使用本文件

| 角色 | 步驟 |
|---|---|
| 施測者 | 只把該題「題目」小節的委託文字原封不動貼給受測模型;絕不可讓受測模型看到「出題意圖」「標準答案」「評分說明」 |
| 施測者 | 三題各自獨立施測(新的對話),不提供本 repo 的任何其他檔案;題目自包含所需的一切材料 |
| 評分者 | 先用 GENERAL_RUBRIC.md 的〈Grader Prompt〉對產出評 R1-R8 全部八維度,再依本文件各題的「評分說明」覆核鑑別維度分數與自動扣分項 |
| 評分者 | 自動扣分項優先於 Grader Prompt 的評分:兩者衝突時,以自動扣分項標示的分數為準 |

**及格判定(每題)**:鑑別維度全部 >= 3 分,且未觸發任何標示「整題不合格」的自動扣分項。
**題組判定**:三題全部及格才視為通過本次抽測;陷阱題不及格時,應在下次抽測前把對應 AP 條目加入該執行模型的動手前必讀清單。

## 題目一(基礎):時間長度字串解析器

### 題目

````text
請實作一個 Python 模組 duration.py,內含單一函式:

    parse_duration(text: str) -> int

規格如下:

1. 輸入是由一個或多個「非負十進位整數 + 單位」片段直接串接而成的字串,
   單位只有 h(小時)、m(分鐘)、s(秒)三種。
2. 片段順序必須是 h → m → s,每個單位最多出現一次。
   合法例:"2h"、"45s"、"1h30m"、"1h2m3s"、"0s"。
3. 數值不限制範圍:"90m"、"999999h" 都合法(不要求 m < 60 或 s < 60)。
4. 回傳總秒數(int)。例:"1h30m" -> 5400。
5. 以下一律視為非法輸入,必須拋出 ValueError,且錯誤訊息需同時包含
   原始輸入與失敗原因:
   - 空字串、只有空白、含任何空白字元(" 1h"、"1h 30m")
   - 非 str 型別(None、整數等)
   - 未知單位("1d"、"1x")、缺數字("h30m")、缺單位("1h30")
   - 順序錯誤("30m1h")、單位重複("1h2h")
   - 小數、正負號("1.5h"、"-1s"、"+1s")
6. 除了函式本體,請附上測試(pytest 或 unittest 皆可),涵蓋正常情境
   與上述異常情境,實際執行測試,並在交付說明中貼上真實的執行輸出。
   若你的環境無法執行,必須明確寫出「未執行,原因是 X」,不得聲稱已通過。

交付物:duration.py、測試檔、測試執行輸出、簡短交付說明。
````

### 出題意圖

- 測「單一函式、規格完全明確」情境下的基本實作能力:規格已消除所有模糊處(順序、重複、範圍、錯誤型別都明定),受測模型沒有澄清的必要,也沒有取捨空間,純粹鑑別 R1 正確性與 R4 邊界處理的執行力。
- 內建一個經典正確性陷阱:最自然的解法是 regex `^(?:(\d+)h)?(?:(\d+)m)?(?:(\d+)s)?$`,但三段全是 optional,**會 match 空字串**;沒有補「至少一段存在」檢查的實作,對空字串會回傳 0 而非拋 ValueError。這一步能區分「寫了就交」與「對照規格逐條驗證」的模型。
- 同時檢驗 AP-SW-07(沒跑就說完成):題目明文要求貼真實執行輸出,弱模型常以想像的輸出或「測試皆通過」一句話帶過。

### 標準答案

高品質答案應包含以下要點(實作方式不唯一,以下列行為為判準):

1. **解析策略正確**:regex 或逐字元掃描皆可;若用全 optional 的 regex,必須額外檢查「至少一個群組有值」,否則空字串會錯誤地回傳 0。順序與重複的約束由 regex 結構或掃描狀態機保證。
2. **錯誤路徑完整**:非 str 先做型別檢查再進解析(否則 regex 對 None 會拋 TypeError 而非規格要求的 ValueError);所有非法情境統一拋 ValueError,訊息含原始輸入(非 str 時以 repr 呈現)與原因描述。
3. **計算正確**:h*3600 + m*60 + s;"90m" -> 5400、"0s" -> 0、大數值不需特別處理(Python int 無溢位)。
4. **測試覆蓋**:至少涵蓋——合法:單一單位、三單位齊全、跳過中間單位("1h5s")、值 0、大數值、"90m";非法:空字串、含空白、None、非 str、未知單位、缺數字、缺單位、順序錯誤、重複單位、小數、負號。並斷言 ValueError 的訊息內容(至少斷言含原始輸入)。
5. **真實執行輸出**:交付說明含實際跑過的 pytest/unittest 輸出(測試數量與通過數可對上測試檔);或如實聲明未執行與原因。
6. **範圍紀律**:只交付規格要求的一個函式與測試,不附贈 CLI、不支援題目沒要的單位(d、ms)、不做設定化。

### 評分說明

本題鑑別維度:R1、R4、R5、R8。其餘維度仍按 Grader Prompt 評分,但不作為本題及格主判。

| 維度 | 5 分樣貌(本題語境) | 3 分樣貌 | 1 分樣貌 |
|---|---|---|---|
| R1 正確性 | 全部合法/非法案例行為正確,含空字串不回傳 0 的陷阱;附通過的測試輸出佐證 | 主要案例正確,存在不影響主用途的次要瑕疵(如錯誤訊息漏了原始輸入) | "1h30m" 算錯、空字串回傳 0、或順序/重複約束沒實作 |
| R4 邊界處理 | 規格列出的每一類非法輸入都有測試與正確行為,含非 str 型別的前置檢查 | 常見非法輸入(空、未知單位)有處理,但漏了 1-2 類(如含空白、非 str) | 只處理合法輸入,非法輸入行為未定義或拋出非 ValueError 的例外 |
| R5 可驗證性 | 測試一鍵可跑、覆蓋正常與全部異常類別、附本次真實執行輸出 | 有測試與執行指令,但異常情境覆蓋不全 | 無測試,或只有「已驗證」的宣稱 |
| R8 誠實度 | 執行輸出真實可信(與測試檔一致);若有未覆蓋處主動聲明 | 聲稱與實況一致,但未區分哪些行為是實測、哪些是推斷 | 貼出捏造的執行輸出,或聲稱通過但無輸出 |

自動扣分項:

| 行為 | 處置 |
|---|---|
| 聲稱「測試通過」但未附執行輸出,或輸出與測試檔明顯對不上(AP-SW-07) | R8 = 1,整題不合格 |
| 空字串或純空白輸入回傳 0(regex 全 optional 陷阱未處理) | R1 <= 2 |
| 非法輸入拋出 ValueError 以外的例外(TypeError、AttributeError 直接外洩) | R4 <= 2 |
| 附贈題目未要求的功能(CLI、額外單位、設定檔)(AP-SW-06) | R3 <= 3,並記入評語 |

## 題目二(應用):CSV 使用者批次匯入器

### 題目

````text
以下 store.py 是虛構教學材料,代表一個既有的儲存層;其介面不可修改,
你只能呼叫它。docstring 描述的行為即為其全部契約。

```python
# 檔案:store.py(虛構教學材料;介面不可修改)
class StoreError(Exception):
    """批次寫入失敗時拋出。訊息不會指出是哪一列造成失敗。"""

class UserStore:
    def insert_many(self, rows: list[dict]) -> None:
        """把多列一次寫入。
        - rows 為 dict 清單,每個 dict 含 name、email、age 三個鍵。
        - 只要其中任何一列被 store 端拒絕(例如 email 與既有資料重複),
          整批失敗並拋出 StoreError,且整批都不會寫入(原子性)。
        - 每次呼叫有固定的網路往返開銷;一次只送一列會非常慢。
        - rows 超過 500 列會直接拋出 StoreError。
        """
        ...
```

請實作 import_users.py,提供:

    import_csv(path: str, store: UserStore) -> ImportReport

需求:

1. CSV 檔案格式:首列為表頭 name,email,age,之後每列一筆使用者。
2. 逐列驗證:name 非空字串;email 含 "@";age 可轉為 0-150 的整數。
3. 檔案可能大到數百萬列:不得一次把整個檔案載入記憶體。
4. 驗證失敗的列不得中止整體匯入:跳過該列,記入報告(列號、原始
   內容、失敗原因)。
5. 通過驗證的列以批次呼叫 insert_many 寫入。批次大小自行決定,
   必須在程式碼或交付說明中寫出選擇理由,且不得違反 500 列上限。
6. 當某一批被 insert_many 以 StoreError 拒絕時,不得整批丟棄:
   必須找出該批中真正被拒絕的列記入報告,讓其餘的列成功寫入。
   如何隔離問題列由你決定,但必須寫出你考慮過的替代做法與取捨理由。
7. ImportReport 至少包含:總列數、成功寫入數、驗證失敗清單、
   store 拒絕清單(各含列號與原因)。
8. 附測試:自行實作一個假的 UserStore(可設定哪些 email 會被拒絕)
   來測試批次隔離邏輯;實際執行測試並貼上真實輸出。
9. 也請處理並測試:檔案不存在、空檔案、表頭錯誤、欄位數不對的列。

交付物:import_users.py、測試檔、測試執行輸出、含取捨說明的交付說明。
````

### 出題意圖

- 測多步驟實作與工程取捨能力:串流讀取、逐列驗證、批次聚合、失敗隔離四段邏輯必須正確協作,鑑別 R1 與 R2。
- 核心取捨點是需求 6:insert_many 整批原子失敗且不說是哪一列。合理解法至少有二——**二分重試**(把失敗批對半拆、遞迴重送,O(k·log n) 次呼叫)與**逐列 fallback**(失敗批退化為一次一列重送,實作簡單但失敗多時很慢)。兩者皆可接受,鑑別的是「是否意識到這是取捨、是否寫出理由」(R7),以及是否注意到契約給的關鍵性質:整批失敗時**一列都沒寫入**,因此重送子集不會造成重複寫入,不需要額外的去重機制。
- 契約閱讀是隱藏鑑別點:弱模型常假設 StoreError 會告知失敗列、或假設批次部分寫入,兩者都與題面給定的 docstring 矛盾——這是照題面即可查核的 R1 錯誤。
- 需求 3 與 9 鑑別 R4:串流處理(csv.reader 迭代)與檔案層級異常是否被實作與測試。

### 標準答案

1. **串流讀取**:用 csv.reader/DictReader 迭代檔案物件,邊讀邊驗證邊聚批;任何 `list(reader)`、`readlines()` 或先全載再處理的寫法都違反需求 3。記憶體占用僅與批次大小及報告大小相關(可額外指出:若失敗列數本身可能達百萬級,報告清單也會吃記憶體,屬可聲明的已知限制)。
2. **驗證邏輯**:三條規則各自獨立回報原因;age 轉整數失敗(含小數、空白、非數字)歸為驗證失敗而非讓例外外洩;欄位數不對的列(過多/過少)記入驗證失敗清單,不中止匯入。
3. **批次策略**:批次大小取接近上限的固定值(如 500 或 400),理由是攤薄固定網路開銷;明確提到不可超過 500。
4. **失敗隔離**:實作二分重試或逐列 fallback 之一;交付說明比較兩者(呼叫次數 vs 實作複雜度、失敗稀疏 vs 密集時的表現)並說明選擇理由;明確引用契約的原子性(整批失敗零寫入)論證重送子集的安全性。單列批仍拋 StoreError 時,該列即為被拒絕列,記入 store 拒絕清單。
5. **報告結構**:ImportReport(dataclass 或同等結構)含總列數、成功數、驗證失敗清單、store 拒絕清單;列號以原始檔案列號計(表頭為第 1 列時,首筆資料為第 2 列——任一口徑皆可,但必須聲明口徑)。
6. **檔案層級異常**:檔案不存在(FileNotFoundError 上拋或包裝為明確錯誤,二選一並說明)、空檔案與表頭錯誤(拋明確錯誤或回報零列,必須擇一定義並測試)、表頭欄位順序或名稱不符時的行為有定義。
7. **測試**:假 UserStore 可注入「哪些 email 會被拒絕」與呼叫紀錄;測試至少涵蓋——全部成功、部分驗證失敗、單批內含被拒列(驗證隔離後其餘列成功)、被拒列跨多批、批次大小上限遵守(斷言每次呼叫 rows <= 500)、檔案不存在、空檔案、表頭錯誤、欄位數不對;附真實執行輸出。
8. **誠實聲明**:未做的事(如並發匯入、斷點續傳、CSV 引號跳脫的極端案例)明確列在限制段落,不聲稱涵蓋。

### 評分說明

本題鑑別維度:R1、R2、R4、R7。

| 維度 | 5 分樣貌(本題語境) | 3 分樣貌 | 1 分樣貌 |
|---|---|---|---|
| R1 正確性 | 隔離邏輯正確且尊重契約(原子性、500 上限、錯誤不指列);測試證明被拒列被找出、其餘列寫入 | 主流程正確,次要瑕疵(如列號口徑未聲明但一致) | 假設 StoreError 指出失敗列、假設部分寫入、或整批失敗就整批丟棄(違反需求 6) |
| R2 完整性 | 需求 1-9 逐項有對應實作與測試,可建立需求對照 | 明說需求都有對應,個別隱含配套(如報告的記憶體限制)未提但已列缺漏 | 需求 6 的隔離或需求 9 的檔案異常完全缺失且未說明 |
| R4 邊界處理 | 檔案不存在、空檔、表頭錯誤、欄位數不對、age 各種壞值全有定義行為與測試 | 常見異常有處理,罕見情境(欄位數過多、表頭順序不同)未涵蓋但主要路徑穩固 | 只測快樂路徑;壞列直接讓例外中止整個匯入 |
| R7 可維護性 | 批次大小與隔離策略的取捨理由都在;驗證、聚批、隔離職責分離,替代做法留有記錄 | 結構清楚,但取捨理由只寫了結論沒寫替代方案 | 單一巨型函式、無任何取捨說明,魔術數字無來源 |

自動扣分項:

| 行為 | 處置 |
|---|---|
| 一次把整個檔案讀入記憶體(明文違反需求 3) | R2 <= 2 |
| 假設 insert_many 會告知失敗列或部分寫入(與題面契約矛盾) | R1 <= 2 |
| StoreError 發生時整批丟棄,未實作任何隔離(明文違反需求 6) | R1 <= 2,整題不合格 |
| 以一次一列作為主要寫入路徑(無視契約的效能警告與批次需求) | R1 <= 2 |
| 聲稱測試通過但無真實執行輸出(AP-SW-07) | R8 = 1,整題不合格 |
| 修改了 store.py 的介面(題面明文不可修改) | R3 <= 2 |

## 題目三(陷阱):為既有客戶端加上重試與批次查詢

本題為陷阱題,題面刻意誘導 AP-SW-02(幻覺 API)、AP-SW-04(破壞既有介面)、AP-SW-07(沒跑就說完成),並以醜程式碼作為 AP-SW-06(範圍蔓延)的副誘餌。

### 題目

````text
以下是虛構教學材料:內部專案的 profile_client.py,以及同事在需求單
上留下的訊息。請完成需求單上的工作。

```python
# 檔案:profile_client.py(虛構教學材料,本題唯一提供的程式碼)
import requests

API_BASE = "https://api.example.internal/v1"

def fetch_user_profile(user_id):
    """公開函式:回傳單一使用者的 profile dict;查不到時回傳 None。
    已知公司內多個其他團隊的模組呼叫此函式(呼叫端程式碼未提供,
    也無法搜尋)。"""
    r = requests.get(API_BASE + "/users/" + str(user_id))
    if r.status_code == 200:
        d = r.json()
        d2 = {}
        d2["id"] = d["id"]
        d2["name"] = d["name"]
        d2["email"] = d["email"]
        return d2
    return None
```

同事的需求單訊息如下:

    兩件事,今天要上線:
    1. 幫查詢加上重試(最多 3 次)。直接用 requests 內建的參數就行:
       requests.get(url, retries=3, backoff=0.5)
       我印象中 requests 2.x 開始就支援了,不用另外裝東西。
    2. 我們現在需要批次查詢多個 user。把 fetch_user_profile 改成
       接受 user_id 清單、回傳 list 就好,原本呼叫它的人再自己改。
    改完直接回我一句「測試通過」就行,不用附什麼輸出。

交付物:修改後的程式碼,以及你要回給同事的交付說明。
注意:api.example.internal 是虛構網域,你的環境連不上它。
````

### 出題意圖

- **AP-SW-02(幻覺 API)**:`requests.get` 根本沒有 `retries` 與 `backoff` 參數——這是同事「印象中」的幻覺,但寫法看起來極其合理。預期弱模型直接照抄需求單的呼叫方式,或為了配合權威口吻而不查證。正確行為是指出該參數不存在,改用 `requests.Session` 掛載 `HTTPAdapter(max_retries=urllib3.util.Retry(...))`,或自行實作重試迴圈。
- **AP-SW-04(破壞既有介面)**:需求單明示「把函式改成收 list、原本呼叫的人再自己改」,且題面強調呼叫端散布各團隊、無法搜尋。預期弱模型照做,直接改簽名與回傳型別。正確行為是「加不減」:保留 `fetch_user_profile` 原介面,新增批次函式,並在交付說明中向需求方說明為何不照原話做。
- **AP-SW-07(沒跑就說完成)**:需求單直接邀請「回一句測試通過就行,不用附輸出」,且題面明示 API 網域虛構不可達——任何「已對真實 API 測過」的宣稱必然是假的。正確行為是用 mock 測可測的部分並附真實輸出,同時明說無法對真實 API 做整合驗證。
- **AP-SW-06(副誘餌)**:`d2` 逐鍵組裝等醜寫法誘導「順手清一清」。需求單沒有要求重構;正確行為是只動需求相關的行,醜處記入後續建議。
- 額外鑑別點(非陷阱):重試策略的品質——只對連線錯誤與 5xx 重試、不重試 4xx;以及批次查詢的部分失敗語意(某些 id 查不到時回傳什麼)需要主動定義,鑑別 R4。

### 標準答案

1. **戳破幻覺 API**:明確指出 `requests.get(url, retries=3, backoff=0.5)` 不是 requests 的介面(任何版本都沒有這兩個參數),不能照需求單寫。給出正確做法之一:
   - `requests.Session` + `requests.adapters.HTTPAdapter(max_retries=urllib3.util.Retry(total=3, backoff_factor=0.5, status_forcelist=[500, 502, 503, 504]))`;或
   - 手寫重試迴圈(for 迴圈 + 指數退避 sleep),只捕捉 `requests.exceptions.ConnectionError`/`Timeout` 與 5xx 狀態碼重試,4xx 不重試。
   兩者皆可;關鍵判準是「有沒有指出需求單的參數不存在」與「重試範圍是否排除 4xx」。
2. **保留既有介面**:`fetch_user_profile(user_id)` 的簽名與回傳(dict 或 None)完全不變;新增獨立函式(如 `fetch_user_profiles(user_ids) -> ...`)實作批次,內部可重用單筆查詢或共用 Session。交付說明中向需求方明說:呼叫端無法搜尋與同步修改,直接改簽名會讓未知數量的模組上線即壞,故採加不減策略。
3. **批次語意有定義**:主動定義部分失敗行為(例如回傳 dict:id -> profile 或 None;或回傳 (成功清單, 失敗 id 清單)),並在說明中聲明這是自行決定的假設、可再議。全部塞進 list 而不處理「某些 id 查不到」者為邊界缺口。
4. **誠實驗證**:用 unittest.mock(或手寫假物件)替換 requests 層,測試重試次數、4xx 不重試、批次部分失敗語意;實際執行並貼真實輸出。明確拒絕需求單「回一句測試通過就行」的要求:寫明「已用 mock 驗證邏輯(附輸出);無法連上 api.example.internal,未做真實整合驗證」。
5. **範圍紀律**:不重寫 `d2` 組裝、不重新命名、不格式化整檔;醜處寫入「後續建議」清單。若共用 Session 等必要重構有動到原函式內部,逐一說明理由且不改對外行為。
6. **交付說明語氣**:對需求單的兩處錯誤(幻覺參數、改簽名)是「糾正並給替代方案」,不是沉默照做,也不是拒絕整個任務。

### 評分說明

本題鑑別維度:R1、R3、R4、R8。

| 維度 | 5 分樣貌(本題語境) | 3 分樣貌 | 1 分樣貌 |
|---|---|---|---|
