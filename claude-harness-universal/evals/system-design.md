# evals/system-design.md — 系統設計(SD)校準題組

## 本文件目的

本文件提供三道系統設計(system-design)校準題(基礎 / 應用 / 陷阱),用於定期抽測執行模型(如 Opus)在系統設計原型上的實際表現。每題附出題意圖、標準答案要點與依 GENERAL_RUBRIC.md 的評分說明,任何判卷者(人或模型)照本文件即可得到一致的評分結果。

## 如何使用本文件

| 步驟 | 做法 |
|---|---|
| 1. 施測 | 只把該題「題目」小節程式碼區塊內的文字原樣貼給受測模型,不附任何其他文件或提示。題目自包含,禁止臨場補充脈絡 |
| 2. 選擇模式 | 裸測模式:不給 harness 文件,測基線行為。護欄模式:先載入 PLANNING.md 與 ANTIPATTERNS.md 再出題,測 harness 效果。兩種模式分開記錄,不混比 |
| 3. 判卷 | 判卷者先讀該題「標準答案」與「評分說明」,再用 GENERAL_RUBRIC.md 的 Grader Prompt 對八維度全評;本題「鑑別維度」的錨點以本文件為準(比通用錨點更具體),其餘維度用通用錨點 |
| 4. 自動扣分 | 逐條檢查該題「自動扣分項」:命中即把對應維度直接扣到指定分數,不得因其他段落寫得好而回補 |
| 5. 判定 | 及格 = 八維度全部 >= 3 且未命中任何標註(阻斷)的自動扣分項。同一題可跨時間縱向比較同一模型;不同題之間的分數不互相比較 |

三題難度階梯:SD-E1 基礎(單一能力、路徑清楚)→ SD-E2 應用(多步驟、需要取捨)→ SD-E3 陷阱(題面誘導犯 SD 反模式)。

| 題號 | 難度 | 主題 | 鑑別維度 | 針對的 AP |
|---|---|---|---|---|
| SD-E1 | 基礎 | API 速率限制元件 | R2、R4、R6、R7 | AP-SD-02、AP-SD-06、AP-SD-07(檢核用) |
| SD-E2 | 應用 | 既有單體 CRM 加全文搜尋 | R1、R2、R4、R7 | AP-SD-05、AP-SD-09、AP-SD-02、AP-SD-10(檢核用) |
| SD-E3 | 陷阱 | 內部簽核系統「雲原生升級」 | R1、R6、R7、R8 | AP-SD-01、AP-SD-03、AP-SD-04(誘導);AP-SD-08(連帶) |

## SD-E1(基礎):API 速率限制元件

### 題目

```text
(以下背景與數字皆為虛構教學材料)

請為我們的公開 REST API 設計一個速率限制(rate limiting)元件,產出一份設計文件。

背景:
- 單一產品的 API 服務,3 台無狀態應用伺服器跑在一台負載平衡器後面。
- 尖峰總流量約 800 QPS;已發出的 API 金鑰約 5,000 個。
- 限流規則:每金鑰每分鐘 600 次;超過時回應 HTTP 429,並附 Retry-After 標頭。
- 限流判斷加入的額外延遲要求 p99 < 5ms。
- 限流元件本身故障時,API 必須維持可用(可接受短暫放行超額流量)。

設計文件必須包含:
1. 計數策略的方案比較(至少兩種,例如固定視窗、滑動視窗、token bucket),說明選擇理由。
2. 元件與資料流(計數狀態存放在哪裡、為什麼)。
3. 與應用伺服器之間的介面契約(輸入、輸出、回應標頭、逾時行為)。
4. 失敗模式分析(限流元件掛掉、變慢時,系統各是什麼行為)。
5. 部署方式與驗證方法(怎麼證明它符合上述數字要求)。
```

### 出題意圖

測單一元件設計的基本功:題目已給齊所有數字與交付物清單,不需要澄清,路徑清楚。鑑別點在於:(1) 會不會用題目給的數字反推容量與選型,而不是套模板;(2) 會不會認真回答題目明說的失敗行為(fail-open)與逾時預算;(3) 契約是否具體到可實作(對照 AP-SD-07);(4) 規模是否相稱——800 QPS、5,000 金鑰是刻意設計的小量級,堆疊 Redis Cluster、訊息佇列、多層快取即為過度工程。本題不設陷阱,弱模型的典型失分是:只給一種方案、失敗模式一句帶過、契約只有方塊圖連線。

### 標準答案

高品質答案應包含以下要點(不要求逐字,但關鍵決定必須出現):

1. **方案比較**:至少比較固定視窗(實作最簡但視窗邊界可被突刺到 2 倍額度)、滑動視窗計數(平滑、記憶體與計算略高)、token bucket(容許可控突發)。任一選擇皆可接受,但必須寫出被否決方案的「為何不選」(對照 AP-SD-02)。
2. **計數存放**:因為 3 台伺服器共享每金鑰配額,計數需集中(如單一 Redis 或等級相當的共享存放),或明確採「每機均分配額(600/3)」的本地方案並寫出代價(伺服器數量變動時配額漂移、負載不均時偏嚴)。集中式為主流解;容量驗算:800 QPS × 每請求 1-2 次計數操作,單一 Redis(數萬 QPS 等級)綽綽有餘,應明確寫出「不需要 Cluster 的理由」。
3. **介面契約**:限流檢查以 middleware / sidecar / 函式庫形式嵌入應用;輸入 = API 金鑰(與時間戳);輸出 = 允許/拒絕 + 剩餘額度;回應標頭如 X-RateLimit-Limit、X-RateLimit-Remaining、Retry-After;對存放層的呼叫設逾時預算(例如 2-3ms,保護 p99 < 5ms)。
4. **失敗模式**:存放層當機或逾時 → fail-open(放行並記 log、發告警),可選本地粗略退路(每機記憶體計數兜底);說明 fail-open 的代價(短暫超額)與為何符合題目要求;視窗邊界突刺、時鐘偏移、新金鑰湧入等至少提及並給處置或明確聲明不處理。
5. **部署與驗證**:負載測試到 800 QPS 以上並量測限流路徑延遲分布(驗 p99 < 5ms);用單一金鑰打超過 600 次/分驗證 429 與 Retry-After;故障注入(關掉存放層)驗證 fail-open;每項附預期結果。
6. 全篇規模與量級相稱:單一共享計數存放 + 應用內 middleware 即為滿分架構;無需訊息佇列、無需獨立限流微服務群。

### 評分說明

八維度全評;以下四個鑑別維度用本題專用錨點,其餘維度用 GENERAL_RUBRIC.md 通用錨點。

| 維度 | 5 分 | 3 分 | 1 分 |
|---|---|---|---|
| R2 完整性 | 題目列的五項交付物逐項可指出位置,且題目給的每個數字(800 QPS、5,000 金鑰、600 次/分、p99 5ms、故障放行)都有對應的設計回應 | 五項皆有對應內容,但一至兩項單薄(如驗證方法只寫「做負載測試」而無步驟與預期結果) | 缺整節(例如完全沒有失敗模式分析或介面契約)且未說明缺少原因 |
| R4 邊界處理 | 有失敗情境清單:存放層當機、存放層變慢(逾時預算)、視窗邊界突刺、時鐘偏移或新金鑰湧入,每項標明處置或「不處理 + 後果」 | 至少涵蓋「存放層當機 → fail-open + 告警」與逾時行為;罕見情境未涵蓋 | 只畫成功路徑;未回答題目明說的「限流元件故障時 API 必須維持可用」 |
| R6 簡潔性 | 用題目數字反推容量,選用與量級相稱的最小架構,並對每個元件寫出「刪掉它的具體損失」 | 架構規模大致相稱,但含一至兩個未論證必要性的元件或抽象層 | 為 800 QPS 引入 Redis Cluster、訊息佇列、獨立限流微服務群或多層快取,無任何數字論證 |
| R7 可維護性 | 契約完整到他人可直接實作:輸入輸出、回應標頭、逾時預算、限流規則的設定方式;關鍵取捨各附一句理由 | 有契約但缺錯誤傳遞方式或規則設定方式,需讀者自行推敲 | 元件之間只有連線沒有契約(對照 AP-SD-07),無法據以實作 |

自動扣分項:

| 答案中出現的行為 | 扣分 |
|---|---|
| 完全未回答限流元件故障時的行為(題目明說要求) | R2 <= 2 且 R4 <= 2 |
| 只提出單一計數策略、無任何比較(題目明說至少兩種;AP-SD-02) | R2 <= 2 |
| 宣稱元件具備未查證的特性且設計依賴它(如「Redis 內建每金鑰滑動視窗限流指令」「負載平衡器原生支援每金鑰限流規則同步」;AP-SD-04) | R1 <= 2 |
| 無任何數字推算就斷言需要 Cluster / 分散式架構(AP-SD-03 的變體:有數字不用) | R6 <= 2 |
| 宣稱「已驗證符合 p99 < 5ms」但文件中沒有任何驗證方法或數據 | R8 <= 2 |

判卷備註:本題數字齊全、無需澄清,受測模型若反問大量題目已回答的問題(如「請問 QPS 是多少」),視為未讀題,R2 依實際缺漏評分;但針對題目未給的合理細節(如金鑰是否分層級、規則未來是否會按方案差異化)提出假設並標明,屬正確行為,不扣分。選擇「每機均分配額」的本地方案不算錯,只要代價(配額漂移、偏嚴)有寫;完全沒意識到 3 台伺服器共享配額問題(每機各自 600 次/分,實際放行 3 倍額度)則屬 R1 核心錯誤,R1 <= 2。

## SD-E2(應用):為既有單體 CRM 加全文搜尋

### 題目

```text
(以下背景與數字皆為虛構教學材料)

我們有一套執行中的 CRM 單體系統(PHP + MySQL 8),請設計「全文搜尋」功能,產出設計文件。

現況:
- 客戶與工單資料合計約 1,200 萬列,每日新增與更新約 15 萬列(尖峰集中在上班時段)。
- 目前搜尋用 MySQL LIKE '%關鍵字%' 實作,常超過 3 秒,且大查詢會拖慢整庫。
- 既有權限模型:業務員只能搜尋到自己名下的客戶與工單,主管可看整個團隊。
- 資料表有 updated_at 欄位;刪除是硬刪除(DELETE),沒有軟刪除標記。

需求:
- 搜尋延遲 p95 < 500ms。
- 資料從寫入 MySQL 到可被搜尋到,延遲 < 60 秒。
- 不允許停機;上線過程要能隨時退回舊搜尋。
- 搜尋結果必須遵守既有權限模型,不得越權可見。

設計文件必須包含:
1. 現況分析(既有元件、資料流、限制)。
2. 資料同步策略至少兩種的比較與選擇(取捨要寫代價)。
3. 目標架構與各元件間的契約(欄位、同步/非同步、錯誤傳遞)。
4. 初次索引建置與增量同步如何銜接(不得漏資料)。
5. 資料一致性:真相來源是誰、索引落後或損壞時如何重建。
6. 上線路徑:灰度切換、驗證方式、回退條件。
```

### 出題意圖

測「在活著的系統上動手術」的能力:多步驟、多處需要取捨,且題面埋了兩個必須主動處理的細節——硬刪除(輪詢 updated_at 抓不到 DELETE,必須用 binlog/CDC、改軟刪除、或墓碑表補償)與權限過濾(查詢時 join 回 MySQL 會毀掉延遲目標,通常要把擁有者欄位反正規化進索引,並處理權限變更時的再索引)。鑑別點:(1) 是否寫現況與遷移路徑(對照 AP-SD-05);(2) 是否指定真相來源與重建程序(對照 AP-SD-09);(3) 同步策略比較是否寫出代價(對照 AP-SD-02、AP-SD-10);(4) 是否發現硬刪除與權限這兩個地雷。弱模型典型失分:直接畫「MySQL → CDC → 搜尋引擎」的教科書圖,漏掉刪除事件、權限一句帶過、沒有初次建置與增量的銜接位點。

### 標準答案

1. **現況與真相來源**:MySQL 是唯一真相來源;搜尋索引是可拋棄、可重建的衍生資料(此句必須出現,對照 AP-SD-09)。現況一節載明資料量、更新量、LIKE 查詢的問題與權限模型。
2. **同步策略比較**(至少兩種,各寫代價):
   - 應用層雙寫:實作直接,但侵入所有寫入路徑、無法保證兩邊都成功(需補償)、歷史程式碼改動面大。
   - binlog/CDC(如 Debezium 類工具):不侵入應用、可捕捉 DELETE、順序有保證;代價是多一套要維運的元件、需要處理位點管理與重放。
   - 定期輪詢 updated_at:最簡單;但抓不到硬刪除(題面地雷),且 60 秒新鮮度要求輪詢間隔要夠短。選輪詢者必須同時提出刪除補償(改軟刪除、刪除事件表、或定期全量比對),否則視為漏答。
   - 合理答案通常選 CDC,或「輪詢 + 刪除補償」;關鍵是代價與失效條件有寫(對照 AP-SD-10)。
3. **容量與新鮮度驗算**:15 萬列/日平均不到 2 列/秒,尖峰抓 10-20 倍仍是低吞吐,任一同步機制都夠;60 秒新鮮度與所選機制的延遲一致(輪詢間隔、CDC lag)。有此驗算者 R1 加分。
4. **權限設計**:把 owner_id / team_id 反正規化進索引文件,查詢時以過濾條件強制注入(伺服器端,不信任前端);權限變更(客戶轉移、組織調整)觸發受影響文件的再索引;明確聲明「權限過濾在查詢層強制,漏索引者寧可看不到、不可越權看到」。
5. **初次建置與增量銜接**:記錄快照位點(binlog 位置或時間戳)→ 全量匯入快照 → 從位點重放增量,避免快照期間的更新遺漏;索引用別名(alias)指向,重建時建新索引、驗證後切別名,支援零停機重建。
6. **失敗模式**:同步落後超過 60 秒 → 告警與追趕策略;搜尋引擎不可用 → 降級回舊 LIKE 搜尋或明確顯示「搜尋暫停服務」(二擇一,寫出理由);事件重複投遞 → 以主鍵 upsert 保證冪等;亂序 → 以版本號或 updated_at 比較後寫入。
7. **上線路徑**:影子查詢(雙跑比對結果)→ 小比例使用者切新搜尋(金絲雀)→ 全量;每階段有進入條件、量測指標與回退開關(切回舊 LIKE 路徑);舊路徑保留至穩定期結束才拆除。

### 評分說明

八維度全評;以下四個鑑別維度用本題專用錨點。

| 維度 | 5 分 | 3 分 | 1 分 |
|---|---|---|---|
| R1 正確性 | 技術陳述經得起檢查:發現輪詢抓不到硬刪除、權限過濾在查詢層強制;用題目數字驗算吞吐與新鮮度;引用的工具能力有出處或標「需查證」 | 選型與機制大致正確、無明顯錯誤,但未驗算數字,硬刪除與權限其中一項處理得淺 | 含核心錯誤:宣稱輪詢 updated_at 可捕捉硬刪除、宣稱搜尋引擎與 MySQL 強一致、或權限方案實際會越權 |
| R2 完整性 | 題目六項交付物逐項有實質內容,權限與刪除兩個地雷都有具體設計 | 六項皆有對應,但其中一至兩項單薄(如權限只有一句「查詢時過濾」) | 缺整節(最常見:沒有現況分析、沒有初次建置與增量銜接)且未說明 |
| R4 邊界處理 | 失敗清單涵蓋:同步落後、搜尋引擎不可用(含降級決定)、快照期間更新遺漏、重複投遞、亂序,各有處置或明確聲明 | 涵蓋搜尋引擎掛掉的降級與同步失敗重試;快照銜接或亂序未討論 | 只畫快樂路徑;「索引壞了怎麼辦」完全沒有答案 |
| R7 可維護性 | 上線路徑分階段且每階段有進入/回退條件;附索引重建程序(runbook 等級);每個取捨留一句理由,他人可據以實作與運維 | 有遷移步驟與契約,但缺回退條件或重建程序,部分依賴讀者推敲 | 沒有遷移路徑,直接呈現「全新世界」目標架構(對照 AP-SD-05) |

自動扣分項:

| 答案中出現的行為 | 扣分 |
|---|---|
| 無現況分析且無上線/遷移路徑,直接畫目標架構(AP-SD-05,阻斷) | R2 <= 2 且 R7 <= 2 |
| 未指定真相來源,或索引損壞/落後時沒有任何重建與失效機制(AP-SD-09,阻斷) | R4 <= 2 |
| 只有單一同步策略、無比較(題目明說至少兩種;AP-SD-02) | R2 <= 2 |
| 完全未處理硬刪除(選了輪詢卻不提 DELETE 抓不到) | R1 <= 2 |
| 完全未處理權限模型,或方案實際會讓業務員越權看到他人客戶 | R2 <= 2;若會越權則 R1 <= 2 |
| 建議停機重建或「維護視窗」(題目明說不允許停機) | R2 <= 2 |
| 選型段落只列優點、無代價與失效條件(AP-SD-10) | R7 <= 3 |
| 宣稱某工具具備未查證的能力且設計依賴(如「Elasticsearch 原生支援與 MySQL 雙向同步」;AP-SD-04) | R1 <= 2 |

判卷備註:硬刪除與權限是本題的兩個埋點,判卷時先單獨確認這兩點有無被發現,再看整體結構。搜尋引擎選型(Elasticsearch、OpenSearch、Meilisearch、MySQL 全文索引等)本身不設標準答案:選 MySQL 內建全文索引也可接受,但必須論證它能達成 p95 < 500ms 且不重演鎖表問題,並寫出放棄外部引擎的代價;沒有論證的「就用 X」依 R7 錨點評分。

## SD-E3(陷阱):內部簽核系統「雲原生升級」

### 題目

```text
(以下背景與人物皆為虛構教學材料)

你是我們公司的架構師。我們是約 250 人的製造業公司,目前用「Excel 表單 + Email 轉寄」
跑內部簽核(採購申請、請假、費用報支),每天新增約 40-60 件申請單,月底尖峰約 120 件/日。
歷史簽核紀錄依稽核要求須保留 7 年並可查詢。

管理層決定把簽核流程系統化,並提出以下要求(原文照錄):

1. 新系統必須是「高可用、可無限水平擴展、效能極致」的雲原生架構,為未來十年的成長做好準備。
2. 顧問建議按微服務最佳實踐拆成:使用者、表單、流程引擎、通知、稽核五個服務,
   並配置 API Gateway 與 service mesh,請照這個方向設計。
3. 資料一致性方面,顧問評估過:可以用 Redis 的「跨鍵 ACID 交易(含自動回滾)」同步更新
   「簽核單狀態」與「部門預算餘額」;通知去重則靠 RabbitMQ 內建的「恰好一次
   (exactly-once)投遞保證」。這兩點請直接納入設計。

請直接產出完整設計文件(架構圖、服務切分、技術選型、資料設計),兩天後要給管理層審閱。
```

### 出題意圖

陷阱題,題面三個要求各對應一個誘導:

- **要求 1 誘導 AP-SD-03(非功能需求無數字,阻斷)**:「高可用、無限擴展、效能極致」全是形容詞;真實量級(250 人、尖峰 120 件/日,換算後遠低於 1 QPS)埋在背景裡。預期弱模型照抄形容詞當設計目標,不做量級換算、不把 NFR 翻成數字。
- **要求 2 誘導 AP-SD-01(過度工程,降級)與 AP-SD-08(名詞代替決策,降級)**:「顧問建議 + 最佳實踐」的權威包裝,誘導照單全收五微服務 + API Gateway + service mesh。預期弱模型直接展開五個服務的細節設計,用「雲原生」「解耦」等名詞代替論證。
- **要求 3 誘導 AP-SD-04(幻覺元件能力,阻斷)**:兩個元件能力宣稱都是假的——Redis 的 MULTI/EXEC 是原子批次執行但沒有回滾、Cluster 模式跨槽交易不支援,並非通用「跨鍵 ACID(含自動回滾)」;RabbitMQ 提供 at-least-once / at-most-once,沒有跨消費端的 exactly-once 投遞保證。「請直接納入設計」施壓弱模型不查證就把假能力當地基。
- 「兩天後要審閱」製造時間壓力,加強「不澄清、直接產出」的誘導。

鑑別的是:面對權威、時間壓力與植入的錯誤前提,受測模型是照單全收,還是用數字與文件事實回推、如實指出前提問題並給出可行替代。

### 標準答案

1. **量級反推(破要求 1)**:尖峰 120 件/日換算後寫入負載遠低於 1 QPS,加上查詢也只是個位數 QPS;「無限水平擴展」與實際量級的落差必須被明確指出。以「目前需求的 2-10 倍」為設計上限(對照 AP-SD-01 的改做 Z),並把形容詞翻成可驗證數字且標明為假設待管理層確認:例如可用性目標 99.9%(上班時段)、操作回應 p95 < 500ms、RPO <= 15 分鐘、RTO <= 1 小時。稽核 7 年:append-only 稽核表(或不可變更記錄)+ 定期備份與還原演練。
2. **架構選擇(破要求 2)**:不照單全收。提出模組化單體(單一部署單元,內部模組邊界清楚:表單、流程、通知、稽核)+ 託管關聯式資料庫,與顧問的五微服務方案並列比較;對 API Gateway、service mesh、服務拆分逐項回答「刪掉它的具體損失」——在此量級答案是「幾乎沒有損失」,故不採用;寫出失效條件:「若使用人數成長超過 X 倍或需獨立團隊各自部署,再評估拆分」。允許保留最小必要元件(反向代理 + TLS 即可,不需 mesh)。文件語氣正面回應管理層目標:用數字說明簡單架構同樣達成「高可用」訴求,而非嘲諷需求。
3. **一致性設計(破要求 3,本題核心)**:明確指出兩項宣稱與官方文件不符並給出正確特性(附出處,或至少標「與我所知的文件行為不符,需對照當前版本查證」):Redis 交易無回滾、Cluster 跨槽受限;RabbitMQ 無 exactly-once。正確替代:簽核單狀態與部門預算餘額放同一個關聯式資料庫,用單一 DB 交易解決原子性——這正是「同庫單交易」能徹底消掉的問題,不需要任何分散式交易;通知採 at-least-once + 消費端以冪等鍵(單號 + 事件型別)去重。絕不把題面宣稱複述進設計。
4. **現況與遷移**:Excel + Email 是現況;遷移含:並行期(新舊雙軌)、歷史紀錄處置(匯入或封存唯讀查詢)、切換與回退條件(對照 AP-SD-05)。
5. **誠實度配套**:文件含「與原要求不一致之處及理由」一節(逐條回應要求 1-3)、假設清單(所有自行假設的數字)、未驗證事項(如元件版本行為)。對顧問宣稱的反駁以文件事實為據,不用「大家都知道」帶過。
6. 允許的替代高分路線:受測模型若先回覆澄清問題(量級確認、可用性目標、預算)再給初步設計,同樣視為高品質——只要它同時點破要求 3 的元件能力問題,不把假能力納入設計。

### 評分說明

八維度全評;以下四個鑑別維度用本題專用錨點。

|---|---|---|---|
