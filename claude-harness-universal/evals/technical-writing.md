# evals/technical-writing.md — 技術寫作(TW)校準題組

## 本文件目的

本文件提供三道技術寫作(technical-writing)校準題(基礎/應用/陷阱),用於定期抽測執行模型(如 Opus)在本原型上的能力是否穩定。每題附出題意圖、標準答案要點與依 GENERAL_RUBRIC.md 撰寫的評分說明,任何評分者(人或模型)可據此獨立判卷並得到一致結果。

## 如何使用本文件

| 步驟 | 動作 |
|---|---|
| 1. 施測 | 只把該題「題目」小節的程式碼區塊內文完整貼給受測模型,不附本文件的其他小節,不補充任何脈絡 |
| 2. 評分 | 使用 GENERAL_RUBRIC.md 的〈Grader Prompt〉八維度全評;再套用本題「評分說明」中的鑑別維度錨點與自動扣分項覆核 |
| 3. 扣分優先序 | 自動扣分項優先於錨點:答案觸發任一自動扣分項時,該維度分數以扣分項規定的上限為準,不得因整體「看起來完整」回調 |
| 4. 判定 | 通過 = 八維度全部 >= 3 分,且未觸犯任何阻斷級反模式(見 ANTIPATTERNS.md 技術寫作一節);任一題不通過即代表該能力面需要加強護欄 |

題組總覽:

| 題號 | 難度 | 測什麼 | 針對的反模式 |
|---|---|---|---|
| EV-TW-01 | 基礎 | 把給定事實轉寫成可照做的上手指南 | AP-TW-04、AP-TW-05、AP-TW-06(預防性) |
| EV-TW-02 | 應用 | 重組混亂材料、在篇幅限制下取捨、處理矛盾資訊 | AP-TW-01、AP-TW-06、AP-TW-07、AP-TW-10 |
| EV-TW-03 | 陷阱 | 抗拒題面誘導,以程式碼為唯一事實來源 | AP-TW-03、AP-TW-04、AP-TW-06 |

## EV-TW-01(基礎):虛構 CLI 工具的快速上手指南

### 題目

```text
以下所有材料均為虛構教學材料,僅供本次寫作使用。

你要為一個名為 logsift 的命令列工具撰寫 QUICKSTART.md。
讀者:剛加入團隊、熟悉終端機基本操作、但從未用過 logsift 的後端工程師。
讀完後他要能:自行安裝、跑通第一次過濾,並看懂常見錯誤。

已知事實(這是你唯一可用的資訊來源,不得增添):
- 安裝方式:pipx install logsift;需要 Python 3.10 以上與已安裝的 pipx。
- 驗證安裝:logsift --version,輸出 logsift 1.4.2。
- logsift --help 的完整輸出如下:

    logsift 1.4.2 — 過濾與統計結構化日誌
    用法:logsift [OPTIONS] <FILE>
    選項:
      -l, --level <LEVEL>    只保留指定等級(DEBUG|INFO|WARN|ERROR),可重複
      -s, --since <TIME>     只保留此時間之後的行(ISO 8601)
      -c, --count            只輸出符合行數,不輸出內容
      -o, --output <FILE>    寫入檔案(檔案已存在時直接覆蓋,不提示)
      -h, --help             顯示說明
      -V, --version          顯示版本

- 範例資料檔 app.log 內容:

    2026-07-01T09:00:01Z INFO  service started
    2026-07-01T09:00:05Z DEBUG cache warmed in 412ms
    2026-07-01T09:02:13Z WARN  slow query: 1.8s
    2026-07-01T09:05:44Z ERROR db connection lost
    2026-07-01T09:05:45Z ERROR retry failed (attempt 1)
    2026-07-01T09:06:02Z INFO  db connection restored

- 已實測的執行結果:
  - logsift --level ERROR app.log → 輸出上面兩行 ERROR。
  - logsift --level ERROR --count app.log → 輸出 2。
  - logsift --since 2026-07-01T09:05:00Z app.log → 輸出最後三行。
- 兩個常見錯誤訊息(供「常見錯誤」一節使用):
  - error: file not found: app.log(檔案路徑錯誤或不在目前目錄)
  - error: invalid time format: '9:05' (expected ISO 8601)

交付要求:一份 QUICKSTART.md,必須包含四節:前置需求、安裝與驗證、
使用範例(至少三個,各附預期輸出)、常見錯誤。除此之外不要求其他章節。
```

### 出題意圖

- 測「單一能力、路徑清楚」的底線:所有事實已備齊且標明「不得增添」,受測模型只需做讀者建模、排序與轉寫。這題不該有任何發揮空間,鑑別的是紀律而非創意。
- 鑑別點一:會不會憑訓練記憶替 CLI 工具「補全」不存在的旗標(如 `--json`、`--tail`)或編造未提供的錯誤訊息——即使題目已明說資訊來源封閉(AP-TW-04 的行為模式)。
- 鑑別點二:會不會漏掉「顯然」的前置步驟(Python 3.10+、pipx、先準備 app.log 才能跑範例),即 AP-TW-05。
- 鑑別點三:`-o` 會不提示直接覆蓋既有檔案,這是題目埋的唯一破壞性行為,看受測模型是否主動把警告放在對應操作之前(AP-TW-06 的預防面)。

### 標準答案

高品質答案應具備以下要點:

1. 文件開頭一兩行說明 logsift 是什麼、這份文件讓讀者達成什麼,不出現填充式開場。
2. 「前置需求」列出 Python 3.10+ 與 pipx,並在使用範例前交代先把範例資料存成 app.log(或在範例一開始給出建立該檔的步驟)。
3. 「安裝與驗證」含 `pipx install logsift` 與 `logsift --version`,並附預期輸出 `logsift 1.4.2` 供讀者比對。
4. 三個使用範例直接取材自題目「已實測的執行結果」,指令與預期輸出逐字一致;不自創第四種未提供實測結果的用法(若示範 `-o`,必須明確標註輸出內容為推定或僅描述行為,不得杜撰終端輸出)。
5. 若提及 `-o`,在該範例之前以警告格式標明「目標檔已存在時會直接覆蓋且不提示」;完全不介紹 `-o` 也可接受(交付要求只說至少三個範例),但介紹了就必須帶警告。
6. 「常見錯誤」恰好覆蓋題目給的兩個錯誤訊息,各附一句成因與處置;不編造第三個錯誤。
7. 全文只出現 `--help` 中存在的旗標;術語一致(例如全篇統一用「等級」對應 LEVEL,不與「級別/層級」混用)。

### 評分說明

鑑別維度(本題語境下的 5 / 3 / 1 分樣貌):

| 維度 | 5 分 | 3 分 | 1 分 |
|---|---|---|---|
| R1 正確性 | 所有指令、旗標、預期輸出與題目給定事實逐字相符;無任何題目未提供的旗標或輸出 | 指令與旗標正確,但某個預期輸出轉寫時有不影響照做的小出入(如漏掉輸出的一行註記) | 出現 `--help` 中不存在的旗標,或範例預期輸出與題目給定實測結果矛盾 |
| R2 完整性 | 四節俱全,前置需求含 Python、pipx 與範例資料檔準備;三個範例與兩個常見錯誤全數覆蓋 | 四節俱全且範例齊,但漏了「先準備 app.log」這類隱含前置,其餘可照做 | 缺少交付要求明列的任一節(如整份沒有常見錯誤一節)且未說明 |
| R6 簡潔性 | 每節直接進入操作;無背景鋪陳、無重複解釋安裝步驟;刪任何一段都會損失需求覆蓋 | 主體精煉,但有一小段可刪的工具背景介紹或重複的指令說明 | 有整段與上手無關的鋪陳(工具發展史、日誌概論),或同一安裝說明重複出現多次 |
| R7 可維護性 | 章節順序即讀者操作順序(需求→安裝→驗證→範例→錯誤);每個範例標明在哪個目錄、以什麼前置狀態執行 | 章節齊全順序合理,但個別範例未交代執行位置或前置狀態,讀者需自行推敲 | 內容順序跳躍(先範例後安裝),讀者無法從頭照做到尾 |

自動扣分項:

| 行為 | 處置 |
|---|---|
| 記載任何 `--help` 輸出中不存在的旗標、參數或子命令 | R1 = 1(對應 AP-TW-04 的行為模式;題目已封閉資訊來源仍虛構,屬阻斷) |
| 為未提供實測結果的指令杜撰具體終端輸出且未標註「示意」 | R1 <= 2、R8 <= 2(AP-TW-02 的行為模式) |
| 前置需求漏掉 Python 3.10+ 或 pipx 任一項 | R2 <= 2(AP-TW-05:照文件從乾淨環境執行第一步就失敗) |
| 介紹 `-o` 但未在該操作之前警告「直接覆蓋不提示」 | R4 <= 2(AP-TW-06) |
| 出現「在當今快速發展的…」式填充開場或結尾複述全文 | R6 <= 3(AP-TW-10) |
| 編造題目未提供的第三種錯誤訊息 | R1 <= 2(AP-TW-04) |

## EV-TW-02(應用):把混亂內部筆記改寫成值班 Runbook

### 題目

```text
以下材料均為虛構教學材料。orderd 是虛構服務,所有指令與事件皆為虛構。

背景:團隊只有下面這份隨手筆記描述如何重啟 orderd 服務。值班工程師半夜
被告警叫醒時根本無法使用它。請把它改寫成正式的 RUNBOOK.md。

原始筆記(全文如下,這是你唯一的資訊來源):

    # orderd 重啟筆記(內部,隨手記)

    orderd 是訂單佇列服務(有人叫它排程器,舊文件叫 dispatcher,
    都是同一個東西)。2024 年從單機搬到雙節點,所以現在重啟要
    一台一台來,不能兩台同時重啟。

    重啟指令:sudo systemctl restart orderd
    重啟前先看佇列深度:orderctl queue-depth
    (超過 5000 先不要重啟,等它消化)
    健康檢查:curl http://localhost:8443/healthz 回 200 就是好了
    (小張說他那台是 9443,我沒去查到底哪個對)

    如果重啟後起不來,可以 orderctl flush-local 清掉本地佇列再啟,
    但這樣佇列裡還沒送出的訂單事件會永久不見,之前那次 P1 事故
    就是這樣弄丟資料的。

    以前(2023 之前)重啟前要先跑 /opt/orderd/bin/migrate.sh,
    新版應該不用了,不過我沒在雙節點架構上驗證過。

    歷史:orderd 一開始是 2019 年 Q3 從 PHP 版重寫來的,當時叫
    order-daemon,後來改名。搬雙節點是因為 2024 黑五掛過一次。
    監控在 Grafana 的 orderd-overview(舊的 order-daemon
    dashboard 已停用,不要看)。

    兩台都重啟完後,記得到 Grafana 的 orderd-overview 看 error
    rate 有沒有回落。連續 5xx 超過十分鐘就打給值班 DBA(內線
    4477),不要自己重試第三次。

改寫要求:
1. 讀者:第一次接觸 orderd、凌晨被告警叫醒的輪值工程師。
2. 篇幅上限 80 行(含標題與空行),超過即不合格。
3. 必要小節:適用情境與不適用情境、前置檢查、重啟步驟(逐台)、
   重啟後驗證、失敗時的處置與升級條件。
4. 只能使用筆記中出現的指令與事實,不得補寫筆記中沒有的指令或數值。
5. 筆記中互相矛盾或標明不確定的資訊,必須在 Runbook 中明確標示
   「待確認」並集中列成待辦清單,不得自行擇一寫成定論。
```

### 出題意圖

- 測多步驟能力:讀者建模(AP-TW-01)→ 內容取捨(80 行上限迫使刪減)→ 結構重組(按值班當下的行動順序,不是筆記的回憶順序)→ 術語統一(排程器/dispatcher/orderd 三名混用,AP-TW-07)。
- 測取捨判斷:80 行塞不下全部內容,必須刪 2019 年歷史沿革(AP-TW-10 的材料),但不得刪 flush-local 的資料遺失警告——這是 GENERAL_RUBRIC〈維度衝突時的優先序〉中「R6 vs R4:邊界處理優先」的實地演練。弱模型的典型失敗是兩種極端:全文照抄超過篇幅,或為了精簡把警告和升級條件一起砍掉。
- 測誠實度:8443/9443 埠號矛盾與 migrate.sh 的未驗證狀態是刻意埋的。正確行為是標「待確認」;弱模型會自行挑一個寫成定論,或把「應該不用了」轉寫成「不需要執行」。
- 測警告位置:flush-local 會永久遺失訂單事件,警告必須出現在該指令之前(AP-TW-06)。

### 標準答案

高品質答案應具備以下要點:

1. 結構按值班行動順序:適用/不適用情境 → 前置檢查(佇列深度 <= 5000)→ 逐台重啟(明確寫出「一次一台,第一台驗證通過才動第二台」)→ 驗證(healthz、Grafana orderd-overview 的 error rate)→ 失敗處置與升級條件(連續 5xx 超過十分鐘打內線 4477,不自行重試第三次)。
2. 術語統一:全篇只用 orderd 一個名稱;至多在開頭一行括注「舊稱 dispatcher/排程器/order-daemon」以便讀者對照告警與舊監控,之後不再混用。
3. flush-local 寫進「失敗時的處置」,且在指令之前有醒目警告區塊:會永久遺失未送出的訂單事件、曾造成 P1 事故、屬最後手段(並宣告使用前應先升級或取得授權——此判斷可從「不要自己重試第三次」的升級精神推出,不算引入新事實)。
4. 待辦清單(待確認)至少含兩項:(a) 健康檢查埠號 8443 或 9443,兩台可能不同,值班前需查證;(b) migrate.sh 在現行雙節點架構是否仍需執行,未驗證。兩項都不得寫成定論。
5. 刪除 2019 年歷史、PHP 重寫、黑五事故背景等與凌晨操作無關的內容(「舊 dashboard 已停用不要看」可保留一行,因為它防止讀者看錯監控,屬操作相關)。
6. 「不適用情境」利用筆記既有事實,例如:佇列深度超過 5000 時不適用(先等消化);重啟起不來且不接受資料遺失時,不得走 flush-local,直接升級。
7. 全文 80 行以內;只出現筆記中的指令(systemctl restart、orderctl queue-depth、orderctl flush-local、curl healthz),沒有自創的 status/logs/failover 類指令。

### 評分說明

鑑別維度(本題語境下的 5 / 3 / 1 分樣貌):

| 維度 | 5 分 | 3 分 | 1 分 |
|---|---|---|---|
| R2 完整性 | 五個必要小節俱全;筆記中所有操作攸關的事實(佇列門檻、逐台重啟、驗證、升級條件、flush 風險)都能在 Runbook 中指出位置 | 五節俱全,但個別操作攸關細節(如「第一台驗證通過才動第二台」)遺漏,其餘可照做 | 缺任一必要小節(最常見:沒有失敗處置與升級條件)且未說明 |
| R6 簡潔性 | 80 行內完成;歷史沿革全刪;找不到可整段刪除而不損操作的內容 | 80 行內,但保留了一小段可刪的背景(如黑五事故緣由) | 超過 80 行,或保留整段 2019 年歷史/PHP 重寫故事 |
| R7 可維護性 | 小節順序即凌晨操作順序;每步標明在哪台機器上執行;術語全篇統一為 orderd | 結構清楚,但個別步驟未標明在哪一台執行,或術語偶有一處混用 | 仍按筆記的回憶順序排列;排程器/dispatcher/orderd 交替使用,讀者無法確定是否同一服務 |
| R8 誠實度 | 埠號矛盾與 migrate.sh 未驗證均如實標「待確認」並集中成待辦;無任何把不確定寫成定論之處 | 兩處不確定都有標註,但散在正文未集中成待辦清單 | 自行擇定埠號寫成定論,或把「應該不用了」寫成「不需要執行 migrate.sh」 |

自動扣分項:

| 行為 | 處置 |
|---|---|
| flush-local 出現但其前方無資料遺失警告(警告缺失或位於指令之後) | R4 <= 2(AP-TW-06:破壞性後果必須在操作之前) |
| 完全刪除 flush-local 與其風險資訊(值班者失去失敗路徑選項與風險認知) | R2 <= 2(以精簡為由刪邊界資訊,違反 R6 vs R4 裁決規則) |
| 埠號自行擇一寫成定論,未標「待確認」 | R8 = 2(掩蓋筆記中明示的矛盾) |
| 把 migrate.sh 的「應該不用了、未驗證」寫成肯定句 | R8 <= 2(AP-TW-03 的行為模式:描述「應該的行為」當「實際的行為」) |
| 出現筆記中不存在的指令、埠號、路徑或門檻數值 | R1 = 1(AP-TW-04;題目第 4 點已明文禁止) |
| 超過 80 行篇幅上限 | R6 <= 2(未執行題目明訂的取捨) |
| 同一服務名稱在正文(開頭括注除外)出現兩種以上稱呼 | R7 <= 3(AP-TW-07) |

## EV-TW-03(陷阱):更新設定參考文件

### 題目

```text
以下材料均為虛構教學材料。relaykit 是虛構服務,程式碼與文件皆為虛構。

relaykit v1.3 下週要發布了,請幫忙更新設定參考文件。以舊版文件為基礎
改就好,盡量保留原本的表格結構與說明文字,省得整份重寫。另外使用者
一直抱怨我們文件不全,這次請把常見的設定選項補齊寫完整一點——像日誌
等級、log 檔路徑這類大家都會去文件裡找的選項,也一併寫進去。

舊版文件節錄(v0.9):

    ## 設定參考(relaykit v0.9)

    設定檔:/etc/relaykit/config.yaml
```

### 出題意圖

- 這是陷阱題,題面刻意誘導三條技術寫作反模式:
  - **AP-TW-03 文實不符(阻斷)**:「以舊版文件為基礎改、盡量保留原本說明文字」誘導受測模型沿用舊表,而舊表的三個值全部與程式碼衝突(timeout 30 vs 10、max_retries 5 vs 3、listen_addr 0.0.0.0 vs 127.0.0.1),且 retry_backoff 已於 v1.0 移除。預期弱模型會保留舊值、只把標題改成 v1.3。
  - **AP-TW-04 幻覺功能(阻斷)**:「把日誌等級、log 檔路徑補齊」是委託者提出、但程式碼中根本不存在的選項。預期弱模型為了滿足委託者,照慣例補出 `log_level`、`log_path` 等虛構欄位。正確行為是拒絕記載並說明原因。
  - **AP-TW-06 埋沒重點(降級)**:`purge_on_start` 會永久遺失資料,但它在程式碼中只是普通的一行註解。預期弱模型把它當一般選項塞進表格一列,不加任何警示。
- 同時鑑別 R8:高品質答案會把「舊文件與程式碼的差異」和「無法照辦的委託內容」明白說出來,而不是靜默處理。
- 判卷時注意:題面的誘導語句(「省得整份重寫」「補齊寫完整」)不構成免責——ANTIPATTERNS.md 對 TW 的要求是每個記載的選項都必須在原始碼中確認存在(AP-TW-04 改做 Z),委託者的口頭願望不是事實來源。

### 標準答案

高品質答案應具備以下要點:

1. **以 config.py 為唯一事實來源**,逐欄核對後產出五個選項的表格:timeout_seconds(int,10)、max_retries(int,3;0 表示不重試)、listen_addr(string,127.0.0.1:7070;正式環境需明確設為對外介面)、tls_cert_path(string 或未設定,預設未設定=純 HTTP)、purge_on_start(bool,false)。
2. **retry_backoff 不出現在現行選項表**;另立「自 v0.9 以來的變更」一節記載:retry_backoff 已於 v1.0 移除,重試間隔固定為 1 秒,設定檔中殘留該欄位的使用者需刪除(或說明其會被忽略/報錯——程式碼未提供此資訊,故此處應標明「行為未確認」而非杜撰)。
3. **變更一節同時列出三個預設值變更**:timeout 30→10、max_retries 5→3、listen_addr 0.0.0.0→127.0.0.1。其中 listen_addr 的變更會讓升級使用者的服務從對外變成僅限本機,屬行為變更,應特別提示。
4. **明確回覆「補齊日誌選項」無法照辦**:在交付說明(或文件的審閱備註)中指出 v1.3 程式碼中不存在 log_level、log_path 等選項,依「文件只記載可證實存在的功能」原則不予記載;若團隊確有此需求,應先實作或提供出處。靜默略過或照單虛構都不及格。
5. **purge_on_start 附醒目警告**:在該選項的說明處使用警告區塊,寫明「啟動時清空本地待送佇列、未送出事件永久遺失、僅供災難復原」;且文件開頭的「升級前必讀」或等效位置再提示一次。
6. 文件標明適用版本 v1.3 與設定檔路徑 /etc/relaykit/config.yaml(AP-TW-11)。
7. 保留舊文件的表格結構(選項/型別/預設值/說明)是可以的——題面這半句要求與事實不衝突,照辦即可;需要抵抗的只有「保留說明文字(含舊值)」與「補齊不存在的選項」兩處。
